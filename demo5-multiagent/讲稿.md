## 第 1 章 结论：demo5 比 demo1 多了什么

### 【口播 · 开场】

demo1 给我们打下了 Agent 的心跳——`LLM × 工具 × 循环 × 状态`（ReAct）。demo2/3/4 分别加了记忆轴、工具轴、规划轴——但都还在**单个 Agent 内部**做文章。

单 Agent 的根本痛点：**messages 是一条线性流水**。所有工具调用、所有中间结果、所有思考文本都往这一条流水里塞。遇到两类场景就出问题：

1. **相互独立的子任务**——「统计文件数 + 读某文件首行」两个互不相干的事，塞进同一个 messages 互相污染
2. **有依赖 + 需通信 + 要质检的任务**——「调研 → 写报告 → 质检打回 → 重写」这种流水线，单 Agent 既不能分包、又不能累积多角色记忆、更没有质检闸门

demo5 的轴是「多 Agent」，就解决这两件事——用**两条机制**分别对应：

> ### demo5 = base × 多 Agent
> （demo1 是 base，demo5 在 base 上独立加「多 Agent」轴——不叠加 demo2 的记忆轴、不叠加 demo3 的工具轴、不叠加 demo4 的规划轴）

### 【屏显 · 表格 1-1 — demo5 vs demo1】

| 维度 | demo1 | demo5 |
|---|---|---|
| **Agent 数量** | 1 个 | **N 个（Subagent 一次性 / Team 持久项目组）** |
| **messages** | 单 Agent 一份线性累积 | **每个 Agent 独立一份**——互不污染 |
| **子任务分包** | ❌ 全塞进同一 messages | ✅ **Subagent：独立 context + 结束即销毁** |
| **Agent 间通信** | ❌ 无 | ✅ **Team：inbox 收件箱 + 长期 messages** |
| **质检闸门** | ❌ 主 Agent 自己看一眼 | ✅ **Team：Reviewer 状态机 + 打回重做** |

### 【口播 · base 不变】

注意——**demo1 的核心都没动**：

1. **ReAct 循环不变**（LLM → tool_use → tool_result → ... → end_turn）
2. **三个本地工具不变**（execute_bash / read_file / write_file 原封不动）
3. **system prompt + messages 结构不变**

demo5 只做**两件新事**，分别落在两个文件里：

- **`agent_sub.py`**（70% 权重）——加一个 `subagent` 本地工具，主 Agent 遇到独立子任务时派一个一次性 Subagent 干完即销毁
- **`agent_team.py`**（30% 权重）——加 `Agent` 类（持久对象 + inbox）+ `Team` 类（Researcher→Writer→Reviewer 流水线 + 状态机）

两条机制对应两类工业范式：

| 机制 | 文件 | 工业对应 |
|---|---|---|
| **Subagent**（一次性外包） | `agent_sub.py` | Claude Code 的 Task tool / Cursor 的 agent / Devin 的子任务派发 |
| **Team**（持久项目组） | `agent_team.py` | AutoGen / CrewAI 范式 |

接下来 5 章，逐层拆开。

## 第 2 章 为什么需要多 Agent

### 【口播 · 单 Agent 上下文膨胀痛点】

让我们看 demo1 在一个**看起来很简单**的任务上的真实表现。假设用户输入：

> 1) 统计 demo5-multiagent 下 .py 文件数；2) 读 agent_sub.py 第 1 行注释

demo1 的 ReAct 会这样做：

```
第 1 轮：LLM 决策 → execute_bash("find demo5-multiagent -name '*.py' | wc -l")
第 2 轮：LLM 看结果 → read_file("demo5-multiagent/agent_sub.py")
第 3 轮：LLM 总结两个结果
```

三步、三轮、看起来也漂亮。但这是**简单任务**——两个子任务都能用单步工具完成。

现在换一个**真正复杂**的任务：

> 调研一下 Python 的 GIL 是什么，写一份结构化的研究报告，写完要有人质检——不通过就打回来重写。

demo1 的单 Agent 会怎样？

- **调研的中间笔记**、**报告的多个草稿版本**、**质检员的反馈意见**——全塞进同一条 messages
- 主 Agent 上一秒还在当研究员写笔记，下一秒要切到撰稿人视角写报告，再下一秒又要当质检员挑自己写的毛病——**角色混乱**
- 没有「质检不通过 → 打回重做」的显式机制——只能靠 LLM 在 messages 里自己记得"我现在是被打回的第 2 次"

本质原因：**单 Agent 的 messages 是线性的，没有"角色边界"、没有"任务流转"、没有"状态机"**。

### 【屏显 · 表格 2-1 — 两类痛点 vs 两条机制】

| 痛点 | 场景 | demo5 的解药 | 所在文件 |
|---|---|---|---|
| **相互独立的子任务** | 「1) 统计文件数；2) 读首行」——两个子任务互不相干 | **Subagent**：独立 context，干完即销毁 | `agent_sub.py` |
| **有依赖 + 需通信 + 要质检** | 「调研 → 写报告 → 质检打回 → 重写」流水线 | **Team**：持久 Agent + inbox + 状态机 | `agent_team.py` |

### 【口播 · 两条机制的设计哲学】

这两条机制的差别不是"功能强弱"，而是**设计哲学不同**：

1. **Subagent 是"无状态的外包工"**——主 Agent 把一个独立子任务连角色带任务描述一起扔出去，Subagent 用全新的 messages 干完，把结果摘要交回来，自己就销毁。主 Agent 看到的只是"一段结果文本"，Subagent 内部的思考过程完全透明丢弃。
2. **Team 是"有状态的项目组"**——每个 Agent 是一个持久对象，有固定 name/role、有跨多次 `chat()` 累积的 messages、有可被其他 Agent 塞消息的 inbox。任务在 Agent 之间流转靠事件驱动状态机，质检员是总闸门。

接下来两章分别拆解。

## 第 3 章 机制一：Subagent（一次性外包）

### 【口播 · Subagent 是什么】

Claude Code 里有个工具叫 **Task**——它的作用很简单：让主 Agent 把一个独立子任务"分包"给一个临时 Agent，临时 Agent 用自己独立的 messages 干完，把结果摘要交回来就销毁。

听起来很朴素，但它的本质是**给主 Agent 减负**：

- 主 Agent 的 messages 不再被无关子任务的中间结果污染
- 每个 Subagent 用全新 context 专注一件事——做完就销毁，不占内存
- 多个独立子任务可以派多个 Subagent（主 Agent 看到的只是 N 段结果摘要）

### 3.1 subagent 工具的定义

### 【屏显 · 代码 3-1 — subagent 工具定义（agent_sub.py Part 2）】

```python
{
    "name": "subagent",
    "description": (
        "委派一个独立的 Subagent 来完成子任务。Subagent 拥有独立的角色（system_prompt）"
        "和独立的上下文（messages），执行完返回结果摘要后即销毁。\n\n"
        "**使用时机**：**相互独立的子任务**才用——每个子任务派一个 Subagent。"
        "当用户输入包含「1) 2) 3)」这种编号列表、且列出 2 个及以上相互独立的子任务时，"
        "**必须**为每个子任务**调用一次 subagent 工具**，由对应 Subagent 完成。\n\n"
        "**不适合的场景**：后一步要用前一步结果的链式任务（例如「先读文件，再把内容排序后写入新文件」），"
        "这种应让主 Agent 自己顺序完成——Subagent 干完就销毁，无法把结果传给下一个 Subagent。"
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "role": {"type": "string", "description": "Subagent 的角色，例如「Python 工程师」/「文件统计员」"},
            "task": {"type": "string", "description": "交给 Subagent 完成的具体任务描述"},
        },
        "required": ["role", "task"],
    },
}
```

### 【口播 · description 的关键设计】

description 里有两段引导，决定 LLM 何时用、何时不用：

1. **正向触发**：「编号列表 + 2 个及以上相互独立的子任务 → 必须派 Subagent」
2. **反向抑制**：「链式依赖任务（先 A 再 B）让主 Agent 自己顺序做——Subagent 干完就销毁，无法把结果传给下一个」

这两段缺一不可。只给正向条件，LLM 会**滥用**（每个任务都派 Subagent）；只给反向条件，LLM 会**从不使用**。

### 3.2 subagent 不在路由表里——主循环单独拦截

### 【屏显 · 代码 3-2 — LOCAL_FUNCTIONS 不含 subagent（agent_sub.py Part 3）】

```python
# 路由表：工具名 → 实际函数（不含 subagent）
LOCAL_FUNCTIONS = {
    "execute_bash": execute_bash,
    "read_file":    read_file,
    "write_file":   write_file,
}
```

### 【口播 · 为什么 subagent 不在路由表】

注意——`subagent` 工具的 schema 在 `ALL_TOOLS` 里（LLM 能看到、能调用），但它的实现**不在** `LOCAL_FUNCTIONS` 字典里。为什么？

因为 Subagent 不是"调一个函数"那么简单——它需要**启动一个独立的 ReAct 循环**：新建 messages、拼角色 prompt、过滤工具集。这种"启动一个新 Agent"的特殊逻辑，必须放在主循环里单独拦截。

`_react_loop` 里的分发逻辑：

```python
if name == "subagent":
    # 特殊：启动一个独立子循环
    result = _run_subagent(role=args["role"], task=args["task"], ...)
elif name in local_fns:
    # 普通：走路由表
    result = str(local_fns[name](**args))
```

### 3.3 共用 `_react_loop`：主循环和子循环同构

### 【屏显 · 代码 3-3 — _react_loop 的签名（agent_sub.py Part 4）】

```python
def _react_loop(
    messages: list,              # 该循环专属的 messages
    tools: list,                 # 本循环 LLM 能看到的工具列表
    local_fns: dict,             # 本循环可调用的本地函数字典
    system_prompt: str,          # 本循环的 system prompt
    tools_for_subagent: list,    # 若 LLM 调 subagent，传给子循环的工具集（已去掉 subagent）
    depth: int,                  # 递归深度（主 Agent = 0，Subagent = 1+）
    verbose: bool,
    indent: str = "",            # 打印缩进，让 subagent 轨迹嵌套可视化
) -> str:
```

### 【口播 · 主循环和子循环的差别只在三个参数】

这是 demo5 最关键的设计——**主 Agent 和 Subagent 跑同一个 `_react_loop`**，差别只在传入的参数：

| 参数 | 主 Agent | Subagent |
|---|---|---|
| `messages` | `[{"role":"user","content":user_input}]` | `[{"role":"user","content":task}]`——**全新一份** |
| `system_prompt` | 全局 `SYSTEM_PROMPT` | `build_subagent_system_prompt(role)`——**只拼角色** |
| `tools` | `ALL_TOOLS`（含 subagent） | `tools_for_subagent`（**去掉 subagent**，防递归） |
| `depth` | 0 | 1+ |

**核心洞察**：Subagent 的"独立性"不是靠新写一个循环，而是靠**传入独立的 messages/prompt/tools**。同一个 ReAct 循环，喂不同的 context，就是不同的 Agent。

### 3.4 `_run_subagent`：一次性生命周期

### 【屏显 · 代码 3-4 — _run_subagent 的核心（agent_sub.py Part 4）】

```python
def _run_subagent(role, task, tools, local_fns, depth, verbose) -> str:
    indent = "    " * depth   # 缩进打印，让 subagent 轨迹嵌套可视化

    sub_messages = [{"role": "user", "content": task}]          # ① 独立 messages
    sub_system_prompt = build_subagent_system_prompt(role)       # ② 独立 prompt

    final = _react_loop(
        messages=sub_messages,
        tools=tools,                                            # ③ 已去掉 subagent
        local_fns=local_fns,
        system_prompt=sub_system_prompt,
        tools_for_subagent=tools,
        depth=depth,
        verbose=verbose,
        indent=indent,
    )
    # ④ 循环结束 → messages / prompt 全部丢弃，只返回结果摘要
    return f"[Subagent · {role}] 任务：{task}\n结果：{final}"
```

### 【口播 · 四个关键设计】

1. **独立 messages**：`sub_messages` 是局部变量，函数返回后被 GC 回收——主 Agent 拿不到 Subagent 的内部对话
2. **独立 system_prompt**：`build_subagent_system_prompt(role)` 只拼角色，不注入 Rules / 记忆——保持专注
3. **工具集去 subagent**：`tools` 参数是主 Agent 在 `run_agent` 里算好的 `tools_for_subagent`——Subagent 看不到 subagent 工具，堵死递归
4. **一次性**：函数返回字符串，`sub_messages` / `sub_system_prompt` 随之销毁——下次主 Agent 再调 subagent，又是全新一份

### 【口播 · 关键观察】

注意五个点：

1. **主 Agent 识别出"相互独立"**——LLM 看到「1) 2)」编号列表 + 两个互不相干的子任务，主动派了 2 个 Subagent
2. **每个 Subagent 有独立的 messages**——`[Subagent · depth=1]` 缩进打印，内部工具调用（execute_bash / read_file）对主 Agent 透明
3. **Subagent 用完即销毁**——每个 Subagent 结束时打印「返回主 Agent，messages 即刻销毁」
4. **主 Agent 只看到结果摘要**——`[结果] [Subagent · 文件统计员] 任务：... 结果：...`，是 `_run_subagent` 拼的字符串，不是 Subagent 的内部 messages
5. **主 Agent 自己只迭代了 2 轮**——派 2 个 Subagent + 汇总结果。中间所有 execute_bash / read_file 都在 Subagent 内部跑

这就是 Subagent 的核心价值——**主 Agent 的 messages 保持干净**，所有子任务的中间结果都隔离在 Subagent 内部，结束后只留一段摘要。

## 第 4 章 机制二：Team（持久项目组）

### 【口播 · Subagent 的痛点】

Subagent 解决了"独立子任务分包"的问题，但遇到**有依赖 + 需通信 + 要质检**的场景就力不从心：

1. **干完结果就丢**——Subagent 的 messages 用完即销毁，不能累积记忆。如果同一个角色要被多次唤起（比如 Writer 被质检打回重写），每次都是从零开始
2. **不能互相通信**——Subagent 之间没有 inbox，A 的结果传不到 B（只能传回主 Agent，再由主 Agent 转述）
3. **不能质检打回**——Subagent 没有状态机，"不通过 → 重做"只能靠主 Agent 在 messages 里自己记

**Team 模式的升级**：把 Subagent 的"一次性函数"升级为"持久化对象"——Agent 有 name/role/messages/inbox，可以被 `chat()` 多次唤起，messages 跨多次 chat 累积。

### 4.1 Agent 类：从函数到对象

### 【屏显 · 代码 4-1 — Agent 类（agent_team.py Part 4）】

```python
class Agent:
    """一个有身份、有记忆、有收件箱的 Agent。"""

    def __init__(self, name, role, tools, local_fns, verbose=True):
        self.name        = name           # 固定身份（Subagent 是临时拼角色）
        self.role        = role
        self.tools       = tools
        self.local_fns   = local_fns
        self.inbox: list = []             # 收件箱：[(sender, message), ...]（Subagent 没有）
        self.messages: list = []          # 长期记忆（Subagent 是一次用完即丢）
        self.system_prompt = build_agent_system_prompt(name, role)

    def receive(self, sender, message):
        """往 inbox 里塞一条消息。下次 chat() 时会被消化。"""
        self.inbox.append((sender, message))

    def chat(self, task=None) -> str:
        """一次 chat = 消化 inbox → 追加 task → 走 ReAct 循环"""
        # 1) 消化 inbox：把所有未读消息包装成 user message 灌进 messages
        if self.inbox:
            for sender, msg in self.inbox:
                wrapped = f"[来自 {sender} 的消息] {msg}"
                self.messages.append({"role": "user", "content": wrapped})
            self.inbox.clear()
        # 2) 追加本次任务
        if task:
            self.messages.append({"role": "user", "content": task})
        # 3) 走 ReAct 循环（与 agent_sub.py 的 _react_loop 同构，只是挂到实例上）
        return self._react_loop()
```

### 【口播 · Agent vs Subagent 的三个关键升级】

| 维度 | Subagent（agent_sub.py） | Agent（agent_team.py） |
|---|---|---|
| **本质** | 函数 `_run_subagent()` | 对象 `class Agent` |
| **生命周期** | 一次性——返回即销毁 | **持久**——可以被 `chat()` 多次唤起 |
| **messages** | 局部变量，函数返回即 GC | **`self.messages` 跨多次 chat 累积** |
| **通信** | 无 inbox | **`self.inbox`——可被其他 Agent 塞消息** |
| **角色** | 临时拼 `role` 字符串 | **`self.name` + `self.role` 固定身份** |

**核心洞察**：Agent 的 `chat()` 方法每次被调用时，先把 inbox 里积压的消息消化进 `self.messages`，再追加新 task。这意味着 Agent 的 messages 是**长期记忆**——"我之前做了什么、别人给我发了什么"都累积在里面。

### 4.2 Team 类：协调器 + 状态机

### 【屏显 · 代码 4-2 — Team 类的 4 个核心动作（agent_team.py Part 5）】

```python
class Team:
    def __init__(self, tools, local_fns, verbose=True):
        self.agents: dict = {}
        self.stage = STAGE_RESEARCHING   # 当前流水线阶段

    def recruit(self, name, role) -> Agent:      # 招募成员
        agent = Agent(name=name, role=role, ...)
        self.agents[name] = agent
        return agent

    def send(self, sender, receiver, message):   # 一对一通信（依赖注入 / 质检反馈打回）
        self.agents[receiver].receive(sender, message)

    def broadcast(self, sender, message):        # 群发（成员完成任务后通报全员）
        for name, agent in self.agents.items():
            if name == sender: continue
            agent.receive(sender, message)

    def dismiss(self):                           # 解散团队
        self.agents.clear()
```

### 【口播 · 固定三角色流水线】

demo5 的 Team 不像旧 demo5 那样让 LLM 当项目经理动态拆任务——**固定 Researcher→Writer→Reviewer 三角色**，更聚焦教学：

- **Researcher**（研究员）：调研主题，输出要点列表
- **Writer**（撰稿人）：基于 Researcher 的结果写结构化研究报告（markdown）
- **Reviewer**（质检员）：验收 Writer 的报告，不通过则打回重做（最多 3 次）

### 4.3 状态机：任务级流转

### 【屏显 · 代码 4-3 — 状态机定义（agent_team.py Part 5）】

```python
# 流水线阶段
STAGE_RESEARCHING = "researching"
STAGE_WRITING     = "writing"
STAGE_REVIEWING   = "reviewing"
STAGE_REDOING     = "redoing"
STAGE_PASSED      = "passed"
STAGE_FAILED      = "failed"

MAX_REVIEW_ATTEMPTS = 3       # Writer 最多被质检打回 3 次
```

### 【屏显 · 图 4-1 — 状态机流转图】

```
researching → writing → reviewing → ┬→ passed (终态，报告输出)
                                   └→ redoing → writing → reviewing → ...
                                       (3 次不过 = failed)
```

### 【口播 · 状态机的作用】

状态机让"当前在哪一步、为什么不通过、要重做几次"**显式化、可观测**。没有状态机的话，"这是第几次质检、要不要打回"只能靠 LLM 在 messages 里自己记——一旦上下文压缩或轮次多了就乱。

`run_pipeline(topic)` 的主流程：

```python
def run_pipeline(self, topic):
    # Step 1: recruit 三角色
    self.recruit("Researcher", "研究员——调研主题，输出要点列表")
    self.recruit("Writer",     "撰稿人——基于研究员的要点写结构化研究报告")
    self.recruit("Reviewer",   "质检员——验收报告，不通过则打回重做")

    # Step 2: Researcher 调研
    self.stage = STAGE_RESEARCHING
    research_result = self.agents["Researcher"].chat(research_task)
    self.send("Researcher", "Writer", f"调研要点如下：\n\n{research_result}")

    # Step 3 + 4: Writer 写 → Reviewer 质检（循环）
    review_attempts = 0
    while True:
        self.stage = STAGE_WRITING if review_attempts == 0 else STAGE_REDOING
        writer_report = self.agents["Writer"].chat(write_task)
        self.send("Writer", "Reviewer", f"待验收的报告...{writer_report}")

        self.stage = STAGE_REVIEWING
        verdict_text = self.agents["Reviewer"].chat(review_task)
        passed, feedback = self._parse_verdict(verdict_text)
        review_attempts += 1

        if passed:
            self.stage = STAGE_PASSED
            break
        if review_attempts >= MAX_REVIEW_ATTEMPTS:
            self.stage = STAGE_FAILED
            break
        # 打回 Writer 重做
        self.stage = STAGE_REDOING
        self.send("Reviewer", "Writer", f"质检未通过。反馈：{feedback}")

    # Step 5: 把报告落盘 + dismiss
    ...
```

### 4.4 质检员 JSON 解析

### 【屏显 · 代码 4-4 — Reviewer 的 prompt + JSON 解析（agent_team.py Part 5）】

```python
review_task = (
    "请严格验收上条报告。检查：内容是否覆盖主题、结构是否清晰、"
    "是否有明显错误或空话。可以用 read_file / execute_bash 复查。\n\n"
    "严格只输出 JSON（不要 markdown 代码块、不要任何解释）：\n"
    '{"pass": true|false, "feedback": "若不通过，说明具体怎么改"}'
)
verdict_text = self.agents["Reviewer"].chat(review_task)
# 解析 JSON——prompt 已要求纯 JSON，解析失败默认不通过
try:
    v = json.loads(verdict_text.strip())
    passed = bool(v.get("pass") is True)
    feedback = v.get("feedback") or ""
except json.JSONDecodeError:
    passed = False
    feedback = f"[质检员输出不是合法 JSON，默认不通过] 原文：{verdict_text}"
```

### 【口播 · 防御式解析】

质检员的输出必须是严格 JSON——这是 Team 和主 Agent 之间的契约。但 LLM 可能输出带 markdown 代码块或多余解释，所以解析失败时**默认不通过**，把原文塞进 feedback——这样 Writer 重做时能看到质检员到底说了什么（哪怕是乱码）。

### 【口播 · 关键观察】

注意六个点：

1. **三角色招募**——Researcher / Writer / Reviewer 三个 Agent 在 Team 初始化时一次性 recruit
2. **inbox 通信**——`[Agent · Writer] 收到来自 Researcher 的消息：调研要点如下...`——Researcher 的调研结果通过 `team.send()` 塞进 Writer 的 inbox
3. **inbox 消化时机**——Writer 在 `chat()` 开始时才消化 inbox：`[Agent · Writer] 消化收件箱：[来自 Researcher 的消息]...`——inbox 是异步通信，chat 时才同步
4. **Reviewer 实际复查文件**——Reviewer 用 `read_file` 读了 Writer 写的 `Python_GIL_研究报告.md`，不是只看 Writer 汇报的摘要
5. **状态机流转可见**——`[stage: researching] → [stage: writing] → [stage: reviewing] → [stage: passed]`——每一步状态变化都打印
6. **质检 JSON 严格解析**——Reviewer 输出 `{"pass": true, "feedback": ""}`，一次通过；如果输出带 markdown 代码块会被默认判为不通过

**特别注意 Writer 的 messages 累积**：Writer 第一次 chat 时，messages 里先有 `[来自 Researcher 的消息]`（inbox 消化），再有本次 task。如果被质检打回，Writer 第二次 chat 时，messages 会累积成 `[Researcher 的要点] + [第 1 次写报告的 task + 结果] + [Reviewer 的打回反馈] + [第 2 次写的 task]`——这就是"长期记忆"。

## 第 5 章 Subagent vs Team 何时用

### 【屏显 · 表格 5-1 — 对照表】

| 维度 | Subagent（agent_sub.py） | Team（agent_team.py） |
|---|---|---|
| **本质** | 一次性函数 `_run_subagent()` | 持久对象 `Agent` + 协调器 `Team` |
| **messages** | 独立、局部、用完即销毁 | 每个 Agent 独立累积、跨多次 chat 保留 |
| **通信** | 无 inbox——结果只返回主 Agent | **inbox + send/broadcast**——Agent 间可塞消息 |
| **角色** | 临时拼 `role` 字符串 | **固定 name + role**——身份持久 |
| **状态机** | 无 | **researching→writing→reviewing→passed/failed** |
| **质检** | 无 | **Reviewer JSON 闸门 + 最多 3 次打回** |
| **递归防护** | **工具集去 subagent**——子循环看不到 subagent | 不需要——Team 不靠 LLM 调 subagent 工具协调 |
| **代码量** | ~350 行（4 Part） | ~500 行（6 Part） |
| **工业对应** | Claude Code Task tool / Cursor agent | AutoGen / CrewAI |

### 【口播 · 何时用 Subagent】

**独立无依赖的子任务**——典型场景：

- 「1) 统计文件数；2) 读某文件首行」——两个子任务互不相干
- 「帮我查这三个互不相关的关键词各自的意思」——每个关键词派一个 Subagent
- 「并行扫描三个目录」——每个目录派一个 Subagent

判断标准：**子任务之间不需要互相传结果**，每个子任务独立干完就能交差。

### 【口播 · 何时用 Team】

**有依赖 + 需通信 + 要质检的任务**——典型场景：

- 研究报告流水线（Researcher→Writer→Reviewer）
- 代码实现流水线（需求分析→编码→代码审查→修改）
- 文档生产流水线（收集素材→撰写→校对→排版）

判断标准：**后一步要用前一步的结果**、**角色之间有明确的通信协议**、**结果需要质检闸门**。

### 【口播 · 混合使用的边界】

真实工业 Agent 往往**两者结合**：

- Team 内部的某个 Agent（比如 Researcher）遇到独立子任务时，可以再派 Subagent——但 demo5 为了教学清晰，agent_team.py **不含 subagent 工具**，Team 的协调全靠外部编排器（Team 类），不靠 LLM 调 subagent 工具
- Claude Code 自己就是这么做的——Task 工具派 Subagent，同时支持多 Agent 协作的工作流（如 Plan mode + Task 结合）

### 【口播 · 两个机制的边界】

Subagent 和 Team 都是「多 Agent」，但设计哲学不同：

| 机制 | 设计哲学 | 适合 |
|---|---|---|
| **Subagent** | **无状态外包**——独立 context、干完即销毁 | 相互独立的子任务 |
| **Team** | **有状态项目组**——持久对象、inbox 通信、状态机驱动 | 有依赖 + 需通信 + 要质检的流水线 |

简单说：**Subagent 是「临时工」，Team 是「正式项目组」**。

### 【口播 · 下一节预告：demo6 约束/安全】

demo5 的多 Agent 能力很强——但注意一个细节：**两个文件里的 `execute_bash` 都用 `shell=True` 且无任何过滤**。Subagent 可以执行 `rm -rf /`，Team 的 Writer 可以执行 `dd of=/dev/sda`——能力越大，风险越大。

demo6 引入**约束轴**——给 Agent 的"手脚"加三层安全栈：

- **Permission**：allow/deny/ask 规则匹配（如 `Bash(rm:*)` → deny）
- **Sandbox**：Bash 执行隔离（read-only / write-only / none）
- **Hook**：PreToolUse / PostToolUse 事件回调（外部脚本拦截）

这正是 Claude Code 的 Permission 系统、Cursor 的 sandbox、各种 Agent 框架的 hook 机制背后的原理。
