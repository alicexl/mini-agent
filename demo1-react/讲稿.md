## 第 1 章 先说结论

### 【口播 · 三维度对比】

要理解 Agent，最快的方法是把它跟普通的 AI 对话放在一起对比。我们看 3 个维度——交互模式、能力边界、状态管理。

### 【屏显 · 表格 1-1】

| 维度 | 普通对话 | Agent |
|---|---|---|
| **交互模式** | 一问一答，**靠用户问题驱动**；模型无自主性，必须由用户触发 | 给一个**目标**就能自主推进；多轮循环、自主选择下一步、自觉逼近目标 |
| **能力边界** | 只能生成文本（多模态模型支持的文件） | **工具扩展空间巨大**——本地 CLI、自定义函数、远程 MCP 都能接 |
| **状态管理** | 靠模型服务的会话状态做记忆 | **本地维护完整 messages 历史**，不完全依赖大模型的双向窗口 |

### 【口播 · 三维度总结】

三个维度合起来，就是 Agent 和 Chatbot 的全部区别——**多了工具扩展、多了目标驱动的循环、多了本地的状态管理**。

### 【口播 · 三要素公式】

我把 Agent 浓缩成一个公式：

> ### Agent = LLM × 工具 × 循环
>
> - **LLM** = 思考能力 → **大脑**
> - **工具** = 本地工具 + MCP 等远程能力 → **手脚**
> - **循环** = 任务分解 + 迭代，直到目标达成

普通对话只是「你问我答」；Agent 是「**为了一个目标，努力去想办法、去完成它**」。

## 第 3 章 逐层解读

### 3.1 Part 1 — LLM 客户端

### 【口播】

第一层目标：**调通大模型**。

本 demo 默认走**智谱 BigModel 的 Anthropic 兼容网关** + `glm-5.2` 模型。关键是——Anthropic SDK 的调用接口是标准，换成官方 API 或别的兼容网关，调用代码一行不用改。

用户只需配置 API Key，三种方式（优先级递减）：
1. 改下面代码里的 `API_KEY` 变量（持久化，推荐）
2. 设环境变量 `ANTHROPIC_API_KEY`
3. 都没设 → 运行时交互式输入（不持久化，下次还要再输）

### 【屏显 · 代码 3-1】

```python
import os
from anthropic import Anthropic

# ↓↓↓ 只需改这一行 ↓↓↓
API_KEY = ""

# 默认配置（一般无需修改）
BASE_URL = "https://open.bigmodel.cn/api/anthropic"   # 智谱 BigModel 兼容网关
MODEL    = "glm-5.2"

def load_config() -> dict:
    """环境变量优先于代码默认值"""
    return {
        "api_key":  os.environ.get("ANTHROPIC_API_KEY") or API_KEY,
        "base_url": BASE_URL,
        "model":    MODEL,
    }

def init_client() -> None:
    """模块级 client 初始化；没配 Key 时交互式提示输入"""
    global client
    config = load_config()
    if not config["api_key"]:
        config["api_key"] = input("请输入 API Key: ").strip()
    client = Anthropic(api_key=config["api_key"], base_url=config["base_url"])
```

### 【口播】

一句话总结这一层：**SDK 照抄就行**。重点是后面三层——工具和循环才是 Agent 的灵魂。

### 3.2 Part 2 — 工具定义

### 【口播】

第二层是**工具说明书**。大模型本身不会执行代码——它只是在每次请求里看到这份说明书，然后**知道自己可以请求调用**哪个函数、用什么参数。真正的执行发生在我们本地。

### 【屏显 · 代码 3-2】

```python
TOOLS = [
    {
        "name": "execute_bash",
        "description": "执行任意 shell 命令，可用于文件操作、系统命令等",
        "input_schema": {
            "type": "object",
            "properties": {
                "command": {"type": "string", "description": "要执行的 shell 命令"}
            },
            "required": ["command"],
        },
    },
    # read_file / write_file 同理
]
```

### 【口播 · schema 三要素】

每个工具的 schema 有三部分：

1. **`name`**——大模型调用时通过这个名字找到工具
2. **`description`**——大模型做决策时要和它比对。模型不一定一次选对，可能先看 description 判断这个工具能不能做、要不要试
3. **`properties`**——告诉大模型这个工具有哪些参数、参数是什么类型、必填还是可选

这样大模型就知道怎么样去传递参数。大模型按这份 schema **吐出结构化调用指令**（tool_use），本地解析后调函数。

### 【口播 · Function Calling 是训练能力，不是 prompt 魔法】

注意这里的关键认知：**Function Calling 是大模型经过训练的能力，不是 prompt 魔法**。

模型在训练阶段见过大量「工具调用」样本，所以它认识这份 schema，会按格式吐出结构化的调用指令。我们在 prompt 里写再多"你有工具"，模型没经过训练也用不起来。

### 【屏显 · 表格 3-1 — 协作分工】

这是一个**协作分工**——模型只负责"想"，我们负责"做"：

| 角色 | 职责 |
|---|---|
| **大模型**（大脑） | 思考、决策、生成工具调用指令（tool_use） |
| **本地 Agent + 工具**（手脚） | 解析指令、执行真正的函数、返回结果（tool_result） |

流程是：
1. 我们告诉大模型：「我有这些能力」（tools 参数）
2. 大模型思考后回吐：「**你**去调这个函数，参数是这样」（tool_use block）
3. 我们解析后执行真正的函数
4. 把结果再告诉大模型（tool_result）
5. 大模型判断结果是否符合预期，决定下一步

### 3.3 Part 3 — 工具实现 + 路由表

### 【口播】

第三层是真正的执行函数。这一层有两个关键设计点——**超时**和**错误字符串化**：

### 【屏显 · 表格 3-2 — 工具实现的两个设计点】

| 设计点 | 代码 | 不这么做的后果 |
|---|---|---|
| **错误也字符串化返回** | `return f"[错误] {e}"` | 大模型不知道失败，会重复试同一个命令 |
| **设置 60 秒超时** | `timeout=60` | 一个 `sleep 9999` 卡死整个 Agent |

### 【屏显 · 代码 3-3】

```python
def execute_bash(command: str) -> str:
    try:
        result = subprocess.run(
            command,
            shell=True,          # ← 风险换能力
            capture_output=True,
            text=True,
            timeout=60,          # ← 防死循环
        )
        # 错误信息也字符串化 → 让大模型看到后自己调整策略
        ...
    except subprocess.TimeoutExpired:
        return "[错误] 命令执行超时（60 秒）"
```

### 【口播 · 路由表】

接下来一个**关键设计**——**路由表**：从工具 name 找到实际函数。

### 【屏显 · 代码 3-4】

```python
# 路由表：工具名 → 实际函数（调度核心）
AVAILABLE_FUNCTIONS = {
    "execute_bash": execute_bash,
    "read_file":    read_file,
    "write_file":   write_file,
}
```

### 【口播】

为什么用字典、不用 `if/elif`？

**两个理由**：
1. **单点注册**——加减工具只改一处
2. **可扩展**——把「调用工具」从散落的 if/elif 收敛成一个点，既方便在调用点注入通用逻辑（日志、权限、计时），也方便在函数定义端贴装饰器做专属处理

### 3.4 Part 4 — Agent 主循环

### 【口播】

最后一层是 Agent 的**心脏**：主循环。每一轮做四件事——决策、判停、行动、感知。

### 【屏显 · 代码 3-5 — 主循环骨架】

```python
MAX_ITERATIONS = 30  # 防止大模型陷入死循环

def run_agent(user_input: str, verbose: bool = True) -> str:
    messages = [{"role": "user", "content": user_input}]

    for loop_idx in range(1, MAX_ITERATIONS + 1):
        # 1. 决策：大模型思考下一步
        response = client.messages.create(
            model=MODEL, max_tokens=4096,
            system=SYSTEM_PROMPT, tools=TOOLS, messages=messages,
        )

        # 2. 判停：stop_reason 决定循环是否继续
        if response.stop_reason != "tool_use":
            return "".join(b.text for b in response.content if b.type == "text")

        # 3. 行动：执行工具（每个 tool_use 带 ID）
        messages.append({"role": "assistant", "content": response.content})
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                fn = AVAILABLE_FUNCTIONS.get(block.name)   # 未知工具防御
                if fn is None:
                    result = f"[错误] 未知工具: {block.name}"
                else:
                    result = fn(**block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,  # ← Tool ID 精确匹配
                    "content": result,
                })

        # 4. 感知：工具结果作为 user 消息追加，下一轮大模型就能看到
        messages.append({"role": "user", "content": tool_results})

    return "[错误] 超过最大循环次数（{}）".format(MAX_ITERATIONS)
```

> **注**：`verbose=True` 时还会在每轮打印决策与工具结果预览（教学演示用，上方骨架省略了这部分打印代码）。

### 【口播 · MAX_ITERATIONS】

**为什么是 30？**——实测大部分任务在 10 轮内完成，30 是 3 倍安全垫。

**为什么必须有？**——大模型会陷入死循环。比如 `execute_bash("cat 不存在的文件")` 失败 → 换个姿势再 `cat` → 还是失败 → 再 `cat`...模型不一定能自己跳出来。设了上限，程序才能保证最终停下来。

### 【口播 · Tool ID 的作用】

注意循环里每个 `tool_use` block 都带一个 `tool_use_id`，result 里也要带同一个 ID 回去。**ID 的根本作用是协议配对**——让 result 能精确指回是哪一次 call，这是 Anthropic API 的硬性要求，**串行单调用也需要**。它最直观的价值体现在**并行工具调用**场景：一次响应里返回多个 tool_use，没有 ID 就没法把多个 result 一一对应回各自的 call。

### 【口播 · stop_reason 是循环的"门"】

循环退出条件靠 `stop_reason`——大模型返回的"信号灯"：

### 【屏显 · 表格 3-3 — stop_reason 三种值】

| `stop_reason` | 含义 | Agent 行为 |
|---|---|---|
| `"tool_use"` | 模型要调工具 | 执行工具，继续循环 |
| `"end_turn"` | 模型认为任务完成 | 退出循环，返回文本 |
| `"max_tokens"` | 输出长度到上限被截断 | 隐式停止（也算异常退出） |

我们代码里 `if response.stop_reason != "tool_use"` 就是这扇门——不是 `tool_use` 就走退出分支。三种 stop_reason 都覆盖到了。

```
│ [user]   统计...                                     │
│ [assistant] tool_use(execute_bash, "find ... | wc -l")│
│ [user]   tool_result("1")      ← 工具结果            │
└─────────────────────────────────────────────────────┘
                            │

                  大模型看到「1」+ 原始任务
                  → 决定调用 write_file
                            │

                            │
                            ▼
第 3 轮 messages（5 条，追加 2 条）：
┌─────────────────────────────────────────────────────┐
│ [user]   统计...                                     │
│ [assistant] tool_use(execute_bash)                   │
│ [user]   tool_result("1")                            │
│ [assistant] tool_use(write_file, count.txt, "共1个...")│

└─────────────────────────────────────────────────────┘
                            │

                  → 判断任务完成

                  → 返回最终文本
                            │

                   - 当前目录下共有 1 个 Python 文件
                   - 结果已写入 count.txt 文件」
```

### 【口播 · 关键观察】

三个细节：

1. **每一轮 messages 都在变长**——第 1 轮 1 条 → 第 2 轮 3 条 → 第 3 轮 5 条。每轮追加 2 条：一条 assistant（LLM 的 tool_use 决策）、一条 user（本地执行后的 tool_result）。

2. **大模型自己选了 `find ... | wc -l`**——我们没教它具体命令，它根据 description「执行任意 shell 命令」+ 任务描述自己推出来的。注意它还加了 `-maxdepth 1`——只统计当前目录，不递归子目录。这是 LLM 的常识，不是我们写死的。

3. **`stop_reason` 是循环的门**——第 1、2 轮返回 `tool_use`（要调工具），第 3 轮返回 `end_turn`（任务完成）。代码里 `if response.stop_reason != "tool_use"` 就是这扇门——不是 `tool_use` 就走退出分支。

这就是 Agent **比 Chatbot 贵** 的原因——**每轮都把整个上下文重发给大模型**。每轮成本线性增长，**累计总成本是平方级**（O(n²)）。

但这也是它的力量来源：每一轮大模型都看到完整历史，所以能基于"我做过什么、结果是什么"做下一步决策。**短期记忆就是这样产生的。**

**每轮全量重发还有一个隐藏的好处**——**大模型可以中途热切换**。因为每轮都把完整上下文发过去，下一轮换成另一个模型也能正常决策，不影响 Agent 执行任务。这是 messages 全量重发设计带来的副作用优势。

**也埋了一个雷**：messages 会**无限增长**，遇到复杂任务时整个上下文可能撑爆大模型的窗口，请求就会失败。这就是后面**记忆模块**要解决的核心问题——既要保留跨任务的长期记忆，又要做压缩防止撑爆窗口。

## 第 5 章 总结和展望

### 5.1 Agent 的本质

### 【口播】

Agent 是什么？它是一个**执行的过程**，循环做三件事——**决策 → 行动 → 感知**：

1. **决策**：把任务丢给大模型，让它思考下一步（Reasoning）
2. **行动**：大模型决定调工具，本地执行（Acting）
3. **感知**：工具结果反馈给大模型，让它看到成功/失败/数据（Observation）

然后回到第 1 步继续决策——这就是 **ReAct**（Reasoning + Acting）循环，一直迭代到大模型判断任务完成。

### 【屏显 · ReAct 循环图】

```
        ┌─────────────────────────────┐
        │                             │
        ▼                             │
   ┌─────────┐                  ┌─────────┐
   │  决策   │  ← messages ←    │  感知   │
   │ (LLM)   │                  │ (结果)  │
   └─────────┘                  └─────────┘
        │                             ▲
        │  tool_use                   │
        ▼                             │
   ┌─────────┐                        │
   │  行动   │  ── tool_result ──────┘
   │ (工具)  │
   └─────────┘
```

### 【口播】

**决策 → 行动 → 感知 → 再决策**。不管是什么大模型、不管是什么 Agent——底层都是这个循环。

### 5.2 后续 demo 的扩展轴

### 【口播】

### 【屏显 · 表格 5-1 — demo1–6 扩展轴路线图】

| Demo | 扩展轴 | 解决的问题 |
|---|---|---|
| **demo1** | **ReAct 循环**（base） | Agent 最小心跳——决策 × 工具 × 循环 |
| **demo2** | **记忆** | 跨任务长期记忆（`agent_memory.md`）+ 上下文动态压缩（`compact_messages`） |
| **demo3** | **工具** | 本地工具不够 → MCP 让工具跨进程/跨机器/跨语言服务化 |
| **demo4** | **规划** | todo_write 让 LLM 拆步骤显式化；Skill 把常见任务的工作流预固化成技能 |
| **demo5** | **多 Agent** | 单 Agent 上下文膨胀 → Subagent 一次性外包独立任务 + Team 持久协作（状态机 + inbox 通信 + 质检） |
| **demo6** | **约束** | 工具太自由（`rm -rf`） → Permission 规则 + 执行隔离 + Hook 回调三层安全栈 |
