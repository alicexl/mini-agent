## 第 1 章 整体把握：demo4 做了什么

### 【口播 · 回顾前三节】

> ### demo1 = LLM × 工具 × 循环
> ### demo2 = demo1 × 记忆 × 规划
> ### demo3 = demo2 × Rules × MCP（plan 自动决策）

到了 demo3，我们做完了**规划模式的自动决策**、加了**真正的 MCP 工具接入方式**、加了**本地的 Rules**。但 demo3 仍有两个问题：

1. **上下文越来越长**：不管拆多少个 step，所有 step **共享一份 messages**，子任务越多，prompt 越膨胀
2. **必须按顺序执行**：step 是串行的，没办法并行——一个人按步骤干完所有事

### 【口播 · demo4 的解法】

这时候我们想到——**能不能把任务拆了之后，做成"子 Agent"的形式**？每个子 Agent 独立干活，**如果可以的话并行执行**。这就是 demo4 的核心：**subagent**。

### 【屏显 · 表格 1-1 — demo4 vs demo3】

| 维度 | demo3 | demo4 |
|---|---|---|
| **执行单元** | 单 Agent + 共享 messages 多步推进 | **主 Agent + 多个一次性 Subagent** |
| **上下文** | 所有 step 共享一份 messages | **每个 Subagent 独立 messages，结束即销毁** |
| **角色身份** | 全程一个 system_prompt | **每个 Subagent 有自己的 role / system_prompt** |
| **生命周期** | 主 Agent 持续整个 task | **Subagent 一次性：派生 → 干活 → 返回 → 消亡** |

### 【口播 · 一减一加】

注意——**demo4 的核心改动只有两件**：

- **减法**：
  - **去掉 MCP**——demo3 为了演示 MCP 起了一个独立服务进程，demo4 不需要这套，把三件小工具（add / multiply / weather）从 MCP Server **搬回本地函数**，避免多起一个进程
  - **去掉 plan 工具**——plan 和 subagent 都是"拆任务"，两个一起用会让逻辑嵌套变复杂（plan 拆 step、step 里再派 subagent），**本次只验证 subagent 这一个场景**，所以 plan 暂时拿掉
- **加法**：
  - **新增 `subagent` 本地工具**——大模型通过调它派生一个独立的 Agent 循环

新公式：

> ### demo4 = demo3 × Subagent（分工合作）

### 【屏显 · 表格 1-2 — demo4 的 7 个本地工具】

| 工具名 | 来源 | 用途 |
|---|---|---|
| `execute_bash` | 沿用 | 执行 shell 命令 |
| `read_file`    | 沿用 | 读文件 |
| `write_file`   | 沿用 | 写文件 |
| `add`          | **从 MCP Server 搬回本地** | 加法 |
| `multiply`     | **从 MCP Server 搬回本地** | 乘法 |
| `weather`      | **从 MCP Server 搬回本地** | 天气查询 |
| `subagent`     | **demo4 新增** | 委派独立 Subagent |

> 后续 demo 都不会再起 MCP 服务——demo3 那套只是为了演示，真实场景都用本地工具。

## 第 2 章 Subagent：本质、schema、实现、设计决策

### 2.1 本质：派生一个独立的 Agent 循环

### 【口播】

Subagent 不是什么新协议、新框架——**它就是一个普通工具，调一下它，主 Agent 内部启动一个新的 Agent 循环**。这个新循环和主循环**用的是同一份 `_react_loop` 代码**，区别只在于三件事：

1. **独立的 messages**：从空列表开始，看不到主 Agent 的历史对话
2. **独立的 system_prompt**：基于 `role` 字段拼一个角色化提示
3. **独立的工具集**：去掉 `subagent` 工具本身，防止无限递归

干完活后，subagent 把结果摘要返回给主 Agent，**它的 messages 和 system_prompt 立刻销毁**——一次性。

### 【屏显 · 图 2-1 — Subagent 的生命周期】

```
主 Agent（持续整个 task）
   │
   ├─ LLM 决定调用 subagent(role="加法专家", task="算 35+47")
   │       │
   │       ▼
   │   ┌──────────────── Subagent（一次性）────────────────┐
   │   │  · system_prompt = "你是加法计算专家..."           │
   │   │  · messages = [{user: "算 35+47"}]                │
   │   │  · tools = [bash, read, write, add, multiply, ...]│  ← 去掉 subagent
   │   │                                                    │
   │   │  ReAct 循环：add(35, 47) → 82 → end_turn          │
   │   │                                                    │
   │   │  返回："35 + 47 = 82"                              │
   │   └────────────────────────────────────────────────────┘
   │       │
   │       ▼ messages 即刻销毁
   ├─ tool_result 灌回主 Agent 的 messages
   │
   └─ 继续主循环……
```

### 【口播 · 项目经理 vs 临时工】

主 Agent 像一个**项目经理**——简单任务自己干，多任务场景就派出去。Subagent 像一个**临时工**：你给它一个身份（"你是测试工程师"）、一个任务（"测这个函数"），它用自己独立的脑子干活（独立 messages + system_prompt），干完汇报结果就**走人**（messages 销毁，不留记忆）。

### 2.2 schema：role + task + 强制规则

### 【屏显 · 代码 2-2 — subagent 工具定义】

```python
{
    "name": "subagent",
    "description": (
        "委派一个独立的 Subagent 来完成子任务。\n"
        "**强制规则**：当用户输入包含「1) 2) 3)」这种编号列表、且列出 2 个及以上子任务时，"
        "**必须**为每个子任务**调用一次 subagent 工具**，由对应 Subagent 完成——"
        "**禁止**主 Agent 自己在循环里连调 add/multiply/weather/write_file 等工具来直接做完。"
        "（即使主 Agent 自己做更省 token，也必须委派——这是 demo4 的演示要求。）\n"
        "Subagent 拥有独立的角色（system_prompt）和独立的上下文（messages），"
        "执行完返回结果摘要后即消亡。\n"
        "不适合 subagent 的场景：后一步要用前一步结果的链式任务（例如「先 a+b，再把结果乘 8」），"
        "这种应让主 Agent 自己顺序完成。"
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "role":  {"type": "string", "description": "Subagent 的角色，例如「加法计算专家」/「Python 工程师」"},
            "task":  {"type": "string", "description": "交给 Subagent 完成的具体任务描述"},
        },
        "required": ["role", "task"],
    },
}
```

### 【口播 · 为什么 description 写得这么"硬"]

注意这段 description 写得很硬——「**强制规则**」「**必须**」「**禁止**」。原因很直接：**演示稳定性**。

如果不这么写，LLM 经常会"嘴上说派 subagent"、实际却在主循环里直接把 add/multiply/weather 一口气调完——表面更省 token，但 demo4 的 subagent 场景就演不下去。

> **这种写法不是必须的**——真实生产环境里，是否用 subagent 应该由 LLM 自主判断；demo4 为了让"何时委派"可预测、可复现，才把判断标准写成了硬规则。**把判断标准说清楚，LLM 自己会决策**——这是 description 作为"使用提示"的力量。

### 2.3 system_prompt：角色化 + 专注

### 【屏显 · 代码 2-3 — build_subagent_system_prompt】

```python
def build_subagent_system_prompt(role: str) -> str:
    return (
        f"你是一个被委派来的 Subagent。你的角色是：**{role}**。\n"
        f"请专注于交给你完成的任务，做完后用一两句话汇报结果。"
    )
```

### 【口播 · 为什么不注入 Rules / 记忆】

| 主 Agent system_prompt | Subagent system_prompt |
|---|---|
| 基础说明 + **Rules** + **历史记忆** | **只拼角色 + "专注"指令** |

- **不注入 Rules**：Subagent 已经有角色约束行为，Rules 是更全局 / 框架级的规范——demo4 简化掉，避免 prompt 拉长
- **不注入记忆**：Subagent 干的是一次性临时活，不需要全局长期记忆，避免无关历史干扰

> 真实场景里 Subagent 也可以遵循全局 Rules（甚至应该遵循），demo4 只是为了讲清独立 subagent 机制而省略了。

### 2.4 实现：派生独立 ReAct 循环

### 【屏显 · 代码 2-4 — _run_subagent 实现】

```python
def _run_subagent(role, task, tools, local_fns, depth, verbose):
    """启动一个独立的 Subagent ReAct 循环"""
    indent = "    " * depth   # 缩进打印让嵌套轨迹可视化

    sub_messages = [{"role": "user", "content": task}]   # ← 独立 messages
    sub_system_prompt = build_subagent_system_prompt(role)  # ← 独立 prompt

    final = _react_loop(                                  # ← 与主 Agent 同一个 ReAct
        messages=sub_messages,
        tools=tools,                                       # ← 已去掉 subagent 工具
        local_fns=local_fns,
        system_prompt=sub_system_prompt,
        tools_for_subagent=tools,
        depth=depth,
        verbose=verbose,
        indent=indent,
    )
    # ← sub_messages / sub_system_prompt 在函数返回时即被丢弃

    return f"[Subagent · {role}] 任务：{task}\n结果：{final}"
```

### 【口播 · 主循环与子循环的差别只在一处】

代码很短——主 Agent 和 Subagent **跑同一个 `_react_loop`**，差别只有三个"不一样"：

- **拼的 system_prompt 不一样**（主拼基础+Rules+记忆；子拼角色）
- **拼的 tools 不一样**（主含 subagent；子去掉 subagent）
- **各自持有的 messages 不一样**（完全隔离）

最后返回的字符串——`[Subagent · 角色] 任务：xxx\n结果：xxx`——会被作为 `tool_result` 灌回主 Agent 的 messages。**主 Agent 只看到这个摘要，看不到 Subagent 内部的 tool_use 细节**。

### 2.5 三个关键设计决策

| # | 决策 | 一句话解释 |
|---|---|---|
| 1 | **独立 messages** | 只看自己的任务，干完即销毁——避免上下文膨胀 |
| 2 | **角色化 system_prompt** | 给一个身份，让它在专业领域内行动（工业级 Agent 的核心机制） |
| 3 | **工具集去掉 subagent** | 防递归：`[t for t in all_tools if t.get("name") != "subagent"]` 一行堵死 |

> 决策 1、2 已经在 2.1 / 2.3 讲透，决策 3 单纯就是防递归——**主能调 subagent，子不能再调**，避免 A→B→C→D 无限派生烧光 token。

## 第 3 章 真实案例：四个独立任务派给 Subagent

### 【口播】

理论讲完，跑一个真实案例。给主 Agent 一个**列表形式的多任务**——加法、乘法、查天气、写文件，看它会不会自动委派。

### 3.1 用户输入

### 【屏显 · 用户输入】

```
任务如下：1) 计算 35+47；2) 计算 65*9；3) 查上海天气；4) 把 "hello" 写入 hello.txt
```

### 【口播】

主 Agent 启动 ReAct 循环——大模型看到工具列表里有 `subagent`，又看到任务是「1) 2) 3) 4)」这种列表形式，主动**为每个子任务各派一个 subagent**，每个负责一件事。

### 3.2 真实运行轨迹（节选）

```
============================================================
主 Agent 启动 ReAct 循环
============================================================
[messages] 当前 1 条消息
  [0] user     : 任务如下：1) 计算 35+47；2) 计算 65*9；3) 查上海天气；4) ...

  [LLM 思考] 好的，这 4 个任务彼此独立，我将并行派发 subagent 同时处理！
  [LLM] subagent({'role': '加法计算专家', 'task': '计算 35+47 的结果...'})
  ──────────────────────────────────────────────────────────
  [Subagent · depth=1] role='加法计算专家'
  ──────────────────────────────────────────────────────────
    [LLM] add({'a': 35, 'b': 47})
    [结果] 35 + 47 = 82
    [迭代 2 完成] 35 + 47 的计算结果为 **82**。
  [Subagent · depth=1] 返回主 Agent，messages 即刻销毁

  [LLM] subagent({'role': '乘法计算专家', 'task': '计算 65*9 的结果...'})
  ──────────────────────────────────────────────────────────
    [LLM] multiply({'a': 65, 'b': 9})
    [结果] 65 × 9 = 585
    [迭代 2 完成] 65 × 9 的最终答案是 **585**。
  [Subagent · depth=1] 返回主 Agent，messages 即刻销毁

  [LLM] subagent({'role': '天气查询助手', 'task': '使用 weather 工具查询「上海」的天气...'})
  ──────────────────────────────────────────────────────────
    [LLM] weather({'city': '上海'})
    [结果] 上海 今天天气：多云，气温 22°C
    [迭代 2 完成] 上海今天：多云，22°C。
  [Subagent · depth=1] 返回主 Agent，messages 即刻销毁

  [LLM] subagent({'role': '文件写入助手', 'task': '将字符串 "hello" 写入 hello.txt...'})
  ──────────────────────────────────────────────────────────
    [LLM] write_file({'path': 'hello.txt', 'content': 'hello'})
    [结果] [成功] 文件已写入: hello.txt (5 字符)
    [迭代 2 完成] 已写入 hello.txt。
  [Subagent · depth=1] 返回主 Agent，messages 即刻销毁

  [迭代 2 完成] 4 个子任务已全部并行完成！

助手: 4 个子任务全部完成：

       | # | 任务 | 结果 |
       |---|------|------|
       | 1 | 35 + 47 | **82** |
       | 2 | 65 × 9 | **585** |
       | 3 | 上海天气 | **多云，22°C** |
       | 4 | 写入 hello.txt | ✅ 已写入（5 字符） |
```

### 【口播 · 关键看 LLM 怎么决策】

注意第一行 `[LLM 思考]`——这是大模型第一轮的"自言自语"：

> **"好的，这 4 个任务彼此独立，我将并行派发 subagent 同时处理！"**

LLM 读了 subagent 工具 description 里的强制规则——「**当用户输入包含 1) 2) 3) 这种编号列表、且 2 个及以上子任务时，必须为每个子任务调用一次 subagent 工具**」，再看到用户输入正是这种列表形式，**自己得出了"应该委派"的结论**。

### 【口播 · 三点观察】

| # | 观察 | 启示 |
|---|---|---|
| 1 | **LLM 思考明确说"4 个任务独立 → 派 subagent"** | description 把判断标准说清楚，LLM 自己得出正确结论 |
| 2 | **每个 subagent 只调一次工具就 end_turn** | 角色化 system_prompt + 任务内聚让 Subagent 极其专注——做完即停 |
| 3 | **subagent 之间看不到对方** | 每个 Subagent 的 messages 都是新建的——加法专家完全不知道乘法专家的存在 |

### 【口播 · 关于"并行"的一个说明】

注意——**虽然主 Agent 让 4 个 subagent "并行派发"，但 demo4 实际是串行执行的**（一个 subagent 跑完才跑下一个）。这是为了简化设计方案。

但即使串行执行，subagent 在**上下文管理**上的价值依然成立——**主 Agent 的 messages 只追加 4 个 subagent 的结果摘要**，每个 subagent 内部的 tool_use 细节**全部丢弃**，主 Agent 上下文保持极短，token 消耗显著降低。

### 3.3 简单任务不委派

### 【口播】

刚才看的是"四个独立任务"的委派场景。但日常用户提问大多是**单步就能搞定**的，比如「查上海天气」。这种场景走的是**最普通的 ReAct**——LLM 直接调 weather，不调 subagent。

### 【屏显 · 简单查询的真实运行轨迹】

```
用户: 查上海天气

============================================================
主 Agent 启动 ReAct 循环
============================================================
  [LLM] weather({'city': '上海'})              ← 直接调本地工具，没委派 subagent
  [工具 · 本地]  weather({'city': '上海'})
  [结果] 上海 今天天气：多云，气温 22°C
  [迭代 2 完成] 上海：多云，22°C

助手: 上海天气：多云，22°C
```

大模型读了 subagent 工具 description 里的强制规则——单个查询不符合"编号列表 + 2 个及以上子任务"的条件，自然不调 subagent。

## 第 4 章 Plan vs Subagent + 总结 + demo5 预告

### 4.1 Plan vs Subagent 全面对比

### 【口播】

demo3 的 plan 和 demo4 的 subagent 看起来都是"拆任务"，但**适用场景完全不同**。

### 【屏显 · 表格 4-1 — Plan vs Subagent】

| 维度 | Plan（demo3） | Subagent（demo4） |
|---|---|---|
| **上下文** | step 间**共享** messages | 每个 Subagent **独立** messages |
| **身份** | 全程同一个 system_prompt | 每个 Subagent 有自己的 role / system_prompt |
| **生命周期** | 主 Agent 一直存在，step 是它的子循环 | Subagent 是**一次性**的，干完即销毁 |
| **step / subagent 之间** | 后续 step 能看到前面 step 的 tool_result | Subagent 之间相互看不到 |
| **执行方式** | 严格按步骤顺序 | 各自独立（demo4 仍串行，可扩展为并行） |
| **适合的场景** | **依赖关系**：B 要用 A 的结果 | **相互独立**：四件事各干各的 |
| **类比** | 一个人按步骤做事情 | 把事情**外包**给不同分包公司 |

### 4.2 用 demo3 的例子说明 plan 的适用场景

### 【口播】

demo3 的真实案例：

```
1) 计算 35+47
2) 用第 1 步的结果乘以 8           ← 依赖 Step 1 的 82
3) 查北京天气
4) 把前三步的结果写到 test_report.txt   ← 依赖 Step 1-3 的结果
5) 验证写入成功
```

**Step 2 显式说"用第 1 步的结果"、Step 4 要写"前三步的结果"**——后续 step 必须看到前面 step 的 tool_result。**这正是共享 messages 的 plan 模式擅长的**。

如果用 subagent 派出去：加法专家算完返回 "82"，乘法专家的 subagent **看不到这个 82**——任务就崩了。

### 4.3 用 demo4 的例子说明 subagent 的适用场景

### 【口播】

demo4 的真实案例是四个**相互独立**的任务：

```
1) 计算 35+47        ← 加法，与其它任务无关
2) 计算 65*9         ← 乘法，与其它任务无关
3) 查上海天气        ← 与其它任务无关
4) 写 hello.txt      ← 与其它任务无关
```

每个 subagent **不需要知道其它 subagent 在干什么**——独立 messages 反而保持专注、降低 token 消耗。**这正是 subagent 擅长的**。如果用 plan，4 个 step 的 tool_result 全堆在一份 messages 里，上下文越叠越长。

### 4.4 长远看：subagent 也能扩展出依赖

### 【口播】

那 subagent 是不是永远搞不定"有依赖"的场景？不是——**只要让 subagent 之间能通信**就行：

- **消息队列**：A subagent 算完，往队列里丢一条结果
- **共享黑板**：所有 subagent 读写同一份共享状态
- **事件总线**：A 完成 → 触发事件 → B 监听到再启动

引入通信机制之后，多个 subagent 就能形成 **team 模式**——彼此等待、彼此协作，能处理"有依赖"的复杂场景。

但 demo4 **只展示独立 subagent 这一个机制**，通信 / 协作 / 持久身份等高级功能不在本节范围内。

### 【屏显 · 决策树】

```
任务复杂吗？
├─ 否 → 直接 ReAct（不拆任务）
└─ 是 → 子任务之间有依赖吗？
        ├─ 有依赖 → 用 Plan（共享 messages，后续 step 看到前面结果）
        └─ 相互独立 → 用 Subagent（独立 messages，主 Agent 只看摘要）
```

### 4.5 总结

### 【口播 · demo4 做了什么】

回顾整节，demo4 完成了**一次减法 + 一次加法**：

| # | 改动 | 做法 | 解决的问题 |
|---|---|---|---|
| -1 | **去掉 MCP** | 三件小工具从 MCP Server 搬回本地 | 不用多起一个服务进程 |
| -2 | **去掉 plan** | 删除 plan 工具及顶层决策分叉 | 本次只验证 subagent 场景 |
| +1 | **新增 subagent** | 工具 schema + `_run_subagent` + 共享 `_react_loop` | 单 Agent 上下文膨胀 / 无法分工 |

### 【口播 · 关键洞察】

Subagent 看起来"很高级"，但底层**没有任何新协议、新框架**——它就是：

1. 一个**普通工具**（schema + 调用入口）
2. 调用时**启动一个新的 ReAct 循环**（与主 Agent 同一份代码）
3. 区别只在**独立的 messages / system_prompt / 工具集（去掉 subagent）**

换句话说，**Subagent = 把"启动一个 Agent"这件事本身做成工具**。一个 Agent **演化出了很多子 Agent**——这是 Agent 演化中**最关键的一次自我放大**。
