## 第 1 章 结论：demo2 比 demo1 多了什么

### 【口播 · 开场】

demo1 的 Agent 有三个根本限制：

1. **跨任务无记忆**：上一个任务做完，下一次再问，**什么都不记得**。
2. **任务内消息单调累积**：多轮 ReAct 一直在 `messages` 上累积，**第 N 轮的 messages 会把上下文窗口撑爆**。
3. **固定前缀重复传输**：每次请求都把整个 system prompt + memory 原样重发一遍，**重复传输**。

demo2 就是要补上这三块——**长期记忆 / 短期压缩 / Prompt caching**。这就是「记忆轴」的全部内容。

### 【屏显 · 表格 1-1 — demo2 vs demo1】

| 维度 | demo1 | demo2 |
|---|---|---|
| **工具集** | execute_bash / read_file / write_file | **不变** |
| **ReAct 单步循环** | ✅ | ✅ **不变** |
| **跨任务长期记忆** | ❌ 任务结束 messages 清空 | ✅ `agent_memory.md` 文件持久化 |
| **任务内动态压缩** | ❌ messages 单调累积 | ✅ `compact_messages` 触发时滚动摘要 |
| **Prompt caching** | ❌ system prompt 每次重发 | ✅ `cache_control` breakpoint，命中后服务端复用无需重传 |

## 第 2 章 长期记忆：信息搬运的本质

### 【口播 · 为什么需要记忆】

大家用过 Claude Code 或者 Cursor 吧？你会发现，**在一个长会话里，它能记住你之前说过什么、做过什么**。

但是 demo1 的 Agent——**任务一结束，messages 全清空**。

那为什么不直接用大模型自己的记忆？因为**大模型本质上没有真正的记忆**——它只有一个**有限的上下文窗口**，窗口塞满就溢出。

所以记忆这件事，**本质上是一个信息搬运问题**：

### 【屏显 · 表格 2-1 — 记忆的本质】

| 角色 | 能力 | 限制 |
|---|---|---|
| **大模型** | 有一定上下文窗口、有一定推理能力 | 窗口有限，**没有真正的持久记忆** |
| **本地 Agent** | 可以读写外部存储（文件/数据库/向量库） | 搬运什么、搬多少，要自己决定 |

**记忆 = 把外部存储的信息，有选择地搬运进 prompt**。所有方案的区别，就是「**存在哪、以什么形式存、如何选择搬入上下文**」三个问题的不同答案。

### 2.1 存储方案：一个 Markdown 文件

### 【口播】

demo2 用**最原始**的方案——一个 `agent_memory.md` 文件。**作为演示，足够揭示原理。**

每次任务结束，我们追加三件事：**什么时间、做了什么任务、得到什么结果**。

### 【屏显 · 代码 2-1 — append_memory】

```python
MEMORY_FILE = "agent_memory.md"

def append_memory(task: str, result: str) -> None:
    timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    # 结果摘要限长，避免一次任务把整个文件撑爆
    result_preview = (result or "").strip()
    if len(result_preview) > 500:
        result_preview = result_preview[:500] + "..."
    entry = (
        f"\n## [{timestamp}]\n"
        f"**任务**: {task}\n"
        f"**结果**: {result_preview}\n"
    )
    try:
        with open(MEMORY_FILE, "a", encoding="utf-8") as f:
            f.write(entry)
    except Exception as e:
        print(f"[警告] 写入记忆文件失败: {e}")
```

### 【屏显 · agent_memory.md 实际内容示例】

下面是连续跑了两个真实任务后的 `agent_memory.md`（2026-07-19 实测）：

```markdown

**任务**: 统计当前目录下有多少个 Python 文件，并把结果写入 count.txt
**结果**: 任务完成！统计结果如下：

- **当前目录下共有 1 个 Python 文件（`.py`）**
- 结果已写入 **count.txt** 文件中。

## [2026-07-19 01:10:06]

       4) 计算 任务2的结果/2； 5）把 "hello" 写入 hello.txt；要求一次只执行一个任务，不要并发
**结果**: ✅ 任务 5 完成：已将 "hello" 写入 hello.txt

| 任务 | 描述 | 结果 |
|------|------|------|
| 任务 1 | 检查并删除 hello.txt | hello.txt 不存在，无需删除 |
| 任务 2 | 35 + 47 | 82 |
| 任务 3 | 82 × 9 | 738 |
| 任务 4 | 738 / 2 | 369 |
| 任务 5 | 写入 "hello" 到 hello.txt | 已完成 |
```

### 2.2 加载方案：滑动窗口（最后 50 行）

### 【口播】

存进去了，**怎么读出来**？

最朴素的方案：**整个文件全塞进 prompt**？——不行。一个 Agent 跑了几十次任务，文件会越长越大，最终把大模型的上下文窗口撑爆。

所以我们做一次简化：**只加载最后 50 行**。

### 【屏显 · 代码 2-2 — load_memory】

```python
MEMORY_WINDOW_LINES = 50  # 滑动窗口

def load_memory() -> str:
    if not os.path.exists(MEMORY_FILE):
        return ""
    try:
        with open(MEMORY_FILE, "r", encoding="utf-8") as f:
            lines = f.readlines()
        # 滑动窗口：只取最后 N 行
        return "".join(lines[-MEMORY_WINDOW_LINES:])
    except Exception as e:
        print(f"[警告] 读取记忆文件失败: {e}")
        return ""
```

### 【口播】

这就是**滑动窗口**——窗口大小固定 50 行，旧记忆会随新记忆累积被挤出窗口。

为什么是 50 行？——**没有任何科学依据**，演示用的占位值。重要的是这个**「限制 + 截取」**的模式本身。

### 2.3 Progressive Context：怎么注入 prompt

### 【口播】

记忆加载出来之后，**塞到哪？**——塞到 System Prompt 的**后缀**里，作为「**Progressive Context**」（渐进式上下文）。

### 【屏显 · 代码 2-3 — build_system_prompt】

```python
def build_system_prompt(verbose: bool = False) -> str:
    memory = load_memory()
    if not memory.strip():
        return SYSTEM_PROMPT_BASE
    return SYSTEM_PROMPT_BASE + "\n\n## 历史任务记忆（最近）\n\n" + memory
```

### 【屏显 · 图 2-1 — System Prompt 的两层结构】

```
┌─────────────────────────────────────────────┐
│  System Prompt                              │
│  ┌───────────────────────────────────────┐  │
│  │  基础 Prompt（不变）                    │  │
│  │  「你是一个有用的助手，可以调用工具...」  │  │
│  └───────────────────────────────────────┘  │
│  ┌───────────────────────────────────────┐  │
│  │  Progressive Context（动态）            │  │  ← demo2 新增
│  │  最近 50 行历史任务记忆                  │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

### 【口播 · 记忆时序】

### 【屏显 · 图 2-2 — 长期记忆时序】

```
首次运行（agent_memory.md 不存在）—— 案例 1：统计 .py 文件:
    System (基础 prompt)
        ↓
    user: "统计当前目录下有多少个 Python 文件，并把结果写入 count.txt"
        ↓
    [ReAct 循环] → 结果 "1 个 Python 文件，已写入 count.txt"
        ↓
    append_memory → 写入 agent_memory.md（第一条）


第二次运行（agent_memory.md 已有 1 条）—— 案例 2：5 步串行任务:
    System = 基础 prompt + Progressive Context(最后 50 行)
        ↓                                       ↑
        ↓                  ┌─── 看到案例 1 的任务 + 结果
        ↓                  │
    user: "1) 如果有 hello.txt，先删除；2) 计算 35+47；3) ...；5) 把 hello 写入 hello.txt"
        ↓
    [ReAct 循环 7 轮] → 结果 "5 任务全部完成"（中途触发 compact）
        ↓
    append_memory → 追加到 agent_memory.md（第二条）
```

### 【口播 · 关键观察】

注意一个细节：**System Prompt 整个 task 内不变**。我们在 `run_agent` 入口构建一次 system prompt，传给所有 ReAct 轮次共用。任务进行中不会重新读记忆文件——因为任务还没结束，记忆不会更新。

**只有任务结束 → append_memory 写入 → 下一个任务开始 → 才会读到新记忆**。

### 2.4 从 demo2 抽象出 Memory 的三个核心问题

### 【屏显 · 表格 2-2 — 记忆方案的演进谱系】

| 阶段 | 存在哪 | 以什么形式存 | 如何选择搬入上下文 |
|---|---|---|---|
| **上下文缓存（demo2）** | 文本文件/上下文窗口 | 原始文本 | 顺序读取（最近 N 条） |
| **语义检索记忆** | 向量数据库 | 文本 + embedding + metadata | embedding similarity top-K |
| **Agent Memory** | Memory Store | 结构化事实/事件/偏好 | Agent 决策后查询 |
| **分层 Memory** | 多级存储 | system/project/user/session 分层 | 按作用域+相关性组合检索 |

### 【口播 · 表格里后三行是什么】

- **语义检索记忆**：把记忆切成 chunk 并向量化，user message 也变 embedding，按 cosine 相似度取 top-K。**查询是规则触发的本地运算**（不需要 LLM）。代表：RAG、ChatGPT 早期 Memory。
- **Agent Memory**：先把对话**蒸馏成结构化事实**（偏好/事件/属性），再由 Agent **自己决定要不要查、查什么**——`memory_search` 暴露成工具，跟 `read_file` 一样。查询本身仍是本地运算，差别在「**决策由 LLM 做**」。代表：Claude Code 跨会话记忆、Mem0。
- **分层 Memory**：按作用域分层（系统/项目/用户/会话），每层独立检索策略。代表：Claude Code 的 CLAUDE.md（项目级）+ auto-memory（用户级）。

> **一句话区分**：语义检索是「**按相似度自动取**」，Agent Memory 是「**LLM 决策按需查结构化事实**」——后者存储密度高，但多了一步 LLM 决策。工业级系统常**两者并存**。

所有 Memory 系统最终都可以抽象成三个问题：

- **存在哪**（Storage）
- **以什么形式存**（Representation）
- **如何选择搬入上下文**（Retrieval）

## 第 3 章 短期记忆的崩溃：compact_messages

### 3.1 长任务的真实痛点

### 【口播】

长期记忆解决了「跨任务记得过去」，但**单个任务内**还有个致命问题——每轮 ReAct 都把 `assistant` 回复 + `tool_use` + `tool_result` 全部 append 进 messages，**单调增长，无任何裁剪**。轮次一多，最后撞上下文窗口上限，API 报错，任务失败。

Claude Code 怎么解决？——`auto-compact`：接近窗口上限时，让 LLM 把老消息摘要成一段，接着跑。demo2 我们实现最简版：`compact_messages`。

### 3.2 触发条件 + 摘要策略

### 【口播】

`compact_messages` 做两件事：

1. **什么时候触发**：messages 条数达到或超过阈值（demo2 用 **10 条**——演示用低阈值，方便短任务就触发一次压缩；生产级按 token 占比 ~80% 触发）
2. **怎么压缩**：保留最近 N 条原始消息（demo2 保留 **4 条**），老消息让 LLM 摘要成一段

### 【屏显 · 图 3-2 — compact_messages 工作原理（数字取自 §5.2 案例 2 真实触发）】

```
压缩前 messages（11 条，第 6 轮 ReAct 入口触发）:
┌─────────────────────┐
│  [0]  user          │  ← 老 messages
│  [1]  assistant ...  │     （7 条，
│  [2]  user          │      让 LLM 摘要）
│  [3]  assistant ...
│  [4]  user
│  [5]  assistant ...
│  [6]  user
├─────────────────────┤
│  [7]  assistant ...  │  ← 最近 4 条
│  [8]  user          │     （保留原始）
│  [9]  assistant ...
│  [10] user
└─────────────────────┘
             ↓ compact_messages()
压缩后 messages（6 条）:
┌─────────────────────┐
│  [0] user: [历史对话  │  ← 摘要注入
│      已压缩，摘要如   │
│      下] ...         │
│  [1] assistant: 好的 │     （配合 ack）
├─────────────────────┤
│  [2-5] 最近 4 条原始  │  ← 直接搬过来（原 [7-10]）
└─────────────────────┘
```

### 3.3 代码实现

### 【屏显 · 代码 3-1 — compact_messages + 切点保护】

```python
COMPACT_THRESHOLD_MESSAGES = 10   # 触发阈值（演示用低值，方便短任务就触发）
COMPACT_KEEP_RECENT        = 4    # 保留最近 N 条

def _is_tool_result_message(msg) -> bool:
    """判断 message 是否为「承载 tool_result 的 user 消息」"""
    # （与触发它的 assistant tool_use 配对，切开就会破坏配对）
    ...

def _find_recent_start(messages: list) -> int:
    """找 recent 段起始 index，遇到 tool_result 消息就向前回退一步。"""
    start = max(1, len(messages) - COMPACT_KEEP_RECENT)
    while start > 1 and _is_tool_result_message(messages[start]):
        start -= 1
    return start

def compact_messages(messages: list, verbose: bool = False) -> list:
    if len(messages) < COMPACT_THRESHOLD_MESSAGES:
        return messages

    # 切点保护：不能让 recent 第一条是 tool_result（会切断 tool_use ↔ tool_result 配对）
    recent_start = _find_recent_start(messages)
    old_messages = messages[:recent_start]
    recent_messages = messages[recent_start:]

    # 把老 messages 转成 transcript 文本（统一处理 dict / SDK block 对象）
    transcript_parts = []
    for msg in old_messages:
        role = msg.get("role", "?")
        text = _extract_text(msg.get("content", ""))
        transcript_parts.append(f"### {role}\n{text}")
    transcript = "\n\n".join(transcript_parts)

    # 让 LLM 摘要
    response = client.messages.create(
        model=MODEL,
        max_tokens=1024,
        system=COMPACT_SYSTEM_PROMPT,
        messages=[{"role": "user", "content": f"对话历史：\n\n{transcript}\n\n请输出压缩摘要："}],
    )
    summary = "".join(b.text for b in response.content if b.type == "text")

    # 把摘要作为 [历史对话摘要] 标记消息注入
    summary_msg = {"role": "user",
                   "content": f"[历史对话已压缩，摘要如下]\n{summary}"}
    ack_msg = {"role": "assistant",
               "content": "好的，我已了解历史对话摘要，继续执行当前任务。"}

    return [summary_msg, ack_msg] + recent_messages
```

### 【口播 · 三个设计要点】

### 【屏显 · 表格 3-1 — compact 的三个设计选择】

| 选择 | demo2 做法 | 为什么 |
|---|---|---|
| **触发条件** | 按消息条数（10 条，演示用低阈值） | 简单可测；生产级按 token 占比 ~80% 触发 |
| **保留策略** | 最近 4 条原始不动 + 切点边界保护 | 最近一轮的 tool_use / tool_result 必须完整，否则 LLM 不知道当前进度 |
| **摘要注入** | 用 `[历史对话已压缩]` 标记 + assistant ack | 让 LLM 知道这是压缩摘要而非新指令；ack 保持 user/assistant 交替结构 |

### 【口播 · 关键陷阱】

注意第二个设计点——**为什么必须保留最近 N 条原始**？因为 Anthropic API 要求 messages 末尾如果是 `tool_use`，下一条必须是 `tool_result`；如果 messages 末尾是 `user`，下一条必须是 `assistant`。压缩时一刀切会破坏这个结构。

`COMPACT_KEEP_RECENT=4` 通常能覆盖最近一轮完整的 `tool_use + tool_result` 对，但**固定条数不够保险**——万一落在 tool_result 上，前缀注入的 `summary_msg(user) + ack_msg(assistant)` 会切断它与触发它的 assistant tool_use 的配对，导致 API 报 `tool_result without preceding tool_use`。

demo2 用 `_find_recent_start` 显式做切点边界保护：从 `len(messages) - COMPACT_KEEP_RECENT` 起步，如果切点正好压在一条 tool_result 消息上，就**向前回退一步**，直到切点落在安全的消息上。

## 第 4 章 Prompt caching：避免 system prompt 重复传输

### 4.1 问题：每次请求重发整个 system prompt

### 【口播】

讲完压缩，再看一个**传输效率问题**。

demo1 的每次 API 调用，messages 长这样：

```
[
  {"role": "system", "content": "<200 行 system prompt + 50 行 memory>"},
  {"role": "user",   "content": "统计 .py 文件"},
  {"role": "assistant", "content": "..."},
  {"role": "user",   "content": "<tool_result>"},
  ...
]
```

**system prompt 部分（含记忆）整个 task 内不变**——但每次请求都要重发一遍。如果 task 跑 30 轮 ReAct，**system prompt 被发 30 次，全部是重复传输**。

如果是 Claude Code 这种工业 Agent，system prompt 里塞了 CLAUDE.md（项目约定）+ auto-memory（跨会话记忆）+ skill descriptions（技能说明），随便就 10k+ tokens。30 轮 ReAct × 10k tokens = 30 万 tokens **全是重复传输**。

Anthropic 的解药：**Prompt caching**。

### 4.2 cache_control breakpoint

### 【口播】

Anthropic API 允许在 system blocks 上加 `cache_control` 标记，告诉服务端：「**到这里为止的内容如果没变，就给我命中缓存**」。

### 【屏显 · 代码 4-1 — system prompt 从字符串变成 blocks】

```python
# 旧做法（demo1）：纯字符串
response = client.messages.create(
    system=system_prompt,
    ...
)

# demo2 新做法：blocks + cache_control
response = client.messages.create(
    system=[
        {
            "type": "text",
            "text": system_prompt,
            "cache_control": {"type": "ephemeral"},
        }
    ],
    ...
)
```

### 【屏显 · 图 4-1 — 缓存命中机制】

```
第 1 次请求（缓存未建）:
    client → [system (10k) + messages (2k)] → server
    server → 计算 10k tokens，建缓存，存 5min
    server → 返回结果 + usage.cache_creation_input_tokens=10000

第 2 次请求（缓存命中，5min 内 + system 没变）:
    client → [system (10k) + messages (5k)] → server
    server → 命中缓存！system 段直接复用，只处理 5k 新 tokens
    server → 返回结果 + usage.cache_read_input_tokens=10000  ← 无需重传
```

### 4.3 demo2 中的实现

### 【屏显 · 代码 4-2 — _build_system_param 兼容两种模式】

```python
USE_CACHE_CONTROL = True  # 某些兼容网关不实现 caching，可关掉

def _build_system_param(system_prompt: str):
    """
    构建 messages.create 的 system 参数。
    - True：返回 blocks + cache_control
    - False：返回纯字符串（兼容不支持 caching 的网关）
    """
    if not USE_CACHE_CONTROL:
        return system_prompt
    return [
        {"type": "text", "text": system_prompt,
         "cache_control": {"type": "ephemeral"}}
    ]
```

### 【口播 · 为什么留一个开关】

注意我们留了一个 `USE_CACHE_CONTROL = True` 开关。**原因**：智谱 BigModel 等 Anthropic 兼容网关，不一定实现了 caching 后端——API 接口可能接受 `cache_control` 字段但不实际缓存，或直接报错。开关在的时候，遇到不支持就关掉，**Agent 仍然能跑**，只是失去缓存优势。

**这是个工程实践细节**：生产级 Agent 接到任意网关，需要 graceful degradation（优雅降级）。

### 4.4 命中统计：让 caching 效果可见

### 【屏显 · 代码 4-3 — _print_cache_stats】

```python
def _print_cache_stats(usage, verbose: bool = True) -> None:
    cache_create = getattr(usage, "cache_creation_input_tokens", 0) or 0
    cache_read   = getattr(usage, "cache_read_input_tokens", 0) or 0
    input_tokens = getattr(usage, "input_tokens", 0) or 0

    if cache_create > 0:
        print(f"  [cache] 创建缓存 {cache_create} tokens + 输入 {input_tokens}")
    elif cache_read > 0:
        print(f"  [cache] 命中缓存 {cache_read} tokens + 输入 {input_tokens} ✓")
    elif USE_CACHE_CONTROL:
        # cache_control 发了但本轮网关没返回命中数据——可能是网关内部缓存策略
        # （如大小上限 / TTL 短），也可能是兼容网关根本不实现 caching 后端
        print(f"  [cache] 未命中 / 输入 {input_tokens} tokens")
    else:
        print(f"  [cache] caching 关闭 / 输入 {input_tokens} tokens")
```

### 【口播】

教学上**让 caching 效果可见很重要**——所以我们在每轮 ReAct 打印 cache hit/miss 统计。

第三个分支「发了 cache_control 但没命中也没创建」处理的是兼容网关行为诡异的情况——智谱 BigModel 等网关未必实现 caching 后端，具体实测见第 5 章 5.1 观察 1。

### 5.2 案例 2 实测：第二次运行（含案例 1 记忆 + 触发 compact）

### 【口播 · 任务设计】

跑一个 5 步串行任务（2026-07-19 实测），目的是把 messages 撑过阈值 10，触发 compact：

> **任务**：1) 如果有 `hello.txt`，先删除；2) 计算 `35+47`；3) 计算 任务1 的结果 `*9`；4) 计算 任务2 的结果 `/2`；5) 把 `"hello"` 写入 `hello.txt`。要求一次只执行一个任务，不要并发。

每个子任务一轮 ReAct（`execute_bash` 或 `write_file`），共 5 轮工具调用 + 1 轮汇报 ≈ 11 条 messages，第 6 轮刚好触发 compact。

### 【屏显 · 实测启动日志】

```

```

### 【口播】

Agent 启动时 `load_memory()` 读到案例 1 的 9 行条目（任务 + 结果 + 2 行 bullet 总结），合到 system prompt 末尾作为 Progressive Context。整个 task 内 system prompt 不变——这给 caching 创造了条件。

### 【屏显 · 表格 5-2 — 案例 2 的 7 轮 ReAct + compact 触发明细】

| 轮次 | 进入时 msgs | 决策 | 工具结果 | cache | input |
|:---:|:---:|---|---|:---:|:---:|
| 1 | 1 | `execute_bash` `[ -f hello.txt ]` | Windows 报错（语法不兼容） | 未命中 | 506 |
| 2 | 3 | `execute_bash` `if exist hello.txt` | `hello.txt not found` | 未命中 | 597 |
| 3 | 5 | `execute_bash` `set /a result=35+47` | `82` | 未命中 | 640 |
| 4 | 7 | `execute_bash` `set /a result=82*9` | `738` | 未命中 | 706 |
| 5 | 9 | `execute_bash` `set /a result=738/2` | `369` | 未命中 | 780 |
| **6** | **11 → 6** | **compact 触发** → `write_file` hello.txt | 写入 5 字符 | **命中 384** ✓ | 346 |
| 7 | 8 | end_turn 汇报 | — | 未命中 | 803 |

**第 6 轮的 compact 真实日志**：

```
[compact] 触发：7 条老消息 → 摘要
[compact] 保留最近 4 条原始消息
[compact] 摘要: 用户要求按顺序执行 5 个任务，禁止并发。任务 1：检查并删除 hello.txt，
              通过 execute_bash 工具确认 hello.txt 不存在，无需删除，已完成。
              任务 2：计算 35+47，通过 execute_bash 工具得出结果为 82。
              当前进度：前两个任务已完成，剩余任务 3（82×9）、任务 4（任务3结果/2）、
              任务 5（将 "hello" 写入 hello.txt）尚未执行...
[compact] 压缩后：6 条消息
```

### 【口播 · compact 触发的三个关键观察】

#### 观察 1：触发点完全符合设计

`compact_messages` 入口检查 `len(messages) >= 10`。第 6 轮进入时 messages=11（1 初始 user + 5 轮工具调用 × 2）≥ 10 → 触发。

**算式**：`old_messages = 11 - 4 = 7 条` → LLM 摘要成 1 段 → 加 1 条 ack → 接 4 条最近 → **6 条新 messages**。

#### 观察 2：摘要保留了「任务上下文」而非「对话原话」

LLM 摘要不是把 7 条老消息原样压缩，而是**重述为「任务列表 + 当前进度」**：「任务 1 已完成；任务 2=82；任务 3、4、5 尚未执行」。这正是压缩摘要该有的形态——**保留事实，丢掉措辞**。

#### 观察 3：compact 后立刻命中缓存——system prompt 不变是关键

第 6 轮 compact 之后，`cache_read=384` 出现——**压缩之后第一次请求就命中**。原因：compact 只动 messages，system prompt 整轮不变，缓存命中边界由 system prompt 决定。这也印证了把 Progressive Context 塞在 system prompt 里的设计——system 整 task 内不变，缓存才能稳；如果塞在 messages 里，每轮 messages 增长会破坏缓存前缀。

### 5.3 两次任务实测汇总：三个机制的实测状态

### 【屏显 · 表格 5-3 — 三个机制（拆为 4 个动作）在两次任务中的触发情况】

| 机制 | 案例 1（首次运行） | 案例 2（第二次运行） |
|---|---|---|
| **长期记忆加载** | 0 次（memory 为空） | **1 次**（加载案例 1 的 9 行条目） |
| **长期记忆写入** | **1 次**（写入案例 1 条目） | **1 次**（追加案例 2 条目） |
| **caching** | **1 次命中**（第 3 轮 320 tokens） | **1 次命中**（第 6 轮 384 tokens，compact 后立即热起来） |
| **compact** | 0 次（5 条 < 阈值 10） | **1 次**（第 6 轮，11 → 6 条） |

### 【口播 · 设计精髓】

**三个机制是正交的**——任意组合都可以，独立开关，互不依赖。案例 1 简单任务只触发了「记忆写入 + 1 次缓存命中」；案例 2 长任务全部触发：记忆加载 + 记忆写入 + compact + compact 后立刻缓存命中。**这就是真实 Agent 运行的样子**：机制按需启用，不是每轮都触发。
