## 第 1 章 结论：demo2 比 demo1 多了什么

### 【口播 · 开场】

demo1 的 Agent 就像一条**金鱼**：

1. **跨任务无记忆**：上一个任务做完，下一次再问，**什么都不记得**。
2. **任务内单线程**：多轮 ReAct 一直在 `messages` 上累积，**第 N 轮的 messages 会把上下文窗口撑爆**。
3. **固定前缀重复传输**：每次请求都把整个 system prompt + memory 原样重发一遍，**重复占用带宽与 token 额度**。

demo2 就是要补上这三块——**长期记忆 / 短期压缩 / Prompt caching**。这就是「记忆轴」的全部内容。

### 【屏显 · 表格 1-1 — demo2 vs demo1】

| 维度 | demo1 | demo2 |
|---|---|---|
| **工具集** | execute_bash / read_file / write_file | **不变** |
| **ReAct 单步循环** | ✅ | ✅ **不变** |
| **跨任务长期记忆** | ❌ 任务结束 messages 清空 | ✅ `agent_memory.md` 文件持久化 |
| **任务内动态压缩** | ❌ messages 单调累积 | ✅ `compact_messages` 触发时滚动摘要 |
| **Prompt caching** | ❌ system prompt 每次重发 | ✅ `cache_control` breakpoint，命中后服务端复用无需重传 |

### 【口播 · 新公式】

demo1 的核心——**工具集 / ReAct 循环 / 路由表**——我们一个都不动。只在 base 旁边并排加一根**记忆轴**。新公式：

> ### demo2 = base × 记忆

## 第 2 章 长期记忆：信息搬运的本质

### 【口播 · 为什么需要记忆】

大家用过 Claude Code 或者 Cursor 吧？你会发现，**在一个长会话里，它能记住你之前说过什么、做过什么**。

但是 demo1 的 Agent——**任务一结束，messages 全清空**。下一次再启动，**金鱼记忆**。

那为什么不直接用大模型自己的记忆？因为**大模型本质上没有真正的记忆**——它只有一个**有限的上下文窗口**，窗口塞满就溢出。

所以记忆这件事，**本质上是一个信息搬运问题**：

### 【屏显 · 表格 2-1 — 记忆的本质】

| 角色 | 能力 | 限制 |
|---|---|---|
| **大模型** | 有一定上下文窗口、有一定推理能力 | 窗口有限，**没有真正的持久记忆** |
| **本地 Agent** | 可以读写外部存储（文件/数据库/向量库） | 搬运什么、搬多少，要自己决定 |

**记忆 = 把外部存储的信息，有选择地搬运进 prompt**。所有方案的区别，就是「**存在哪、怎么存、搬多少**」三个问题的不同答案。

### 【口播 · 一个比喻：内存 vs 外存】

打个比方——**大模型的上下文窗口就是内存**（快但容量小），**文件 / 数据库 / 向量库是外存**（容量大但得主动加载）。

本地 Agent 有读写外存的能力。**会话过程中，按需把外存里有用的东西搬进内存（prompt），让大模型思考时能看到**——这就是记忆系统的全部工作。

### 2.1 存储方案：一个 Markdown 文件

### 【口播】

demo2 用**最原始**的方案——一个 `agent_memory.md` 文件。不引入数据库，不引入向量库。**作为演示，足够揭示原理。**

每次任务结束，我们追加三件事：**什么时间、做了什么任务、得到什么结果**。

### 【屏显 · 代码 2-1 — append_memory】

```python
MEMORY_FILE = "agent_memory.md"

def append_memory(task: str, result: str) -> None:
    timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    result_preview = result[:500]  # 限长，防止文件膨胀
    entry = (
        f"\n## [{timestamp}]\n"
        f"**任务**: {task}\n"
        f"**结果**: {result_preview}\n"
    )
    with open(MEMORY_FILE, "a", encoding="utf-8") as f:
        f.write(entry)
```

### 【屏显 · agent_memory.md 实际内容示例】

```markdown
## [2026-06-29 14:23:11]
**任务**: 统计当前目录下有多少个 Python 文件
**结果**: 已统计完成，当前目录下共有 12 个 Python 文件...

## [2026-06-29 14:25:42]
**任务**: 创建一个 hello world 的 py 文件
**结果**: 已创建 hello.py，内容为 print("Hello, World!")
```

### 【口播】

每条记忆就是一个**三行结构**——**时间戳 + 任务 + 结果摘要**。任务结束后追加。这就是 demo1 在结尾提到「**金鱼记忆**」要解决的核心。

### 2.2 加载方案：滑动窗口（最后 50 行）

### 【口播】

存进去了，**怎么读出来**？

最朴素的方案：**整个文件全塞进 prompt**？——不行。一个 Agent 跑了几十次任务，文件会越长越大，最终把大模型的上下文窗口撑爆。

所以我们做一次简化：**只加载最后 50 行**。

### 【屏显 · 代码 2-2 — load_memory】

```python
MEMORY_WINDOW_LINES = 50  # 滑动窗口

def load_memory() -> str:
    if not os.path.exists(MEMORY_FILE):
        return ""
    with open(MEMORY_FILE, encoding="utf-8") as f:
        lines = f.readlines()
    # 滑动窗口：只取最后 N 行
    return "".join(lines[-MEMORY_WINDOW_LINES:])
```

### 【口播】

这就是**滑动窗口**——窗口大小固定 50 行，旧记忆会随新记忆累积被挤出窗口。

为什么是 50 行？——**没有任何科学依据**，这是个演示用的占位值。真实场景里，**50 行可能很短，也可能很长**，取决于单条记忆的大小和大模型的上下文窗口剩余空间。重要的是这个**「限制 + 截取」**的模式本身。

### 2.3 Progressive Context：怎么注入 prompt

### 【口播】

记忆加载出来之后，**塞到哪？**——塞到 System Prompt 的**后缀**里，作为「**Progressive Context**」（渐进式上下文）。

### 【屏显 · 代码 2-3 — build_system_prompt】

```python
def build_system_prompt(verbose: bool = False) -> str:
    memory = load_memory()
    if not memory.strip():
        return SYSTEM_PROMPT_BASE
    return SYSTEM_PROMPT_BASE + "\n\n## 历史任务记忆（最近）\n\n" + memory
```

### 【屏显 · 图 2-1 — System Prompt 的两层结构】

```
┌─────────────────────────────────────────────┐
│  System Prompt                              │
│  ┌───────────────────────────────────────┐  │
│  │  基础 Prompt（不变）                    │  │
│  │  「你是一个有用的助手，可以调用工具...」  │  │
│  └───────────────────────────────────────┘  │
│  ┌───────────────────────────────────────┐  │
│  │  Progressive Context（动态）            │  │  ← demo2 新增
│  │  最近 50 行历史任务记忆                  │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

### 【屏显 · 演示 — build_system_prompt 现场运行】

口播时配合实机演示：调用 `build_system_prompt()` 打印输出，让读者直观看到**基础 prompt + 历史记忆后缀**拼出来的完整 system prompt——记忆条目清晰可见地附加在基础 prompt 后面。

### 【口播 · 记忆时序】

### 【屏显 · 图 2-2 — 长期记忆时序】

```
首次运行（agent_memory.md 不存在）:
    System (基础 prompt)
        ↓
    user: "统计当前目录有多少 .py 文件"
        ↓
    [ReAct 循环] → 结果 "12 个"
        ↓
    append_memory → 写入 agent_memory.md


第二次运行（agent_memory.md 已有内容）:
    System = 基础 prompt + Progressive Context(最后 50 行)
        ↓                                       ↑
        ↓                  ┌─── 看到第一次任务 + 结果
        ↓                  │
    user: "再创建一个 hello.py"  ← 大模型知道之前做了什么
        ↓
    [ReAct 循环] → 结果 "已创建"
        ↓
    append_memory → 追加到 agent_memory.md
```

### 【口播 · 关键观察】

注意一个细节：**System Prompt 整个 task 内不变**。我们在 `run_agent` 入口构建一次 system prompt，传给所有 ReAct 轮次共用。任务进行中不会重新读记忆文件——因为任务还没结束，记忆不会更新。

**只有任务结束 → append_memory 写入 → 下一个任务开始 → 才会读到新记忆**。这是个简单但重要的设计选择。

### 2.4 这套方案揭示的原理

### 【口播】

demo2 这套长期记忆方案**非常粗糙**，但它揭示了一个**普适原理**：

> **大模型有上下文窗口限制，本地必须把外部存储的信息有选择地搬运进 prompt。**

不管你用什么方案：

### 【屏显 · 表格 2-2 — 记忆方案的演进谱系】

| 方案 | 存在哪 | 怎么存 | 搬多少 |
|---|---|---|---|
| **demo2（演示）** | Markdown 文件 | 追加 | 最后 50 行 |
| **向量数据库** | Chroma / Pinecone | 嵌入向量 | 语义相关的 top-K |
| **Memory 工具** | 外部存储 | Agent 自主存取 | 按需检索 |
| **分层记忆** | 多个文件 | 系统/项目/用户级分层 | 按层级叠加（项目 CLAUDE.md / 用户级 auto-memory） |

**底层都是同一个模式：存在哪、怎么存、搬多少**。第 6 章会展开讲这些演进方向。

## 第 3 章 短期记忆的崩溃：compact_messages

### 3.1 长任务的真实痛点

### 【口播】

长期记忆解决了「跨任务记得过去」，但**单个任务内**还有个致命问题——**messages 单调累积，迟早撑爆上下文窗口**。

来看一个真实场景：让 Agent 「找所有 TODO 整理到 todo.md」。Agent 会做什么？

### 【屏显 · 图 3-1 — 一个长任务的 messages 增长】

```
轮次  操作                              messages 累积
─────────────────────────────────────────────────────
1     grep -r TODO .                    ~1k tokens
2     读取 grep 结果（10 个文件命中）      ~3k tokens
3     read_file 文件1                    ~5k tokens
4     read_file 文件2                    ~7k tokens
5     read_file 文件3                    ~9k tokens
6     read_file 文件4                    ~11k tokens
7     read_file 文件5                    ~13k tokens
...   （读取所有命中文件）                  ...
15    write_file todo.md                 ~25k tokens
16    验证                               ~27k tokens
17    总结                               ~28k tokens
```

每轮 ReAct 把上一轮的 `assistant` 回复 + `tool_use` + `tool_result` 全部 append 进 messages。**messages 是单调增长的，没有任何机制裁剪**。

如果命中 50 个文件、跑 100 轮 ReAct，messages 会冲到几十万 tokens——**直接撞上下文窗口上限（200K），API 报错，任务失败**。

Claude Code 怎么解决？——`auto-compact`：接近窗口上限时，自动把老消息让 LLM 自己摘要成一段，然后接着跑。demo2 我们实现一个最简版：`compact_messages`。

### 3.2 触发条件 + 摘要策略

### 【口播】

`compact_messages` 做两件事：

1. **什么时候触发**：messages 条数超过阈值（demo2 用 20 条，生产级按 token 占比 ~80% 触发）
2. **怎么压缩**：保留最近 N 条原始消息，老消息让 LLM 摘要成一段

### 【屏显 · 图 3-2 — compact_messages 工作原理】

```
压缩前 messages（25 条）:
┌─────────────────────┐
│  [0] user: 找 TODO   │  ← 老 messages
│  [1] assistant: ...   │     （19 条，
│  [2] tool_use grep    │      让 LLM 摘要）
│  [3] tool_result ...  │
│  ...                  │
│  [18] tool_use ...    │
│  [19] tool_result ... │
├─────────────────────┤
│  [20] assistant: ...  │  ← 最近 6 条
│  [21] tool_use ...    │     （保留原始）
│  [22] tool_result ... │
│  [23] tool_use ...    │
│  [24] tool_result ... │
└─────────────────────┘
             ↓ compact_messages()
压缩后 messages（8 条）:
┌─────────────────────┐
│  [0] user: [历史对话  │  ← 摘要注入
│      已压缩，摘要如   │
│      下] ...         │
│  [1] assistant: 好的 │     （配合 ack）
├─────────────────────┤
│  [2-7] 最近 6 条原始  │  ← 直接搬过来
└─────────────────────┘
```

### 3.3 代码实现

### 【屏显 · 代码 3-1 — should_compact + compact_messages】

```python
COMPACT_THRESHOLD_MESSAGES = 20   # 触发阈值
COMPACT_KEEP_RECENT        = 6    # 保留最近 N 条

def should_compact(messages: list) -> bool:
    return len(messages) >= COMPACT_THRESHOLD_MESSAGES

def compact_messages(messages: list, verbose: bool = False) -> list:
    if len(messages) < COMPACT_THRESHOLD_MESSAGES:
        return messages

    old_messages = messages[:-COMPACT_KEEP_RECENT]
    recent_messages = messages[-COMPACT_KEEP_RECENT:]

    # 把老 messages 转成 transcript 文本
    transcript = messages_to_text(old_messages)

    # 让 LLM 摘要
    response = client.messages.create(
        model=MODEL,
        max_tokens=1024,
        system=COMPACT_SYSTEM_PROMPT,
        messages=[{"role": "user", "content": f"对话历史：\n\n{transcript}\n\n请输出压缩摘要："}],
    )
    summary = "".join(b.text for b in response.content if b.type == "text")

    # 把摘要作为 [历史对话摘要] 标记消息注入
    summary_msg = {"role": "user",
                   "content": f"[历史对话已压缩，摘要如下]\n{summary}"}
    ack_msg = {"role": "assistant",
               "content": "好的，我已了解历史对话摘要，继续执行当前任务。"}

    return [summary_msg, ack_msg] + recent_messages
```

### 【口播 · 三个设计要点】

### 【屏显 · 表格 3-1 — compact 的三个设计选择】

| 选择 | demo2 做法 | 为什么 |
|---|---|---|
| **触发条件** | 按消息条数（20 条） | 简单可测；生产级按 token 占比 ~80% 触发 |
| **保留策略** | 最近 6 条原始不动 | 最近一轮的 tool_use / tool_result 必须完整，否则 LLM 不知道当前进度 |
| **摘要注入** | 用 `[历史对话已压缩]` 标记 + assistant ack | 让 LLM 知道这是压缩摘要而非新指令；ack 保持 user/assistant 交替结构 |

### 【口播 · 关键陷阱】

注意第二个设计点——**为什么必须保留最近 N 条原始**？因为 Anthropic API 要求 messages 末尾如果是 `tool_use`，下一条必须是 `tool_result`；如果 messages 末尾是 `user`，下一条必须是 `assistant`。压缩时一刀切会破坏这个结构。

`COMPACT_KEEP_RECENT=6` 保证至少有一对完整的 `tool_use + tool_result`，且末尾是 `tool_result`（紧跟着 LLM 会被调用，正确接续）。

## 第 4 章 Prompt caching：长 prompt 不爆带宽

### 4.1 问题：每次请求重发整个 system prompt

### 【口播】

讲完压缩，再看一个**传输效率问题**。

demo1 的每次 API 调用，messages 长这样：

```
[
  {"role": "system", "content": "<200 行 system prompt + 50 行 memory>"},
  {"role": "user",   "content": "统计 .py 文件"},
  {"role": "assistant", "content": "..."},
  {"role": "user",   "content": "<tool_result>"},
  ...
]
```

**system prompt 部分（含记忆）整个 task 内不变**——但每次请求都要重发一遍。如果 task 跑 30 轮 ReAct，**system prompt 被发 30 次，全部是重复传输**。

如果是 Claude Code 这种工业 Agent，system prompt 里塞了 CLAUDE.md（项目约定）+ auto-memory（跨会话记忆）+ skill descriptions（技能说明），随便就 10k+ tokens。30 轮 ReAct × 10k tokens = 30 万 tokens **全是重复传输**。

Anthropic 的解药：**Prompt caching**。

### 4.2 cache_control breakpoint

### 【口播】

Anthropic API 允许在 system blocks 上加 `cache_control` 标记，告诉服务端：「**到这里为止的内容如果没变，就给我命中缓存**」。

### 【屏显 · 代码 4-1 — system prompt 从字符串变成 blocks】

```python
# 旧做法（demo1）：纯字符串
response = client.messages.create(
    system=system_prompt,
    ...
)

# demo2 新做法：blocks + cache_control
response = client.messages.create(
    system=[
        {
            "type": "text",
            "text": system_prompt,
            "cache_control": {"type": "ephemeral"},
        }
    ],
    ...
)
```

### 【屏显 · 图 4-1 — 缓存命中机制】

```
第 1 次请求（缓存未建）:
    client → [system (10k) + messages (2k)] → server
    server → 计算 10k tokens，建缓存，存 5min
    server → 返回结果 + usage.cache_creation_input_tokens=10000

第 2 次请求（缓存命中，5min 内 + system 没变）:
    client → [system (10k) + messages (5k)] → server
    server → 命中缓存！system 段直接复用，只处理 5k 新 tokens
    server → 返回结果 + usage.cache_read_input_tokens=10000  ← 无需重传
```

### 4.3 demo2 中的实现

### 【屏显 · 代码 4-2 — _build_system_param 兼容两种模式】

```python
USE_CACHE_CONTROL = True  # 某些兼容网关不实现 caching，可关掉

def _build_system_param(system_prompt: str):
    """
    构建 messages.create 的 system 参数。
    - True：返回 blocks + cache_control
    - False：返回纯字符串（兼容不支持 caching 的网关）
    """
    if not USE_CACHE_CONTROL:
        return system_prompt
    return [
        {"type": "text", "text": system_prompt,
         "cache_control": {"type": "ephemeral"}}
    ]
```

### 【口播 · 为什么留一个开关】

注意我们留了一个 `USE_CACHE_CONTROL = True` 开关。**原因**：智谱 BigModel 等 Anthropic 兼容网关，不一定实现了 caching 后端——API 接口可能接受 `cache_control` 字段但不实际缓存，或直接报错。开关在的时候，遇到不支持就关掉，**Agent 仍然能跑**，只是失去缓存优势。

**这是个工程实践细节**：生产级 Agent 接到任意网关，需要 graceful degradation（优雅降级）。

### 4.4 命中统计：让 caching 效果可见

### 【屏显 · 代码 4-3 — _print_cache_stats】

```python
def _print_cache_stats(usage, verbose: bool = True) -> None:
    cache_create = getattr(usage, "cache_creation_input_tokens", 0) or 0
    cache_read   = getattr(usage, "cache_read_input_tokens", 0) or 0
    input_tokens = getattr(usage, "input_tokens", 0) or 0

    if cache_create > 0:
        print(f"  [cache] 创建缓存 {cache_create} tokens + 输入 {input_tokens}")
    elif cache_read > 0:
        print(f"  [cache] 命中缓存 {cache_read} tokens + 输入 {input_tokens} ✓")
    elif USE_CACHE_CONTROL:
        # cache_control 发了但本轮网关没返回命中数据——可能是网关内部缓存策略
        # （如大小上限 / TTL 短），也可能是兼容网关根本不实现 caching 后端
        print(f"  [cache] 未命中 / 输入 {input_tokens} tokens")
    else:
        print(f"  [cache] caching 关闭 / 输入 {input_tokens} tokens")
```

### 【口播】

教学上**让 caching 效果可见很重要**——所以我们在每轮 ReAct 打印 cache hit/miss 统计。

注意第三个分支——**「发了 cache_control 但没命中也没创建」**——这种情况在 Anthropic 官方 API 上少见，但在 **Anthropic 兼容网关**上很常见：

- 智谱 BigModel / Azure OpenAI 兼容层 / 国产大模型兼容网关，**未必实现 caching 后端**
- 有的网关会**默默接受 `cache_control` 字段但不实际缓存**
- 有的网关**有内部缓存策略**（大小上限 / TTL 极短），导致间歇性命中

所以 `_print_cache_stats` 在「发了 cache_control 但 usage 没数据」时要给出**「未命中」**而不是「未启用」——**前者是网关行为，后者是用户配置**，意义完全不同。这是个工程实践细节，演示真实接入异构网关时的可观测性需求。

### 4.5 演进：多 breakpoint + 长 TTL

### 【屏显 · 表格 4-2 — Prompt caching 的演进】

| 能力 | demo2 | Claude Code |
|---|---|---|
| breakpoint 数量 | 1 个（system 末尾） | 多个（system + tools + 部分 messages） |
| TTL | 5min（ephemeral） | 5min / 1h（extended cache） |
| 失效粒度 | system 变任何字符 → 整个缓存失效 | 分段失效：tools 变只失效 tools 段 |

工业级用法是**多 breakpoint**：把 system prompt 拆成「绝对不变」和「偶尔变」两段，分别加 cache_control，让小变更不毁掉整个缓存。

### 【口播 · 表格不用逐行念】

这张表是给读者**课后过目**的，口播不逐行念。讲的时候挑三个关键点说就行——下面三个观察就是挑出来的三个点。

### 【口播 · 三个关键观察】

#### 观察 1：智谱 BigModel 的 caching 行为跟 Anthropic 官方不一样

**`cache_creation_input_tokens` 全程为 0** —— 网关**从不报告建缓存**。但 `cache_read_input_tokens` 间歇出现 —— 说明**网关其实建了缓存**，只是不收创建费、还有内部缓存策略（大小上限 / TTL / LRU）导致**间歇性命中**。

这就是第 4 章说的「**兼容网关行为各异，需要 graceful degradation**」的真实样子——`_print_cache_stats` 的「未命中」分支不是错误，而是真实可观测性。

#### 观察 2：compact 触发后，缓存要「重新热起来」

第 11 轮 compact 触发，messages 从 21 条变成 8 条（结构变了）→ system 之后的 messages 前缀也变了 → 缓存失效 4 轮（12-15 轮）→ 第 16 轮恢复命中。

**关键启示**：caching 的失效边界**依赖具体网关实现**——Anthropic 按 prefix 匹配，只要 system 没变就稳；BigModel 把 messages 的稳定前缀也算进缓存范围，messages 结构一变就丢。

#### 观察 3：compact 触发点完全符合设计

```
[compact] 触发：15 条老消息（~404 tokens）→ 摘要
[compact] 保留最近 6 条原始消息
[compact] 摘要: 用户要求扫描 D:/workspace/mini-agent 目录下所有 agent.py 文件...
[compact] 压缩后：8 条消息（~247 tokens）
```

**算式**：15 条老消息（21 - 6）→ 摘要成 1 段 → 加 1 条 ack → 接 6 条最近 → **8 条新 messages**。Agent 在压缩后继续工作，第 15 轮用 PowerShell 拿到行数，第 16 轮写入 `agent_stats.md` 并结束。

### 5.4 三个机制的叠加效果（基于实测）

### 【屏显 · 表格 5-2 — 实测的三个机制收益】

| 机制 | 实测触发 | 效果 |
|---|---|---|
| **caching** | 11/16 轮命中（69%） | cache_read 累计 ~23,000 tokens 无需重传，**避免了 90% 的 system prompt 重复传输** |
| **compact** | 1 次触发（第 11 轮） | messages 从 21 条 / ~15k tokens → 8 条 / ~2.5k tokens，**控住了上下文增长** |
| **长期记忆** | 1 次写入 | 下次任务能直接看到「demo5 是最长文件，932 行」 |

### 【口播 · 设计精髓】

**三个机制是正交的**：

- caching 解决「**重复 system prompt 的传输**」
- compact 解决「**累积 messages 的容量**」
- 长期记忆解决「**跨任务的知识传递**」

任意组合都可以，独立开关，互不依赖。实测中 compact 触发导致 caching 暂时失效——**这俩机制本身不冲突**，失效是因为兼容网关的缓存边界设计，**不是 demo2 代码逻辑问题**。换成 Anthropic 官方 API（多 breakpoint 缓存）后，compact 后的缓存失效可以最小化。

这就是把「记忆轴」拆成三个子机制的工程价值——**正交设计让复杂度可控，每个机制可以独立调优 / 独立观测 / 独立替换**。

### 5.5 实测中暴露的工程细节

### 【屏显 · 表格 5-3 — 实测中踩到的真实坑】

| 坑 | 表现 | 教学价值 |
|---|---|---|
| **LLM 用 bash find / wc -l** | 第 1-7 轮反复失败（系统找不到路径 / exit 1） | 真实 Agent 跨平台时的痛——demo3 引入 `glob` / `grep` 专用工具的部分动机 |
| **subprocess GBK 解码崩溃** | 第 13 轮 `UnicodeDecodeError: 'gbk' codec can't decode byte 0xad` | `execute_bash` 在 Windows 用 `text=True` 默认 GBK，应改 `encoding="utf-8" errors="replace"` |
| **PowerShell 才拿到行数** | 第 14 轮换 PowerShell 一次性返回 7 个文件行数 | LLM 自己摸索到可用的命令——ReAct 的纠错能力 |
| **caching 不报告创建** | cache_creation 全程为 0 | 兼容网关行为差异——需要 `_print_cache_stats` 的「未命中」分支做可观测性 |

这些坑都是真实生产 Agent 会遇到的——**demo2 的价值不在「完美实现」，而在「让你看到这些坑」**。

## 第 6 章 局限与工业级演进

### 【口播】

demo2 的记忆轴是**最简实现**，每个子机制都有明显的粗糙之处。这一章对照讲工业级 Agent（Claude Code / Cursor / Devin）做了哪些优化。

### 【屏显 · 表格 6-1 — 记忆轴的工业级演进】

| 子机制 | demo2 局限 | 工业级做法 |
|---|---|---|
| **长期记忆存储** | Markdown 文件 + 50 行硬截断 | 分层（CLAUDE.md 项目级 + 用户级 auto-memory）+ 向量检索 top-K |
| **加载策略** | 最后 50 行（按位置） | 按语义相关性（向量数据库 + 嵌入） |
| **compact 触发** | 消息条数（20 条） | token 占比（~80% 上下文窗口） |
| **compact 目标** | 固定保留 6 条 | 自适应：保留最近一轮完整 + 关键决策摘要 |
| **caching breakpoint** | 1 个（system 末尾） | 多个（system + tools + 部分固定 messages） |
| **caching TTL** | 5min（ephemeral） | 5min / 1h 可选 |
| **会话状态持久化** | ❌ 任务结束 messages 丢弃 | ✅ `--resume` 加载上次 messages 接着跑 |

### 【口播 · 这些演进都不是新原理】

表格里的每一条演进，都建立在 demo2 揭示的同一组原理上：

> 记忆 = 把外部存储的信息，有选择地搬运进 prompt。

无论用 Markdown 还是向量库，无论按条数还是按 token 触发，无论单 breakpoint 还是多 breakpoint——**底层都是「存在哪 / 怎么存 / 搬多少」三个问题的不同答案**。理解了 demo2 这套最简方案，看工业级优化就是「在同一个框架下换更精细的参数」。

## 第 7 章 下一节预告：工具扩展轴（demo3）

### 【口播】

demo2 解决了 Agent **记得过去**、**不爆上下文**、**跑得起长 prompt** 三件事。但 Agent 还有第二个根本限制——**手不够长**。

demo1-2 的工具集只有 3 个：`execute_bash / read_file / write_file`。这 3 个工具理论上图灵完备（bash 能干任何事），但实际工程上有两个问题：

1. **read_file + write_file 太粗糙**：改一个文件要把整个文件覆写一遍，大文件 token 浪费严重
2. **接不到外部协议**：想用别人写好的 MCP server（GitHub / Slack / Postgres），不行

### 【屏显 · 表格 7-1 — 三个未解决问题 → demo3 的三件新事】

| demo2 留下的问题 | demo3 的解决 |
|---|---|
| **改文件粗暴**（整文件覆盖） | 引入 `edit` 工具：string replacement，只发改动部分 |
| **搜索靠 bash**（grep/ls 手写命令） | 引入专用工具：`glob` / `grep`（声明式，LLM 不用拼 shell） |
| **接不到外部协议** | 引入 **MCP**（Model Context Protocol）：client-server 架构，挂外部工具 server |

### 【口播 · 收尾金句】

> demo2 的 Agent 已经不是金鱼了——它记得过去。
>
> 但它仍然只有 3 件套的工具。下一节 demo3，我们给它**更多手脚 + 远程工具箱**。
