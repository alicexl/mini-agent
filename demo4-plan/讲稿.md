## 第 1 章 结论：demo4 比 demo1 多了什么

### 【口播 · 开场】

demo1 给我们打下了 Agent 的心跳——`LLM × 工具 × 循环 × 状态`（ReAct）。但 ReAct 是**走一步看一步**：LLM 每轮只决策下一步，做完看结果，再决策下下一步。

这个机制在简单任务上很优雅，但有两个痛点：

1. **复杂任务容易跑偏**——5 步以上的任务，LLM 走到第 3 步可能忘了整体目标，或者重复尝试同一思路
2. **常见任务每次重新想**——「git commit」「code review」「整理目录」这类高频任务，每次都要 LLM 从零推演步骤，浪费 token 又难保证一致性

demo4 的轴是「规划」，就解决这两件事。

> ### demo4 = base × 规划
> （demo1 是 base，demo4 在 base 上独立加「规划」轴——不叠加 demo2 的记忆轴、不叠加 demo3 的工具轴）

### 【屏显 · 表格 1-1 — demo4 vs demo1】

| 维度 | demo1 | demo4 |
|---|---|---|
| **本地工具** | bash / read / write 三件套 | **+ todo_write = 四件套** |
| **任务规划** | ❌ 走一步看一步 | ✅ **TodoWrite：LLM 自动判断复杂度，复杂任务先列步骤** |
| **跨任务复用** | ❌ 每次从零推演 | ✅ **Skill：预消化的工作流，触发词命中即注入 prompt** |
| **谁决定何时规划** | — | **LLM 自主决策**（简单任务跳过 todo_write 直接做） |

### 【口播 · base 不变】

注意——**demo1 的核心都没动**：

1. **ReAct 主循环不变**（LLM → tool_use → tool_result → ... → end_turn）
2. **三个本地工具不变**（execute_bash / read_file / write_file 原封不动）
3. **system prompt + messages 结构不变**

我们只做**两件新事**：加一个本地工具 `todo_write`、加一个 Skill 加载器。

「规划轴」的两个增量分别对应**单次任务的内部规划**和**跨任务的复用规划**：
- **(A) 单次内部规划**：todo_write 让 LLM 把多步拆解显式化、可观测、可追踪
- **(B) 跨任务复用规划**：Skill 把"常见任务的最佳实践步骤"预先固化，相似任务直接套用

接下来 5 章，逐层拆开。

## 第 2 章 为什么需要规划

### 【口播 · ReAct 的本质缺陷】

让我们先看 demo1 在复杂任务上的真实表现。假设用户输入：

> 把 agent.py 里所有 def 函数名整理到 functions.md

demo1 的 ReAct 会这样做：

```
第 1 轮：LLM 决策 → execute_bash("grep -nE '^\\s*def\\s+' agent.py")
第 2 轮：LLM 看结果 → write_file("functions.md", "...")
第 3 轮：LLM 总结完成
```

三步、三轮、漂亮。但这是**简单任务**——单一工具链，无依赖，无分支。

现在换一个**真正复杂**的任务：

> 第一步读 agent.py；第二步提取所有函数名按字母排序；第三步提取 LOCAL_TOOLS 的工具名；第四步整理成 markdown 表格写文件；第五步用 execute_bash 显示文件内容做验证。

demo1 的 ReAct 模式会怎样？

- LLM 可能**跳过中间步骤**——把 5 步压缩成「读文件 → 写文件 → 显示」3 步
- 也可能**重复读同一个文件**——每步都重新 read，因为它忘了第 1 步的结果
- 也可能**迷失目标**——到第 4 步忘了第 2 步排序的函数名
- 也可能**省略验证**——第 5 步的「显示验证」直接跳过

本质原因：**ReAct 的 messages 上下文是线性的，没有显式的「全局计划」让 LLM 锚定**。每一步的决策都只看了前面的工具结果，没看「我自己定的步骤清单」。

### 【屏显 · 表格 2-1 — ReAct vs Plan 的对比】

| 维度 | ReAct（demo1） | Plan（demo4） |
|---|---|---|
| **决策粒度** | 单步——每轮只看下一步 | 单步 + 全局——先列步骤，再逐步执行 |
| **可观测性** | 只有工具结果可见 | **todo_write 让步骤状态可见**（pending / in_progress / completed） |
| **目标锚定** | 靠 LLM 在 messages 里"记住" | **靠 todo 清单显式锚定**，每完成一步就标记 |
| **跨任务复用** | 每次重新想 | **Skill 把工作流预消化**，触发词命中直接套用 |

### 【口播 · 规划的两类增量】

demo4 给的规划能力分两层：

1. **单次任务的内部规划**（todo_write）——LLM 自己判断任务复杂度，简单任务直接 ReAct，复杂任务先列 todo 清单再逐步推进。对应 Claude Code 的 TodoWrite 工具。
2. **跨任务的复用规划**（Skill）——把"代码审查该怎么做""git commit 该怎么做"这类**重复性的工作流**预先写成 SKILL.md 文件，用户输入命中触发词时把 body 注入 system prompt，让 LLM 按预设流程执行。对应 Claude Code 的 Skill 工具。

这两件事合在一起就是 demo4 的轴。接下来两章分别拆解。

## 第 3 章 机制一：TodoWrite（自动决策版）

### 【口播 · TodoWrite 是什么】

Claude Code 里有个工具叫 **TodoWrite**——它的作用很简单：让 LLM 把当前任务拆成显式的 step 列表，每步带状态（pending / in_progress / completed），完成一步就更新一下。

听起来很朴素，但它的本质是**把"规划阶段"从 LLM 脑子里搬到屏幕上**：

- 用户能看到 Agent 当前的待办清单（透明）
- LLM 自己也能看到（避免迷失目标）
- 每步状态变化都是一次"自我承诺"（增强一致性）

### 【屏显 · 代码 3-1 — todo_write 工具定义（agent.py Part 2）】

```python
{
    "name": "todo_write",
    "description": (
        "更新当前任务的待办步骤清单（TodoWrite 风格）。"
        "**使用时机**：只在**复杂的多步任务**（3 步以上、多工具协作、步骤间有依赖）开头规划阶段调用，"
        "把整个任务拆成显式 step 列表；后续每完成一步就再调一次本工具更新对应 step 的状态。\n\n"
        "**不要滥用**：简单的一两步任务（如「统计 .py 文件数」「读一下某文件」）"
        "请直接用 execute_bash / read_file 完成，**不要**为了用而用 todo_write。"
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "todos": {
                "type": "array",
                "items": {
                    "type": "object",
                    "properties": {
                        "subject": {"type": "string", "description": "这一步要做什么（动词开头）"},
                        "status":  {"type": "string", "enum": ["pending", "in_progress", "completed"]},
                    },
                    "required": ["subject", "status"],
                },
            },
        },
        "required": ["todos"],
    },
}
```

### 【口播 · description 的关键设计】

description 里有两段引导，决定了 LLM 是否会用、何时用：

1. **「3 步以上、多工具协作、步骤间有依赖」**——给出**正向触发条件**
2. **「不要滥用：简单的一两步任务请直接用 execute_bash / read_file」**——给出**反向抑制条件**

这两段缺一不可。只给正向条件，LLM 会**滥用**（每个任务都列 todo）；只给反向条件，LLM 会**从不使用**。

### 3.1 todo_write 的实现

### 【屏显 · 代码 3-2 — todo_write 的实现（agent.py 第 258 行）】

```python
TODOS_FILE = os.path.join(os.path.dirname(os.path.abspath(__file__)), "todos.md")

_STATUS_MARK = {
    "pending":     "[ ]",
    "in_progress": "[→]",
    "completed":   "[✓]",
}


def todo_write(todos: list) -> str:
    """更新待办清单（demo4 新增）"""
    lines = ["# 当前任务待办清单\n"]
    n_by_status = {"pending": 0, "in_progress": 0, "completed": 0}

    for i, item in enumerate(todos, 1):
        subject = str(item.get("subject", "")).strip()
        status = str(item.get("status", "pending")).strip()
        n_by_status[status] += 1
        mark = _STATUS_MARK[status]
        lines.append(f"{i}. {mark} {subject}")

    text = "\n".join(lines)
    # 同步落盘（覆盖式）——让用户能随时打开 todos.md 查看
    with open(TODOS_FILE, "w", encoding="utf-8") as f:
        f.write(text + "\n")

    # 终端 pretty-print（让 verbose 模式轨迹清晰）
    print("\n" + "─" * 50)
    print(text)
    print("─" * 50)

    total = len(todos)
    return (
        f"[已更新] 共 {total} 步：✓{n_by_status['completed']} 完成 / "
        f"→{n_by_status['in_progress']} 进行中 / [ ]{n_by_status['pending']} 待办。"
        f"清单已同步写入 {TODOS_FILE}"
    )
```

### 【口播 · 关键不是落盘】

注意 `todos.md` 落盘只是**副产物**——给人看的，给 debug 用的。真正的作用有两层：

1. **终端 pretty-print**——让 verbose 模式的运行轨迹里出现一张可视化的待办清单表，LLM 每次调 todo_write 都会刷一次表
2. **状态变化在 messages 里被 LLM 看到**——返回的字符串「共 5 步：✓1 完成 / →1 进行中 / [ ]3 待办」会作为 tool_result 回到 messages，下一轮 LLM 就能看到自己当前的进度

### 3.2 自动决策：LLM 自己判断要不要用 todo_write

### 【口播 · 为什么是「自动」模式】

demo4 的 todo_write 是**自动决策版**——没有 `/plan` 命令、没有显式的"现在开始规划"开关。LLM 看到工具列表里有 todo_write，自己根据任务复杂度决定何时用。

system prompt 里的引导（agent.py 第 479 行 `build_system_prompt`）：

```
**任务复杂度自判规则**：
- 简单任务（1-2 步、单一工具）→ 直接 ReAct，跳过 todo_write
- 复杂任务（3+ 步、多工具协作、有依赖）→ 先 todo_write 列步骤，再逐步执行并更新状态
```

**为什么不做手动 `/plan` 模式？**

手动模式（用户输入 `/plan <任务>` 强制进入 plan-only 模式）确实更可控，但教学对照会复杂化——需要分两条代码路径（plan 模式 vs 执行模式）。Claude Code 自己也是 LLM 自动决策为主，`/plan` 是次要触发。所以 demo4 只做自动版，把「LLM 怎么判断复杂度」这件事放在聚光灯下。

## 第 4 章 机制二：Skill（预消化的工作流）

### 【口播 · 为什么需要 Skill】

假设你经常让 Agent 做 code review。每次都说：

> 帮我 review 一下 xxx 文件

每次 LLM 都要重新推演：「review 该怎么做？先读文件，然后检查命名、安全、错误处理……最后给出意见」。推演出来的步骤**每次可能不一样**——有时候漏掉安全检查，有时候输出格式不统一。

**Skill 解决的是「重复发明轮子」**：把 code review 的标准工作流预先写成一个 `skills/review.md` 文件，触发词命中时把这个工作流的 body 注入 system prompt，LLM 就**按预设流程执行**，不再每次重新想。

对应 Claude Code 的 Skill 工具——你在 Claude Code 里看到的 `commit` / `review-pr` 这些命令，本质就是预消化的工作流。

### 【屏显 · 表格 4-1 — Skill vs 普通工具】

| 维度 | 普通工具（execute_bash 等） | Skill |
|---|---|---|
| **本质** | 单步操作（执行 shell 命令） | **多步工作流模板**（先做什么、再做什么） |
| **触发** | LLM 自主调用 | **用户输入命中 triggers 关键词** |
| **注入位置** | tools 参数 | **system prompt**（一段额外的 markdown 指令） |
| **作用** | 扩展 LLM 的「手脚」 | **教 LLM 怎么协调手脚做一类事** |

### 4.1 Skill 文件格式

### 【屏显 · 代码 4-1 — skills/review.md 的格式】

```markdown
---
name: review
description: 代码审查工作流——按规范步骤检查代码改动并给出结构化意见
triggers: ["代码审查", "代码评审", "审查代码", "code review", "帮我 review", "review 一下"]
---

# 代码审查工作流

收到「审查代码」类任务时，**必须**按以下步骤执行，不要直接下结论：

## 步骤 1：收集改动
- 如果有 git 仓库：用 execute_bash 跑 git diff 或 git log --oneline -5
- 如果没有 git 或要审整个文件：用 read_file 直接读目标文件

## 步骤 2：逐项检查
| 维度 | 检查点 |
|---|---|
| 命名 | 变量/函数/类名是否清晰、一致 |
| 错误处理 | try/except 是否吞掉异常 |
| 安全 | 命令注入 / SQL 注入 / 硬编码密钥 |
| ... | ... |

## 步骤 3：结构化输出
最终回复必须包含三段：
- ✅ 优点
- ⚠️ 问题（🔴 阻塞 / 🟡 建议 / 🟢 可选）
- 💡 改进建议
```

### 【口播 · frontmatter 三字段】

每个 skill 文件用 YAML frontmatter 声明三件事：

1. **`name`**——skill 的唯一标识（注入 prompt 时引用）
2. **`description`**——一句话说明做什么（启动时被加载到「可用 Skills」元信息段）
3. **`triggers`**——关键词数组（用户输入命中任一关键词，则激活此 skill）

frontmatter 之后的 body 是**真正的工作流正文**——只有在 skill 被激活时才会注入 system prompt（启动时只加载元信息，不加载 body）。

```python
    不引入 PyYAML 依赖——教学代码保持零额外依赖。"""
    # 用正则匹配首尾 ---，再分别抽取 name/description/triggers
    ...


def load_skills() -> dict:
    """扫描 skills/*.md，解析 frontmatter，返回 {name: {description, triggers, body}}"""
    skills = {}
    for fname in sorted(os.listdir(SKILLS_DIR)):
        if not fname.endswith(".md"):
            continue
        with open(os.path.join(SKILLS_DIR, fname), "r", encoding="utf-8") as f:
            raw = f.read()
        meta, body = _parse_frontmatter(raw)
        name = meta.get("name") or fname[:-3]
        skills[name] = {"description": ..., "triggers": ..., "body": body}
    return skills


def match_skill(user_input: str, skills: dict) -> Optional[str]:
    """用户输入命中某 skill 的任一 trigger 关键词则返回该 skill name"""
    text = user_input.lower()
    for name, info in skills.items():
        for trigger in info["triggers"]:
            if trigger.lower() in text:
                return name
    return None
```

### 【口播 · 不引入 PyYAML 的设计取舍】

注意 `_parse_frontmatter` 用正则实现简易 YAML 解析，而不是 `import yaml`。这是个有意的取舍：

- **优点**：零额外依赖，复制 agent.py + skills/ 目录到任何机器都能跑
- **代价**：只支持 `name: str` / `description: str` / `triggers: [str, ...]` 三个字段，不支持嵌套结构

教学代码保持零依赖是值得的——读者下载即跑，不被 `pip install` 卡住。

### 4.3 system prompt 的三层叠加

### 【屏显 · 代码 4-3 — build_system_prompt（agent.py 第 479 行）】

```python
def build_system_prompt(skills: dict, activated_skill: Optional[str]) -> str:
    """
    构造 system prompt（三层叠加）：
        1. 基础说明（角色 + 工具清单 + todo_write 使用指引）
        2. 可用 Skills 元信息（启动时注入一次，每次都带）
        3. 当前激活 skill 的 body（本次输入命中时注入；否则空段）
    """
    parts = [
        "你是一个有用的助手...",
        "可用工具：1. execute_bash ... 4. todo_write ...",
        "**任务复杂度自判规则**：...",
    ]

    # 第二层：可用 Skills 元信息
    skill_meta = build_skill_metadata_section(skills)
    if skill_meta:
        parts.append(skill_meta)

    # 第三层：当前激活 skill 的 body
    if activated_skill and activated_skill in skills:
        body = skills[activated_skill]["body"]
        parts.append(
            f"\n## 当前激活的 Skill：{activated_skill}\n\n"
            f"本次用户输入命中了 `{activated_skill}` skill 的触发词。"
            f"**必须严格按照下面的工作流执行**：\n\n{body}\n"
        )

    return "\n".join(parts)
```

### 【口播 · 三层的不同时机】

这三层叠加的**注入时机不同**，这是 Skill 机制的精髓：

| 层 | 内容 | 何时构造 |
|---|---|---|
| **第 1 层** | 基础说明 + 工具清单 + todo 引导 | 每轮 user 输入时构造（因为后两层可能变） |
| **第 2 层** | 所有 skill 的 name + description + triggers 列表 | 启动时一次性加载到字典，每轮 prompt 都带上 |
| **第 3 层** | **当前激活** skill 的 body 正文 | **每轮 user 输入时重新 match_skill 决定** |

**为什么不让所有 skill 的 body 都常驻 prompt？**

skill 一多就会撑爆上下文（每个 body 几百到几千 token）。第二层只放元信息（每个 skill 一两行），第三层只在命中时才把对应 body 注入——这样 100 个 skill 也只增加几百 token 元信息开销。

### 【口播 · run_agent 里的注入时机（agent.py 第 540 行）】

```python

    if activated:
        print(f"[Skill] 用户输入命中 → 激活 skill: {activated}")

    system_prompt = build_system_prompt(skills, activated)
    messages = [{"role": "user", "content": user_input}]

```

**关键点**：每次用户输入时调一次 `match_skill` 决定第三段——不同输入激活不同 skill。第一次输入「review 一下」激活 review skill；下一次输入「统计 .py 文件数」就一个 skill 都不激活，第三段为空。

### 5.1 场景 A：简单任务——LLM 自动跳过 todo_write

**实测轨迹**：

```
用户: [Skill] 未命中任何 skill（可用: review）

----- ReAct 第 1 轮 -----
[LLM 决策] stop_reason = tool_use
  - tool_use : execute_bash({'command': 'find demo4-plan -type f -name "*.py" | wc -l'})

[执行工具] execute_bash(...)
```

### 【屏显 · 表格 6-1 — demo4 轴覆盖回顾】

| 轴 | 所属 Demo | 核心机制 | demo4 实现 |
|---|---|---|---|
| ReAct 循环 | demo1 | messages → LLM → tool_use → tool_result → ... → end_turn | ✅ 沿用 |
| **TodoWrite**（自动决策） | **demo4** | LLM 自判复杂度，复杂任务先列 step 再逐步推进 | ✅ todo_write 工具 + todos.md 落盘 |
| **Skill**（预消化工作流） | **demo4** | frontmatter triggers 命中 → body 注入 system prompt | ✅ skills/*.md + load_skills + match_skill |

### 【口播 · 轴公式的对齐】

按总览的「视角 A」公式：

```
demo1: base = LLM × 工具 × 循环 × 状态
demo4: = base × 规划            （todo_write 单次规划 + Skill 跨任务复用）
```

demo4 基于 demo1 base，**不叠加** demo2 的记忆轴、demo3 的工具轴。读者可以按任意顺序学 demo2 / demo3 / demo4——三条轴正交。

### 【口播 · 两个机制的边界】

todo_write 和 Skill 都是「规划」，但作用范围不同：

| 机制 | 作用范围 | 持久性 | 触发 |
|---|---|---|---|
| **todo_write** | **单次任务内部** | 任务结束即丢弃（todos.md 是副产物） | LLM 自动判断 |
| **Skill** | **跨任务复用** | 永久（skills/*.md 文件持续存在） | 用户输入命中 triggers |

简单说：**todo_write 是「这次怎么做」，Skill 是「这类事该怎么做」**。

### 【口播 · 下一节预告：demo5 多 Agent】

demo4 的规划能力都还在**单个 Agent 内部**——todo_write 是单 Agent 的自我拆解，Skill 是单 Agent 的工作流模板。

但如果任务**真的很大**（比如「调研一个技术方案 + 写一份报告 + 自我评审」），单个 Agent 的上下文会越堆越长，效率越来越低。

demo5 引入**多 Agent**：

- **Subagent**：把独立子任务**外包**给一次性 Agent（独立 context、干完即销毁）
- **Team**：组建**持久化项目团队**（Researcher → Writer → Reviewer 流水线，inbox 通信 + 状态机驱动）

这正是 Claude Code 的 Task 工具、Cursor 的 agent mode、AutoGen / CrewAI 范式背后的机制。
