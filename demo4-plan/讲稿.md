## 第 1 章 demo4 干了什么 & 为什么需要规划

### 【口播 · 开场】

demo1 给我们打下了 Agent 的心跳——`base = LLM × 工具 × 循环 × 状态`（ReAct）。但 ReAct 是**走一步看一步**：LLM 每轮只决策下一步，做完看结果，再决策下下一步。

这个机制在简单任务上很优雅，但有两个痛点：

1. **复杂任务容易跑偏**——5 步以上的任务，LLM 走到第 3 步可能忘了整体目标，或者重复尝试同一思路。从 A 到 B 的过程中，因为中间某一步的决策慢慢偏离了原定路线，它会在弯路上绕很久。
2. **高频任务每次重新想**——「帮我从微信公众号抓一篇文章」「用 bash 跑个脚本」这类任务非常高频。如果每次都让 LLM 从零推演步骤，**每次推演出来的方案可能不一样**；就算方案一样，每次重新规划一遍也浪费 token。

> ### demo4 = base × 规划
> （demo1 是 base，demo4 在 base 上独立加「规划」轴——不叠加 demo2 的记忆轴、不叠加 demo3 的工具轴）

demo4 的「规划」就解决这两件事，分两个维度：

- **(A) 单次任务的内部规划**：增加一个 **Plan 工具**（对应 Claude Code 的 TodoWrite）——遇到复杂任务时，让 LLM 先把它拆成多个 step，再一个一个小目标去完成
- **(B) 跨任务的复用规划**：增加一个 **Skill 加载器**——把"常见任务的最佳实践"预先写成一个预制工作流，下次同类任务直接套用，不用每次重新发明轮子

### 【屏显 · 表格 1-1 — demo4 vs demo1】

| 维度 | demo1 | demo4 |
|---|---|---|
| **本地工具** | bash / read / write 三件套 | **+ plan = 四件套** |
| **任务规划** | ❌ 走一步看一步 | ✅ **plan：LLM 自动判断复杂度，复杂任务先列步骤** |
| **跨任务复用** | ❌ 每次从零推演 | ✅ **Skill：预消化的工作流，触发词命中即注入 prompt** |
| **谁决定何时规划** | — | **LLM 自主决策**（简单任务跳过 plan 直接做） |

### 【口播 · base 不变】

注意——**demo1 的核心都没动**：

1. **ReAct 主循环不变**（LLM → tool_use → tool_result → ... → end_turn）
2. **三个本地工具不变**（execute_bash / read_file / write_file 原封不动）
3. **system prompt + messages 结构不变**

我们只做**两件新事**：加一个本地工具 `plan`、加一个 Skill 加载器。

### 【屏显 · 表格 1-2 — ReAct vs Plan 的对比】

| 维度 | ReAct（demo1） | Plan（demo4） |
|---|---|---|
| **决策粒度** | 单步——每轮只看下一步 | 单步 + 全局——先列步骤，再逐步执行 |
| **可观测性** | 只有工具结果可见 | **plan 让步骤状态可见**（pending / in_progress / completed） |
| **目标锚定** | 靠 LLM 在 messages 里"记住" | **靠待办清单显式锚定**，每完成一步就标记 |
| **跨任务复用** | 每次重新想 | **Skill 把工作流预消化**，触发词命中直接套用 |

### 【口播 · 规划的两类增量】

所以 demo4 给的规划能力分两层：

1. **单次任务的内部规划**（plan）——LLM 自己判断任务复杂度，简单任务直接 ReAct，复杂任务先列待办清单再逐步推进。对应 Claude Code 的 TodoWrite 工具。
2. **跨任务的复用规划**（Skill）——把"代码审查该怎么做""抓数据该用哪个工具"这类**重复性工作流**预先写成 SKILL.md，用户输入命中触发词时把 body 注入 system prompt。对应 Claude Code 的 Skill 工具。

这两件事合在一起就是 demo4 的轴。接下来两章分别拆解。

## 第 2 章 机制一：plan（自动决策版）

### 【口播 · plan 是什么】

Claude Code 里有个工具叫 **TodoWrite**——demo4 里我们把它实现成 `plan` 工具。它的作用很简单：让 LLM 把当前任务拆成显式的 step 列表，每步带状态（pending / in_progress / completed），完成一步就更新一下。

听起来很朴素，但它的本质是**把"规划阶段"从 LLM 脑子里搬到屏幕上**：

- 用户能看到 Agent 当前的待办清单（透明）
- LLM 自己也能看到（避免迷失目标）
- 每步状态变化都是一次"自我承诺"（增强一致性）

### 【屏显 · 代码 2-1 — plan 工具定义（agent.py Part 2，第 166 行）】

```python
{
    # === demo4 新增 ===
    "name": "plan",
    "description": (
        "更新当前任务的待办步骤清单（规划用）。"
        "**使用时机**：只在**复杂的多步任务**（3 步以上、多工具协作、步骤间有依赖）开头规划阶段调用，"
        "把整个任务拆成显式 step 列表；后续每完成一步就再调一次本工具更新对应 step 的状态。\n\n"
        "**不要滥用**：简单的一两步任务（如「统计 .py 文件数」「读一下某文件」）"
        "请直接用 execute_bash / read_file 完成，**不要**为了用而用 plan。\n\n"
        "**状态规则**：同一时刻最多只能有 1 个 step 处于 in_progress；"
        "开始下一步前，把上一步从 in_progress 改为 completed，把下一步从 pending 改为 in_progress。"
    ),
    "input_schema": { ... },   # todos 数组，每项 {subject, status}
}
```

### 【口播 · description 的关键设计】

description 里有两段引导，决定了 LLM 是否会用、何时用：

1. **「3 步以上、多工具协作、步骤间有依赖」**——给出**正向触发条件**
2. **「不要滥用：简单的一两步任务请直接用 execute_bash / read_file」**——给出**反向抑制条件**

这两段缺一不可。只给正向条件，LLM 会**滥用**（每个任务都列待办）；只给反向条件，LLM 会**从不使用**。

### 2.1 plan 的实现

### 【屏显 · 代码 2-2 — plan 的实现（agent.py Part 3，第 281 行）】

```python
# todos.md 落盘路径——让用户能看到 Agent 当前的「待办清单」
TODOS_FILE = os.path.join(os.path.dirname(os.path.abspath(__file__)), "todos.md")

# 状态标记符号（终端可视化）
_STATUS_MARK = {"pending": "[ ]", "in_progress": "[→]", "completed": "[✓]"}


def plan(todos: list) -> str:
    """更新待办清单（demo4 新增）。
    落盘到 todos.md 只是给人看的副产物；真正的作用是把"规划阶段"在 ReAct 轨迹里显式化。
    """
    lines = ["# 当前任务待办清单\n"]
    n_by_status = {"pending": 0, "in_progress": 0, "completed": 0}

    for i, item in enumerate(todos, 1):
        subject = str(item.get("subject", "")).strip()
        status = str(item.get("status", "pending")).strip()
        n_by_status[status] += 1
        lines.append(f"{i}. {_STATUS_MARK[status]} {subject}")

    text = "\n".join(lines)
    # 同步落盘（覆盖式）——用户可随时打开 todos.md 查看
    with open(TODOS_FILE, "w", encoding="utf-8") as f:
        f.write(text + "\n")

    # 终端 pretty-print（让 verbose 模式轨迹清晰）
    print("\n" + "─" * 50); print(text); print("─" * 50)

    return (
        f"[已更新] 共 {len(todos)} 步：✓{n_by_status['completed']} 完成 / "
        f"→{n_by_status['in_progress']} 进行中 / [ ]{n_by_status['pending']} 待办。"
        f"清单已同步写入 {TODOS_FILE}"
    )
```

### 【口播 · 关键不是落盘】

注意 `todos.md` 落盘只是**副产物**——给人看的、给 debug 用的，有没有都行。真正的作用有两层：

1. **终端 pretty-print**——让 verbose 模式的运行轨迹里出现一张可视化的待办清单表，LLM 每次调 plan 都会刷一次表
2. **状态变化在 messages 里被 LLM 看到**——返回的字符串「共 5 步：✓1 完成 / →1 进行中 / [ ]3 待办」会作为 tool_result 回到 messages，下一轮 LLM 就能看到自己当前的进度

### 2.2 自动决策：LLM 自己判断要不要用 plan

### 【口播 · 为什么是「自动」模式】

demo4 的 plan 是**自动决策版**——没有 `/plan` 命令、没有显式的"现在开始规划"开关。LLM 看到工具列表里有 plan，自己根据任务复杂度决定何时用。

system prompt 里的引导（agent.py Part 5，`build_system_prompt` 第 520 行）：

```
**任务复杂度自判规则**：
- 简单任务（1-2 步、单一工具）→ 直接 ReAct，跳过 plan
- 复杂任务（3+ 步、多工具协作、有依赖）→ 先 plan 列步骤，再逐步执行并更新状态
```

**为什么不做手动 `/plan` 模式？**

手动模式（用户输入 `/plan <任务>` 强制进入 plan-only 模式）确实更可控，但教学对照会复杂化——需要分两条代码路径（plan 模式 vs 执行模式）。Claude Code 自己也是 LLM 自动决策为主，`/plan` 是次要触发。所以 demo4 只做自动版，把「LLM 怎么判断复杂度」这件事放在聚光灯下。

## 第 3 章 机制二：Skill（预消化的工作流）

### 【口播 · 为什么需要 Skill】

假设你经常让 Agent 去某个网站抓一篇文章、或者从 YouTube 下一个视频。每次如果不预先告诉它怎么做，它会**从零开始试探**：先 ping 一下网站通不通、试图加载网页、再试图下载——很多网站有反爬，它可能还要去网上搜各种方案一个个试。

但如果你之前已经成功做过一次，知道该用哪个工具、怎么避开反爬，就可以把这套方案**固化**下来，下次直接套用，不用重新发明轮子。

代码审查也是同样的道理。每次让 Agent「review 一下 xxx 文件」，LLM 都要重新推演：「review 该怎么做？先读文件，然后检查命名、安全、错误处理……」。推演出来的步骤**每次可能不一样**——同一份代码每次审查结果都不一致，这跟我们的预期不符。

**Skill 解决的就是「重复发明轮子」**：把这类工作流预先写成一个 `skills/review.md`，触发词命中时把 body 注入 system prompt，LLM 就**按预设流程执行**，不再每次重新想。对应 Claude Code 的 Skill 工具——你在 Claude Code 里看到的 `commit` / `review-pr` 这些命令，本质就是预消化的工作流。

### 【屏显 · 表格 3-1 — Skill vs 普通工具】

| 维度 | 普通工具（execute_bash 等） | Skill |
|---|---|---|
| **本质** | 单步操作（执行 shell 命令） | **多步工作流模板**（先做什么、再做什么） |
| **触发** | LLM 自主调用 | **用户输入命中 triggers 关键词** |
| **注入位置** | tools 参数 | **system prompt**（一段额外的 markdown 指令） |
| **作用** | 扩展 LLM 的「手脚」 | **教 LLM 怎么协调手脚做一类事** |

### 3.1 Skill 文件格式

### 【屏显 · 代码 3-1 — skills/review.md 的格式】

```markdown
---
name: review
description: 代码审查工作流——按规范步骤检查代码改动并给出结构化意见
triggers: ["代码审查", "代码评审", "审查代码", "code review", "帮我 review", "review 一下"]
---

# 代码审查工作流

收到「审查代码」类任务时，**必须**按以下步骤执行，不要直接下结论：

## 步骤 1：收集改动
- 如果有 git 仓库：用 execute_bash 跑 git diff 或 git log --oneline -5
- 如果没有 git 或要审整个文件：用 read_file 直接读目标文件

## 步骤 2：逐项检查
| 维度 | 检查点 |
|---|---|
| 命名 | 变量/函数/类名是否清晰、一致 |
| 错误处理 | try/except 是否吞掉异常 |
| 安全 | 命令注入 / SQL 注入 / 硬编码密钥 |
| ... | ... |

## 步骤 3：结构化输出
最终回复必须包含三段：
- ✅ 优点
- ⚠️ 问题（🔴 阻塞 / 🟡 建议 / 🟢 可选）
- 💡 改进建议
```

### 【口播 · frontmatter 三字段】

每个 skill 文件用 YAML frontmatter 声明三件事：

1. **`name`**——skill 的唯一标识（注入 prompt 时引用）
2. **`description`**——一句话说明做什么（启动时被加载到「可用 Skills」元信息段，LLM 据此判断要不要用）
3. **`triggers`**——关键词数组（用户输入命中任一关键词，则激活此 skill）

frontmatter 之后的 body 是**真正的工作流正文**——只有在 skill 被激活时才会注入 system prompt（启动时只加载元信息，不加载 body）。

```python
    不引入 PyYAML 依赖——教学代码保持零额外依赖。"""
    # 用正则匹配首尾 ---，再分别抽取 name/description/triggers
    ...


def load_skills() -> dict:
    """扫描 skills/*.md，解析 frontmatter，返回 {name: {description, triggers, body}}"""
    skills = {}
    for fname in sorted(os.listdir(SKILLS_DIR)):
        if not fname.endswith(".md"):
            continue
        with open(os.path.join(SKILLS_DIR, fname), "r", encoding="utf-8") as f:
            raw = f.read()
        meta, body = _parse_frontmatter(raw)
        name = meta.get("name") or fname[:-3]
        skills[name] = {"description": ..., "triggers": ..., "body": body}
    return skills


def match_skill(user_input: str, skills: dict) -> Optional[str]:
    """用户输入命中某 skill 的任一 trigger 关键词则返回该 skill name"""
    text = user_input.lower()
    for name, info in skills.items():
        for trigger in info["triggers"]:
            if trigger.lower() in text:
                return name
    return None
```

### 【口播 · 不引入 PyYAML 的设计取舍】

注意 `_parse_frontmatter` 用正则实现简易 YAML 解析，而不是 `import yaml`。这是个有意的取舍：

- **优点**：零额外依赖，复制 agent.py + skills/ 目录到任何机器都能跑
- **代价**：只支持 `name: str` / `description: str` / `triggers: [str, ...]` 三个字段，不支持嵌套结构

教学代码保持零依赖是值得的——读者下载即跑，不被 `pip install` 卡住。

### 3.3 system prompt 的三层叠加

### 【屏显 · 代码 3-3 — build_system_prompt（agent.py Part 5，第 504 行）】

```python
def build_system_prompt(skills: dict, activated_skill: Optional[str]) -> str:
    """
    构造 system prompt（三层叠加）：
        1. 基础说明（角色 + 工具清单 + plan 使用指引）
        2. 可用 Skills 元信息（启动时注入一次，每次都带）
        3. 当前激活 skill 的 body（本次输入命中时注入；否则空段）
    """
    parts = [
        "你是一个有用的助手，可以通过工具与本地系统交互完成任务。",
        "可用工具：1. execute_bash ... 4. plan ...",
        "**任务复杂度自判规则**：...",
    ]

    # 第二层：可用 Skills 元信息
    skill_meta = build_skill_metadata_section(skills)
    if skill_meta:
        parts.append(skill_meta)

    # 第三层：当前激活 skill 的 body
    if activated_skill and activated_skill in skills:
        body = skills[activated_skill]["body"]
        parts.append(
            f"\n## 当前激活的 Skill：{activated_skill}\n\n"
            f"本次用户输入命中了 `{activated_skill}` skill 的触发词。"
            f"**必须严格按照下面的工作流执行**：\n\n{body}\n"
        )

    return "\n".join(parts)
```

### 【口播 · 三层的不同时机】

这三层叠加的**注入时机不同**，这是 Skill 机制的精髓：

| 层 | 内容 | 何时构造 |
|---|---|---|
| **第 1 层** | 基础说明 + 工具清单 + plan 使用引导 | 每轮 user 输入时构造（因为后两层可能变） |
| **第 2 层** | 所有 skill 的 name + description + triggers 列表 | 启动时一次性加载到字典，每轮 prompt 都带上 |
| **第 3 层** | **当前激活** skill 的 body 正文 | **每轮 user 输入时重新 match_skill 决定** |

**为什么不让所有 skill 的 body 都常驻 prompt？**

skill 一多就会撑爆上下文（每个 body 几百到几千 token）。第二层只放元信息（每个 skill 一两行），第三层只在命中时才把对应 body 注入——这样 100 个 skill 也只增加几百 token 元信息开销。

### 【口播 · run_agent 里的注入时机（agent.py Part 5，第 551 行）】

```python

    if activated:
        print(f"[Skill] 用户输入命中 → 激活 skill: {activated}")

    system_prompt = build_system_prompt(skills, activated)
    messages = [{"role": "user", "content": user_input}]
    # ... 后面是同 demo1 的 ReAct 循环
```

**关键点**：每次用户输入时调一次 `match_skill` 决定第三段——不同输入激活不同 skill。第一次输入「review 一下」激活 review skill；下一次输入「统计 .py 文件数」就一个 skill 都不激活，第三段为空。

## 第 4 章 真实演示案例

> 以下三个场景为 **demo4-plan 真实运行轨迹**（Windows 10 + Git Bash + 智谱 glm-5.2 网关），非虚构。三个场景分别对应「简单任务跳过 plan」「复杂任务 plan 列步骤」「Skill 触发」。

### 4.1 场景 A：简单任务——LLM 自动跳过 plan

**用户输入**：

> 统计 demo4-plan 目录下 .py 文件的数量

**预期**：1-2 步任务，LLM 应跳过 plan，直接 execute_bash。

**实测轨迹**：

```
用户: [Skill] 未命中任何 skill（可用: review）

----- ReAct 第 1 轮 -----
[LLM 决策] stop_reason = tool_use
  - tool_use : execute_bash({'command': 'find demo4-plan -name "*.py" -type f | wc -l'})
[工具结果] 2

----- ReAct 第 2 轮 -----
[LLM 决策] stop_reason = end_turn
  - text     : `demo4-plan` 目录下共有 **2** 个 `.py` 文件。

助手: `demo4-plan` 目录下共有 **2** 个 `.py` 文件。
```

### 【口播 · 关键观察】

注意三个点：

1. **`[Skill] 未命中任何 skill`**——match_skill 检测到「统计」不命中任何 review skill 的 triggers，第三段为空
2. **LLM 没有调 plan**——它判断这是 1 步任务（一个 find 命令搞定），直接 execute_bash
3. **2 轮 ReAct 完成**——一轮工具调用 + 一轮总结，跟 demo1 完全一样

这正是「自动决策」的价值——简单任务**不被 plan 拖累**。

> 注：本次运行 demo4-plan 目录下有 `agent.py` + 一个临时驱动脚本，故结果为 2；纯净 demo4 只有 `agent.py` 一个 .py 文件。

### 4.2 场景 B：复杂任务——plan 列 5 步逐步推进

**用户输入**（明确分 5 步、有依赖、要求按序执行）：

### 4.3 场景 C：Skill 触发——review 工作流自动激活

**用户输入**：

> 帮我 review 一下 demo4-plan/agent.py 这个文件

**预期**：match_skill 命中 review（triggers 含「帮我 review」），review.md 的 body 注入 system prompt，LLM 按工作流执行。

**实测轨迹（共 5 轮）**：

```
用户: [Skill] 用户输入命中 → 激活 skill: review

----- ReAct 第 1~4 轮 -----   ← 工作流步骤 1：收集改动（读完整文件）
[LLM 决策] tool_use : read_file({'path': 'demo4-plan/agent.py'})           ← 首次读取（被截断）
[LLM 决策] tool_use : read_file(...) + execute_bash({'command': 'cd demo4-plan && wc -l agent.py && tail -n +320 agent.py'})   ← 补读尾部
[LLM 决策] tool_use : read_file(...)
[LLM 决策] tool_use : execute_bash({'command': "cd demo4-plan && sed -n '300,560p' agent.py"})   ← 精确补读 Part 3 路由表 + Part 4 Skill 加载器

----- ReAct 第 5 轮 -----   ← 工作流步骤 3：结构化输出
[LLM 决策] stop_reason = end_turn
  - text: 现在我已经完整看过了整个文件（686 行，5 个 Part）。下面给出结构化的代码审查报告。
```

**最终助手回复（结构化报告，节选）**：

```
## ✅ 优点
1. 结构清晰、注释极其到位：5 个 Part 分区明确，每个 Part 都有大段「为什么这样做」的设计说明。
   plan 的 description 写得尤其好——既说了「使用时机」又说了「不要滥用」，对 LLM 的引导很精准。
2. 错误处理路径完整：四个工具函数都把异常字符串化返回给 LLM（而不是抛崩）。
3. Skill 加载器零额外依赖：_parse_frontmatter 自己用正则解析 YAML frontmatter 而非引入 PyYAML。
4. 可观测性强：_print_messages + _preview + plan 的 pretty-print，让 ReAct 调试轨迹清晰可读。

## ⚠️ 问题
### 🔴 阻塞级（必须改）
1. execute_bash 的 shell=True + 用户/LLM 输入拼接 = 命令注入风险（agent.py 第 220 行）
   command 直接来自 LLM 的 tool_use 输入。虽然是本地工具（执行者即用户自己），
   但注释「# 让命令拥有更强能力」把风险当作卖点，演示型代码里是坏示范。
2. read_file 截断信息不准确（agent.py 第 247 行）
   截断后返回「共 X 字符」是原始长度，而 LLM 实际只拿到前 10000 字符，
   可能导致 LLM 误判文件已完整读取（本场景就触发了多次补读）。

- match_skill 子串匹配过宽（第 448 行）：trigger="review" 会误命中 "preview"；
  英文 trigger 应加 \b 词边界。
- _parse_frontmatter 的 triggers 只支持单行 [a,b,c]，多行 YAML 列表会静默返回空（第 368 行）。
- API_TIMEOUT_MS = 3000000（50 分钟）过大，网络挂起会长时间卡死。
- 模块级全局 client = None（第 90 行），run_agent 直接引用全局，不可独立测试。

## 💡 改进建议（按优先级）
| 优先级 | 位置 | 建议 |
|---|---|---|
| 🔴 P0 | execute_bash 第 220 行 | 注释改为明确安全告示，或加 allowlist 机制 |
| 🔴 P0 | read_file 第 247 行 | 截断提示补充「已展示前 10000 / 共 X 字符」 |
| 🟡 P1 | match_skill 第 448 行 | 英文 trigger 加 \b 词边界 |
| 🟡 P1 | client 第 90 行 | run_agent 入参加 assert client is not None |
```

### 【口播 · 关键观察】

1. **`[Skill] 用户输入命中 → 激活 skill: review`**——match_skill 把「帮我 review」匹配到 triggers 数组里的 `"帮我 review"`，返回 `"review"`
2. **LLM 严格按 review.md 的工作流执行**——多次 read_file + sed 读完整文件（步骤 1：收集改动）→ 输出三段式报告（步骤 3）。和场景 B 一样，`read_file` 截断逼着 LLM 自己用 `sed`/`tail` 分段补读
3. **输出格式与 review.md 的 body 模板一致**——✅ 优点 / ⚠️ 问题（🔴🟡🟢 三级）/ 💡 改进建议，完全按 body 里写的结构
4. **没有泛泛而谈**——每条问题都指向具体行号（「第 220 行」「第 247 行」），这是 body 里「禁止凭文件名猜测」指令的功劳

### 【屏显 · 表格 5-1 — demo4 轴覆盖回顾】

| 轴 | 所属 Demo | 核心机制 | demo4 实现 |
|---|---|---|---|

| **Skill**（预消化工作流） | **demo4** | frontmatter triggers 命中 → body 注入 system prompt | ✅ skills/*.md + load_skills + match_skill |

### 【口播 · 轴公式的对齐】

按总览的「视角 A」公式：

```
demo1: base = LLM × 工具 × 循环 × 状态
demo4: = base × 规划            （plan 单次规划 + Skill 跨任务复用）
```

demo4 基于 demo1 base，**不叠加** demo2 的记忆轴、demo3 的工具轴。读者可以按任意顺序学 demo2 / demo3 / demo4——三条轴正交。

### 【口播 · 两个机制的本质】

plan 和 Skill 都是「规划」，本质都是**工具**，区别在触发方式：

| 机制 | 作用范围 | 持久性 | 触发 |
|---|---|---|---|
| **plan** | **单次任务内部** | 任务结束即丢弃（todos.md 是副产物） | LLM 按任务复杂度自动判断 |
| **Skill** | **跨任务复用** | 永久（skills/*.md 文件持续存在） | 用户输入命中 triggers 关键词 |

简单说：**plan 是「这次怎么做」，Skill 是「这类事该怎么做」**。一个靠任务复杂度决策，一个靠关键词命中触发。

---

*demo4 = base × 规划。plan 给单次任务一个"显式计划"，Skill 给一类任务一个"标准答案"。*
