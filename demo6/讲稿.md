## 第 1 章 问题：messages 为什么会爆

### 【口播 · 回顾 demo1–5】

demo1 到 demo5 我们一直在**加东西**——

> ### demo1 = LLM × 工具 × 循环
> ### demo2 = demo1 × 记忆 × 规划
> ### demo3 = demo2 × Rules × MCP
> ### demo4 = demo3 × Subagent
> ### demo5 = demo4 × Agent 类 × Team 类

每加一层能力，大模型要消化的上下文就更多一层。但有一个矛盾我们一直回避了——**多轮 ReAct 会让 messages 越攒越多**。

### 【屏显 · 图 1-1 — 一轮 ReAct 产生多少条消息】

```
[用户]   请帮我做 X                  ← +1 条
[LLM]    我先调工具 A                 ← +1 条（assistant tool_use）
[工具]   A 的返回结果                 ← +1 条（user tool_result）
[LLM]    再调工具 B                   ← +1 条
[工具]   B 的返回结果                 ← +1 条
[LLM]    完成，结果是 ...             ← +1 条
```

**每一轮工具调用至少 2 条消息**：大模型回复 1 条 + 工具结果 1 条。一个 5-step 的 Plan 任务，每个 step 跑 2 轮工具，就是 20 条。加上 system prompt 注入的 rules、plan 的 tool_result、step 之间的提示语……轻松破 30 条。

### 【口播 · 撞顶即崩】

任何大模型的上下文窗口都有上限：

> ### Context Window 是有限的——撞顶即崩

GLM-5.2 现在是 1M tokens，听起来很大，但一个长任务跑几百轮工具调用照样会爆。一旦爆了，请求直接报错，任务就失败了。

**这就是 demo6 要解决的问题——怎么让 messages 不撞顶。**

## 第 2 章 四种解决思路

### 【口播】

我们有四个方向可以选：

### 【屏显 · 表格 2-1 — 四种压缩思路对比】

| 方案 | 思路 | 优 | 劣 |
|---|---|---|---|
| ① **扩窗口** | 换更大 context 的模型（如 GLM-5.2 的 1M） | 零代码成本 | 治标不治本，超长任务仍会撞顶 |
| ② **限循环** | 限制 step 次数，爆了就重启 | 实现简单 | 粗暴，失败重试代价高 |
| ③ **阶段化** | 只保留最近 N 条（demo2 滑动窗口思想） | 实现简单 | 丢弃关键信息，大模型"失忆" |
| ④ **压缩** | 旧消息让大模型做成摘要，保留最近几条原文 | 保留要点 | 多一次 LLM 调用，可能丢细节 |

### 【口播 · 逐个评点】

- **方案 ① 扩窗口**：好，但治标不治本。GLM-5.2 给到 1M tokens 已经很大了，但简单任务我们不希望为窗口焦虑，复杂任务 1M 照样不够。
- **方案 ② 限循环**：粗暴。任务没做完就强行打断，用户体验差。
- **方案 ③ 阶段化**：这是 demo2 的滑动窗口做法，只拿最后 N 条——**会丢关键信息**。大模型可能突然不知道自己之前创建过什么文件、得到过什么关键数据。
- **方案 ④ 压缩**：让大模型把旧消息做成摘要，保留最近几条原文。**这是质量和成本的最佳平衡点**，也是生产级 Agent（如 Claude Code）的核心做法。

> ### demo6 选方案 ④ —— 对话历史压缩

## 第 3 章 压缩原理（图解）

### 【口播 · 压缩前后的对比】

假设我们设两个参数：
- **阈值**：messages 达到 N 条触发压缩（demo6 设 8，演示用）
- **保留**：最近 K 条不动（demo6 设 4）

### 【屏显 · 图 3-1 — 压缩前的 messages】

```
messages = [
  (0) user:      首条任务                          ┐
  (1) assistant: tool_use                          │
  (2) user:      tool_result                       │
  (3) assistant: tool_use                          │ 旧消息
  (4) user:      tool_result                       │ （待压缩）
  (5) assistant: 文本                              │
  (6) user:      step 2 提示                       ┘ ← 理想切点
  ┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄
  (7) assistant: tool_use                          ┐ 最近区
  (8) user:      tool_result                       │ （保留原文）
  (9) assistant: tool_use                          │
  (10) user:     tool_result                       ┘
]
```

### 【屏显 · 图 3-2 — 压缩后的 messages】

```
messages = [
  (0) user:      [对话历史摘要] 用户最初要求...     ← 摘要（新）
  (1) assistant: 好的，我已读取历史摘要...          ← 过渡（新）
  (2) assistant: tool_use                          ┐
  (3) user:      tool_result                       │ 最近的
  (4) assistant: tool_use                          │ （原文）
  (5) user:      tool_result                       ┘
]
```

### 【口播 · 几个关键点】

1. **system prompt 永远保留**——它不在 messages 数组里，是 `client.messages.create` 的独立参数
2. **最近 K 条原文保留**——这是当前 step 最精确的上下文，不能压
3. **旧消息送大模型做摘要**——一次额外的 LLM 调用，换 messages 长度大幅缩短
4. **插一条过渡消息**——让大模型不会因为 messages 序列突变而困惑

## 第 4 章 代码实现：`compact_messages`

### 4.1 函数签名与参数

### 【屏显 · 代码 4-1 — 函数签名】

```python
COMPACT_THRESHOLD = 8   # messages 达到 8 条触发压缩
KEEP_RECENT       = 4   # 保留最近 4 条不动

def compact_messages(
    messages: list,
    threshold: int = COMPACT_THRESHOLD,
    keep_recent: int = KEEP_RECENT,
    verbose: bool = True,
) -> list:
    """
    不足 threshold → 原样返回
    达到 threshold → 旧消息压成 summary，返回新组装的 messages
    """
```

### 【口播 · 为什么阈值故意调低】

注意——**这两个数字是故意调低的**，方便演示。8 条就触发、保留 4 条，意味着随便跑两轮工具调用就能看到压缩日志。**生产环境应该按 token 数算，不是按条数**——这个我们第 6 章对比。

### 4.2 第一步：找安全边界

### 【口播 · 切点不能落在 tool_use/tool_result 中间】

这是压缩最容易被忽视的坑——**Anthropic API 要求 `tool_use` 紧跟 `tool_result`，切断就报错**。

如果理想切点正好落在 tool_use 和它的 tool_result 中间，必须向前回溯。

### 【屏显 · 代码 4-2 — `_find_safe_boundary`】

```python
def _is_tool_use_block(block) -> bool:
    """判断一个 content block 是不是 tool_use（兼容 dict / 对象）"""
    t = getattr(block, "type", None) or (block.get("type") if isinstance(block, dict) else None)
    return t == "tool_use"


def _find_safe_boundary(messages: list, ideal_cut: int) -> int:
    """
    从 ideal_cut 向前回溯，跳过以 assistant(tool_use) 结尾的消息——
    tool_use 必须和后面的 tool_result 在同一区，切断会报 API 错误。

    只需检查 tool_use：tool_result 必跟在 tool_use 之后，
    只要 old 区不以 tool_use 结尾，配对就不会被切断。
    （old 区以 tool_result 结尾是安全的——它的 tool_use 在更前面，同在 old 区。）
    """
    cut = ideal_cut
    while cut > 1:
        msg = messages[cut - 1]
        content = msg.get("content")
        if (msg.get("role") == "assistant" and isinstance(content, list)
                and any(_is_tool_use_block(b) for b in content)):
            cut -= 1
        else:
            break
    return cut
```

### 【口播】

简单说就是——**回溯到找到一个不会切断 tool 配对的切点**。最坏情况会退到 1（只压首条），但实践中通常只回退 1-2 条。

### 4.3 第二步：旧消息送 LLM 做摘要

### 【屏显 · 代码 4-3 — `_summarize` 的 prompt】

```python
summary_prompt = (
    "你是一个对话历史压缩器。下面是 Agent 之前的多轮工具调用对话历史（已序列化）。\n"
    "请生成一段**紧凑摘要**，要求：\n"
    "1. 保留**关键信息**：用户最初的目标、已经完成的步骤、已经创建/修改的文件、"
    "已得到的关键数据 / 结论\n"
    "2. 保留任何**未完成的承诺**或**待办事项**\n"
    "3. 丢弃冗余的工具调用细节（具体命令、返回的原始字节、重复的中间结果）\n"
    "4. 用第三人称、陈述句，300 字以内\n\n"
    "不要输出任何解释 / markdown 标题 / 前后缀，直接写摘要正文。\n"
    "---\n对话历史：\n"
    + transcript_text
)
```

### 【口播 · 摘要的取舍】

注意 prompt 的措辞——**保留关键信息、丢弃冗余细节**。这是摘要质量的核心：
- ✅ 保留：用户目标、已完成的步骤、已创建的文件、关键数据
- ❌ 丢弃：具体 shell 命令、工具返回的原始字节、重复的中间结果

### 4.4 第三步：重组 messages

### 【屏显 · 代码 4-4 — 重组逻辑】

```python
new_messages = []
# 1) 摘要作为新的首条 user 消息
new_messages.append({
    "role": "user",
    "content": f"[对话历史摘要]\n{summary}",
})
# 2) 过渡消息：伪装成 assistant 已收到摘要
new_messages.append({
    "role": "assistant",
    "content": [{"type": "text", "text":
        "好的，我已读取历史摘要，了解了之前的进展。"
        "请给我最新的任务，我会基于这份摘要继续。"}],
})
# 3) 最近 K 条原文
new_messages.extend(recent_messages)
```

### 【口播 · 为什么要过渡消息】

如果不加这条 `assistant` 过渡——直接把摘要当首条 user 消息——大模型会疑惑："我怎么突然凭空有了一段对话历史？"

加一条 assistant 回应，伪装成正常的对话衔接，大模型就自然接受了。

## 第 5 章 实战演示：一个 4-step 任务的压缩全过程

### 【口播 · 压缩的触发时机】

`compact_messages` 写好了，**什么时候调它**？

答案：**在 Plan 每个 step 开始前调一次**。

### 【屏显 · 代码 5-1 — `run_agent` 的 step 循环】

```python
for i, step in enumerate(steps, 1):
    # ★ demo6 核心：step 开头先压缩
    messages = compact_messages(messages, verbose=verbose)

    step_msg = f"【Step {i}/{len(steps)}】请只执行这一步：\n{step}..."
    messages.append({"role": "user", "content": step_msg})
    run_agent_steps(messages, step_tools, local_fns, system_prompt, verbose)
```

### 【口播 · 为什么选这个时机】

- **Plan 模式天然是多 step 的**——每个 step 内部还有多轮工具调用，messages 增长最快
- **step 之间是天然的检查点**——此时大模型刚做完一个 step，是个安全的压缩时机
- **不是每轮 ReAct 都压**——那样会频繁触发 LLM 调用，成本高；step 级粒度刚好

### 【口播 · 顶层决策前也压一次】

除了 step 循环内，`run_agent` 在**顶层决策前**也调了一次 `compact_messages`：

### 【屏显 · 代码 5-1b — 顶层决策前的压缩】

```python
# 顶层决策前也压一次（首条消息不会触发，但保持习惯一致）
messages = compact_messages(messages, verbose=verbose)
```

首条消息只有 1 条，不会触发压缩——但这样写是为了**保持调用习惯一致**：任何拿 messages 去喂大模型之前，都先走一遍压缩检查。这样不管代码后续怎么扩展（比如顶层前塞了更多上下文），都不会遗漏压缩。

光讲原理不够直观，下面跑一个**真实任务**，逐条数 messages 的增长，看压缩到底在什么时候触发、压成了什么。

### 5.1 演示任务

### 【屏显 · 代码 5-2 — 演示输入】

```
用户: 任务如下：1) 计算 35+47；2) 计算 任务1的结果*9；3) 查上海天气；4) 把 "hello" 写入 hello.txt
```

### 【口播 · 为什么选这个任务】

这个任务天然 4 步、覆盖 3 种工具、且 Step 2 依赖 Step 1 的结果——能同时演示 plan 拆解、跨步依赖、工具试错触发压缩这三件事。

### 5.2 LLM 拆解的 Plan

### 【屏显 · 代码 5-3 — 实际 plan 工具返回】

```json
{
  "steps": [
    "计算 35+47，得到结果1",
    "用结果1*9，得到结果2",
    "查询上海天气",
    "将 'hello' 写入 hello.txt"
  ]

```

### 5.3 真实运行回显

### 【屏显 · 代码 5-4 — 实际运行日志（精简）】

```

```

### 【口播 · 讲解】

几个看点——

1. **Step 1 不压**：messages 才 3 条，没到阈值。但 Step 1 内部试错了 3 次工具（`echo $(( ))` 失败、`python3` 找不到、`python` 成功），这一步就把 messages 从 3 条顶到 10 条
2. **第 1 次压缩（10→6）**：Step 2 开头触发，旧 6 条压成 1 条摘要 + 1 条过渡，最近 4 条原文保留。切点正好落在安全位置，无需回退
3. **第 2、3、4 次压缩（9→7、10→7、10→7）**：理想切点落在 tool_use/tool_result 对中间，各回退 1 格才安全
4. **跨步依赖传下来了**：Step 2 的 LLM 拿到了上一步的 `82`，直接 `82 * 9 = 738`——即使 Step 1 的 messages 被压成摘要，关键数字没丢
5. **最终结果全部正确**：82 / 738 / 多云 22°C / hello.txt（5 字节）

一句话：**压缩是"抓大放小"——保留要点和最近细节，丢掉冗余的中间过程**。

## 第 6 章 demo 实现 vs 生产级实现（兼与 demo2 对比）

### 【口播】

demo6 是教学版，简化了很多。这一章我们做两个对比——先对比生产级实现，再对比容易混淆的 demo2 滑动窗口。

### 6.1 demo6 vs 生产级（如 Claude Code）

### 【屏显 · 表格 6-1 — demo vs 生产级对比】

| 维度 | demo6（教学版） | 生产级（Claude Code 等） |
|---|---|---|
| **触发指标** | 固定条数（8 条） | 基于 **token 数**，按上下文窗口占比触发（如 80%） |
| **压缩粒度** | 一次性把旧消息压成一个 summary | **分层压缩**：最近原文 → 稍远摘要 → 更远压得更细 |
| **保留策略** | 固定保留最近 K 条 + 安全边界 | **智能选择**：关键词、关键决策、文件路径、代码片段等 |
| **prompt 定制** | 通用压缩 prompt | **按场景定制**（coding 场景保留路径/决策；research 场景保留事实引用） |
| **失败处理** | 摘要失败直接抛异常 | **重试 + 降级**（保留原文） |

### 【口播 · 逐项解读】

- **触发指标**：demo6 按条数，简单粗暴。生产按 token 数——因为一条 tool_result 可能是 10 字也可能是 10KB，条数不准。比如上一章 Step 4 试错 10 轮攒到 28 条，但每条都很短，实际 token 数并不高；反过来一条读取大文件的 tool_result 可能顶十几条普通消息。按 token 衡量才精确。
- **压缩粒度**：demo6 一次性全压。生产是**分层**——最近的原文不动、稍远的做粗摘要、更远的做精摘要。这样既省 token 又保精度。
- **保留策略**：demo6 只看位置（最近 K 条）。生产会**智能识别关键信息**——比如一个重要的文件路径、一个关键的决策原因，无论多远都保留。
- **prompt 定制**：demo6 用一个通用 prompt。生产会**按任务类型定制**——coding 任务保留路径和代码片段，research 任务保留事实引用和来源。

### 【口播 · 取舍】

demo6 的简化是**有意为之**——目的是让你一眼看懂压缩的核心原理。生产级的优化（分层、智能保留、场景定制）都是建立在这个核心原理之上的增量改进。

> ### 核心原理：旧消息 → LLM 摘要 → 摘要 + 最近原文重组

### 6.2 demo6 vs demo2 滑动窗口

### 【口播 · 容易混淆】

很多同学会把 demo6 的压缩和 demo2 的滑动窗口记忆搞混——**它们都涉及"保留部分历史"，但是完全不同的两件事**。

### 【屏显 · 表格 6-2 — demo2 memory vs demo6 compact】

| 维度 | demo2（滑动窗口 memory） | demo6（压缩） |
|---|---|---|
| **数据载体** | `agent_memory.md` 外部文件 | messages 数组本身 |
| **保留内容** | 任务级摘要（一次任务一条记录） | 对话级摘要（多轮工具调用压缩成一段） |
| **触发时机** | 任务结束时 append | 任务执行中动态触发 |
| **信息来源** | 用户输入 + 最终结果 | 全部 messages（含工具调用细节） |
| **解决的问题** | **跨任务**记忆（A 任务做完，B 任务能用） | **单任务内**不撞顶（一个长任务自己不爆） |

### 【口播】

简单说——

- **demo2 的 memory 解决的是"跨任务"**——做完 A 任务，结果存盘，B 任务能读
- **demo6 的 compact 解决的是"单任务内"**——一个长任务跑到一半，messages 快爆了，压一下继续跑

> ### 两个机制互补：memory 跨任务，compact 任务内

## 第 7 章 demo1–7 系列回顾

### 【口播 · 整体脉络】

用比喻的方式回顾整个 demo 系列——我们一直在**扩展 Agent 的能力边界**：

### 【屏显 · 表格 7-1 — demo1–7 的能力扩展】

| Demo | 比喻 | 关键能力 |
|---|---|---|
| demo1 | 给它**双手**（工具） | LLM × 工具 × 循环 |

### 【口播 · demo6 的定位】

demo1–5 让 Agent **知道能干什么**——给它工具、给它记忆、给它团队。
demo6 让 Agent **能干得更久**——不让多轮调用把自己撑爆。
demo7 将让 Agent **知道不能干什么**——给它安全和权限控制。
