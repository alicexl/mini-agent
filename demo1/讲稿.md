## 第 1 章 结论：先说结论

### 【口播 · 开场钩子】

你用过 Claude Code、Cursor、Devin 这些工具吗？它们可以自动搜索代码、修改文件、执行命令，看起来像有一个程序员在屏幕里干活。

但是——**没有任何魔法**。这一节，我们用 300+ 行 Python 代码，把这套"自动干活"的底层机制完全拆开。

### 【口播 · 5 维度对比】

要理解 Agent，最快的方法是把它跟普通的 AI 对话放在一起对比。我们看 5 个维度。

### 【屏显 · 表格 1-1】

| 维度 | 普通对话 | Agent |
|---|---|---|
| **交互模式** | 一问一答，用户驱动 | 有循环，目标驱动，自主推进 |
| **能力边界** | 只能生成文本（多模态模型也就是看/听模型支持的模态） | 可调用本地工具 + 远程 MCP 等能力，作用于真实世界 |
| **执行流程** | 用户提问 → 模型回答 | 用户下达任务 → LLM 思考 → 调工具 → 观察 → 再思考，循环 |
| **状态管理** | 靠模型后台上下文做记忆 | 本地维护完整消息历史，不完全依赖大模型记忆 |
| **自主性** | 无 | 自主决定下一步、选工具、判停 |

【口播】你不需要全记。**只需要记住三个增量**：工具、循环、自主性。这是 Agent 区别于 Chatbot 的全部秘密。

### 【口播 · 三要素公式】

我把 Agent 浓缩成一个公式：

> ### Agent = LLM × 工具 × 循环
>
> - **LLM** = 思考能力 → **大脑**
> - **工具** = 本地工具 + MCP 等远程能力 → **手脚**
> - **循环** = 任务分解 + 迭代，直到目标达成

普通对话只是「你问我答」；Agent 是「**为了一个目标，努力去想办法、去完成它**」。

记住这个公式，这一节剩下的内容，都是在解释这三个乘数怎么落地。

## 第 2 章 全局架构

### 【口播】

为了实现上面那三个乘数，我们的最简 Agent 拆成 **4 个部分**。前 3 部分是**静态能力**——告诉系统「我能干什么」；第 4 部分是**动态运行时**——告诉系统「我该怎么干」。

## 第 3 章 逐层解读

### 3.1 Part 1 — LLM 客户端

### 【口播】

第一层很无聊但很重要：**能调通大模型就行**。

本 demo 默认走**智谱 BigModel 的 Anthropic 兼容网关** + `glm-5.2` 模型。关键是——Anthropic SDK 的调用接口是标准，换成官方 API 或别的兼容网关，调用代码一行不用改。

用户只需配置 API Key，三种方式（优先级递减）：
1. 改下面代码里的 `API_KEY` 变量（持久化，推荐）
2. 设环境变量 `ANTHROPIC_API_KEY`
3. 都没设 → 首次运行时交互式输入（仅本次有效）

### 【屏显 · 代码 3-1】

```python
import os
from anthropic import Anthropic

# ↓↓↓ 只需改这一行 ↓↓↓
API_KEY = ""

# 默认配置（一般无需修改）
BASE_URL = "https://open.bigmodel.cn/api/anthropic"   # 智谱 BigModel 兼容网关
MODEL    = "glm-5.2"

def load_config() -> dict:
    """环境变量优先于代码默认值"""
    return {
        "api_key":  os.environ.get("ANTHROPIC_API_KEY") or API_KEY,
        "base_url": BASE_URL,
        "model":    MODEL,
    }

def init_client() -> None:
    """模块级 client 初始化；没配 Key 时交互式提示输入"""
    global client
    config = load_config()
    if not config["api_key"]:
        config["api_key"] = input("请输入 API Key: ").strip()
    client = Anthropic(api_key=config["api_key"], base_url=config["base_url"])
```

### 【口播】

一句话总结这一层：**SDK 照抄就行，不要纠结**。重点是后面三层——工具和循环才是 Agent 的灵魂。

### 3.2 Part 2 — 工具定义

### 【口播】

第二层是**工具说明书**。大模型本身不会执行代码——它只是在每次请求里看到这份说明书，然后**知道自己可以请求调用**哪个函数、用什么参数。真正的执行发生在我们本地。

### 【屏显 · 代码 3-2】

```python
TOOLS = [
    {
        "name": "execute_bash",
        "description": "执行任意 shell 命令，可用于文件操作、系统命令等",
        "input_schema": {
            "type": "object",
            "properties": {
                "command": {"type": "string", "description": "要执行的 shell 命令"}
            },
            "required": ["command"],
        },
    },
    # read_file / write_file 同理
]
```

### 【口播 · 重点强调】

注意这里的关键认知：**Function Calling 是大模型经过训练的能力，不是 prompt 魔法**。

模型在训练阶段见过大量「工具调用」样本，所以它认识这份 schema，会按格式吐出结构化的调用指令。我们在 prompt 里写再多"你有工具"，模型没经过训练也用不起来。

### 3.3 Part 3 — 工具实现 + 路由表

### 【口播】

第三层是真正的执行函数。这一层有三个设计点，每个点都对应一个反例，记住反例就记住了设计：

### 【屏显 · 表格 3-1 — 工具实现的三个设计点】

| 设计点 | 代码 | 不这么做的后果 |
|---|---|---|
| **错误也字符串化返回** | `return f"[错误] {e}"` | 大模型不知道失败，会重复试同一个命令 |
| **设置 60 秒超时** | `timeout=60` | 一个 `sleep 9999` 卡死整个 Agent |
| **`shell=True`** | `subprocess.run(cmd, shell=True)` | `ls *.py \| wc -l` 这种管道直接报错 |

### 【屏显 · 代码 3-3】

```python
def execute_bash(command: str) -> str:
    try:
        result = subprocess.run(
            command,
            shell=True,          # ← 风险换能力
            capture_output=True,
            text=True,
            timeout=60,          # ← 防死循环
        )
        # 错误信息也字符串化 → 让大模型看到后自己调整策略
        ...
    except subprocess.TimeoutExpired:
        return "[错误] 命令执行超时（60 秒）"
```

### 【口播 · 路由表】

接下来一个**关键设计**——**路由表**：

### 【屏显 · 代码 3-4】

```python
# 路由表：工具名 → 实际函数（调度核心）
AVAILABLE_FUNCTIONS = {
    "execute_bash": execute_bash,
    "read_file":    read_file,
    "write_file":   write_file,
}
```

### 【口播】

为什么用字典、不用 `if/elif`？

**三个理由**：
1. **单点注册**——加新工具只改一处
2. **职责分离**——名字和实现解耦
3. **可扩展**——未来加装饰器、权限钩子、日志，只需要包一层

当前我们只做一个最简单的字典映射，这个设计在后续 demo 里还会进一步演化。

### 3.4 Part 4 — Agent 主循环

### 【口播】

最后一层是 Agent 的**心脏**：主循环。每一轮做四件事——决策、判停、行动、感知，对应学术界的 **ReAct** 框架（Reasoning + Acting）。

### 【屏显 · 代码 3-5 — 主循环骨架】

```python
MAX_ITERATIONS = 30  # 防止死循环

def run_agent(user_input: str, verbose: bool = True) -> str:
    messages = [{"role": "user", "content": user_input}]

    for loop_idx in range(1, MAX_ITERATIONS + 1):
        # 1. 决策：大模型思考下一步
        response = client.messages.create(
            model=MODEL, max_tokens=4096,
            system=SYSTEM_PROMPT, tools=TOOLS, messages=messages,
        )

        # 2. 判停：是否结束？
        if response.stop_reason != "tool_use":
            return "".join(b.text for b in response.content if b.type == "text")

        # 3. 行动：执行工具
        messages.append({"role": "assistant", "content": response.content})
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                fn = AVAILABLE_FUNCTIONS.get(block.name)   # 未知工具防御
                if fn is None:
                    result = f"[错误] 未知工具: {block.name}"
                else:
                    result = fn(**block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,  # Tool ID 精确匹配
                    "content": result,
                })

        # 4. 感知：工具结果作为 user 消息追加，下一轮大模型就能看到
        messages.append({"role": "user", "content": tool_results})

    return "[错误] 超过最大循环次数（{}）".format(MAX_ITERATIONS)
```

> **注**：`verbose=True` 时还会在每轮打印决策与工具结果预览（教学演示用，上方骨架省略了这部分打印代码）。

### 【口播】

这一段代码就是整个 demo 的灵魂。我们用一个具体例子把它跑起来。

## 第 4 章 示例解读循环的运行时序

### 【口播】

任务：**「统计当前目录下有多少个 Python 文件，并把结果写入 count.txt」**

我们看 messages 数组怎么一轮一轮长大。

### 【屏显 · 时序图 4-1】

```
第 1 轮 messages：
┌─────────────────────────────────────────────────────┐
│ [system] 你是一个有用的助手，可以通过工具与系统交互... │
│ [user]   统计当前目录下有多少个 Python 文件，         │
│         并把结果写入 count.txt                       │
└─────────────────────────────────────────────────────┘
                            │
                            ▼
                  大模型决策 → tool_use
                  execute_bash("ls *.py | wc -l")
                            │
                            ▼
                  本地执行 → "42"
                            │
                            ▼
第 2 轮 messages（追加了 2 条）：
┌─────────────────────────────────────────────────────┐
│ [system] ...                                         │
│ [user]   统计...                                     │
│ [assistant] tool_use(execute_bash, "ls *.py | wc -l")│
│ [user]   tool_result("42")      ← 工具结果           │
└─────────────────────────────────────────────────────┘
                            │
                            ▼
                  大模型看到「42」+ 原始任务
                  → 决定调用 write_file
                            │
                            ▼
                  本地写入成功
                            │
                            ▼
第 3 轮 messages（追加了 2 条）：
┌─────────────────────────────────────────────────────┐
│ [system] ...                                         │
│ [user]   统计...                                     │
│ [assistant] tool_use(execute_bash)                   │
│ [user]   tool_result("42")                           │
│ [assistant] tool_use(write_file, count.txt, "42")    │
│ [user]   tool_result("[成功] 已写入...")              │
└─────────────────────────────────────────────────────┘
                            │
                            ▼
                  大模型看到写入成功
                  → 判断任务完成
                  → stop_reason = "end_turn"
                  → 返回最终文本
                            │
                            ▼
                  「已统计完成，共 42 个 Python 文件，
                    结果已写入 count.txt」
```

### 【口播 · 关键观察】

请观众注意一个细节：**每一轮 messages 都在变长**。

- 第 1 轮：2 条
- 第 2 轮：4 条
- 第 3 轮：6 条

这就是 Agent **比 Chatbot 贵** 的原因——**每轮都把整个上下文重发给大模型**。token 成本随循环次数**线性增长**。

但这也是它的力量来源：每一轮大模型都看到完整历史，所以能基于"我做过什么、结果是什么"做下一步决策。**短期记忆就是这样产生的。**

**也埋了一个雷**：messages 会**无限增长**，遇到复杂任务时整个上下文可能撑爆大模型的窗口，请求就会失败。这就是后面**记忆模块**要解决的核心问题——既要保留跨任务的长期记忆，又要做压缩防止撑爆窗口。

## 第 5 章 深入理解关键设计

### 5.1 为什么要有 MAX_ITERATIONS？

### 【口播】

默认 30 次。**为什么是 30？**——实测大部分任务在 10 轮内完成，30 是 3 倍安全垫。

**为什么必须有？**——大模型会陷入死循环。某些工具调用（比如等待一个永远不会发生的事件）永远不可能成功，但模型会一直重试。设了上限，程序才能保证最终停下来。

这是最简单的中止策略。后续可以演化出更智能的，比如：**连续多次用不同方法尝试同一任务都失败，就主动放弃**——比单纯循环 30 次再失败要好得多，因为大模型有可能是把任务拆成了多个子任务，单纯按轮数掐断不一定合理。

### 5.2 messages 怎么演化？

### 【口播】

这一节 demo 不涉及 messages 的长期保存。**每次循环都做一件事：把整个 messages 全部带上、发给大模型**。

大模型在所有历史里找当前进度、决定下一步。**短期记忆**由此产生——**只在一个任务循环周期内有记忆**，本次循环结束 messages 就清空，下次任务开始时之前的记忆全无。

这里其实埋了**两个问题**：

1. **短期记忆 vs 长期记忆**：任务和任务之间也可能有关联。比如刚做完第一件事，干第二件事时和第一件事的结果相关——如果 Agent 不知道，又得从零开始把所有背景讲给大模型听，非常啰嗦。这就是**长期记忆**的需求。
2. **上下文撑爆**：把所有记忆都喂给大模型，最终会把上下文窗口塞满。这就是后面 message 和记忆模块要做**压缩**的原因。

### 5.3 大模型如何决定使用工具？

### 【口播 · 核心概念】

这是整节课最重要的一段，请仔细听。

大模型**没有真正执行代码**。它和我们本地 Agent 之间，约定了一个接口——**Function Calling 协议**。

这是一个**协作分工**：

### 【屏显 · 表格 5-1 — 协作分工】

| 角色 | 职责 |
|---|---|
| **大模型**（大脑） | 思考、决策、生成工具调用指令（tool_use） |
| **本地 Agent + 工具**（手脚） | 解析指令、执行真正的函数、返回结果（tool_result） |

### 【口播】

流程是：
1. 我们告诉大模型：「我有这些能力」（tools 参数）
2. 大模型思考后回吐：「**你**去调这个函数，参数是这样」（tool_use block）
3. 我们解析后执行真正的函数
4. 把结果再告诉大模型（tool_result）
5. 大模型判断结果是否符合预期，决定下一步

**模型只负责"想"，我们负责"做"。**

### 5.4 Tool ID 的作用

### 【口播】

每次工具调用带一个 `tool_use_id`，API 协议要求。**它真正的作用不是为了防重名**（工具名本身就是 schema 的 key，不会重复），而是为了——**并行工具调用**。

Anthropic API 支持一次响应返回**多个** tool_use block：比如大模型想同时读 3 个文件，会一次性吐 3 个 `tool_use(read_file)`，每个带不同的 ID。我们没有 ID 就没法把 3 个 result 一一对应回 3 个 call。

### 【屏显 · 代码 5-1 — 并行 tool_use 示例】

```python
# 大模型可能一次返回多个 tool_use：
response.content = [
    ToolUse(id="tool_1", name="read_file", input={"path": "a.py"}),
    ToolUse(id="tool_2", name="read_file", input={"path": "b.py"}),
    ToolUse(id="tool_3", name="read_file", input={"path": "c.py"}),
]

# 我们执行后，必须用 tool_use_id 把结果一一对应：
tool_results = [
    {"type": "tool_result", "tool_use_id": "tool_1", "content": "..."},
    {"type": "tool_result", "tool_use_id": "tool_2", "content": "..."},
    {"type": "tool_result", "tool_use_id": "tool_3", "content": "..."},
]
```

### 5.5 循环的"门"：stop_reason

### 【口播】

最后一个关键设计——循环退出条件。`stop_reason` 是大模型返回的"信号灯"：

### 【屏显 · 表格 5-2 — stop_reason 三种值】

| `stop_reason` | 含义 | Agent 行为 |
|---|---|---|
| `"tool_use"` | 模型要调工具 | 执行工具，继续循环 |
| `"end_turn"` | 模型认为任务完成 | 退出循环，返回文本 |
| `"max_tokens"` | 输出长度到上限被截断 | 隐式停止（也算异常退出） |

### 【口播】

我们代码里 `if response.stop_reason != "tool_use"` 就是这扇门——不是 `tool_use` 就走退出分支。三种 stop_reason 都覆盖到了。

## 第 6 章 总结和展望

### 6.1 Agent 的本质

### 【口播】

最后我们再压缩一次。Agent 是什么？

它是一个**执行的过程**，做三件事：

1. **行动**：调用本地工具（Acting）
2. **感知**：通过工具结果感知本地有什么、能获取什么（观察）
3. **决策**：依赖大模型的大脑思考下一步（Reasoning）

**思考 → 行动 → 观察 → 再思考** —— 这就是 **ReAct**（Reasoning + Acting），一直在循环中迭代，直到大模型判断任务完成。

### 【屏显 · ReAct 循环图】

```
        ┌─────────────────────────────┐
        │                             │
        ▼                             │
   ┌─────────┐                  ┌─────────┐
   │  思考   │  ← messages ←    │  观察   │
   │ (LLM)   │                  │ (结果)  │
   └─────────┘                  └─────────┘
        │                             ▲
        │  tool_use                   │
        ▼                             │
   ┌─────────┐                        │
   │  行动   │  ── tool_result ──────┘
   │ (工具)  │
   └─────────┘
```

### 【口播】

思考 → 行动 → 观察 → 再思考。不管是什么大模型、不管是什么 Agent——**Claude Code、Cursor、Devin——底层都是这个循环**。

你看到它们自动搜索代码、修改文件、执行命令，背后就是这样一个驱动。本 demo 用最少的代码，把这套原理完整展现了出来——**非常简洁，非常精致**。

但这个最简 Agent **能干活，却像一条金鱼**——做完任务就忘记。

它有 **3 个局限**，贯穿后续 6 个 demo 一一解决：

### 【屏显 · 表格 6-1 — demo1 局限 → demo2-7 路线图】

| # | 局限 | 现状 | 解决它的 demo |
|---|---|---|---|
| 1 | **只有短期记忆** | 一个任务循环内有记忆，循环结束就全忘 | **demo2**：`agent_memory.md` 跨任务长期记忆；**demo6**：`compact_messages` 解决 messages 在长任务里撑爆窗口的另一面 |
| 2 | **没有规划** | 完全靠大模型当下指令，走一步看一步 | **demo2**：手动开关的 Plan 模式；**demo3**：升级为 LLM 在 `run_agent` 顶层自动决策是否拆步骤 |
| 3 | **无行为约束** | 想执行什么就执行什么（`rm -rf` 都行） | **demo3**：Rules 约束代码风格等"软规范"；**demo7**：三道防线（黑名单 / 用户确认 / 输出截断）兜住"硬危险" |

此外还有**两条能力扩展线**，不属于"补局限"，而是把 Agent 做强：

| Demo | 加的能力 | 比喻 |
|---|---|---|
| **demo4** | Subagent（一次性子 Agent） | 给它**临时助手**——把独立子任务外包出去 |
| **demo5** | Team（持久 Agent + 状态机 + 质检员） | 组建**正式项目团队**——有身份、有记忆、能通信、会被质检 |

### 【口播 · 整体脉络】

整个 demo1–7 是一个**渐进扩展**的过程——前 6 个 demo 一直在**加能力**（工具 → 记忆 → 规划 → 协作 → 团队 → 压缩），demo7 转向给能力加**约束**（安全边界）。

### 【屏显 · 表格 6-2 — demo1–7 能力 × 约束矩阵（预告）】

| Demo | 主题 | 关键能力 / 约束 |
|---|---|---|
| demo1 | LLM × 工具 × 循环 | ReAct、本地工具 |
| demo2 | 记忆 × 规划 | `agent_memory.md`、手动 Plan |
| demo3 | Rules × MCP | 行为约束、JSON-RPC 远程工具、Plan 自动决策 |
| demo4 | Subagent 分工 | 主 Agent 派生一次性独立 Subagent |
| demo5 | Team 协作 + 事件驱动 | 持久 Agent + 状态机 + 质检员持续监听 |
| demo6 | 上下文压缩 | `compact_messages` 动态压缩对话历史 |
| demo7 | 安全边界 | 三道防线：黑名单 / 用户确认 / 输出截断 |
