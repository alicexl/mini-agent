## 第 1 章 结论：demo2 比 demo1 多了什么

### 【口播 · 开场】

上一节 demo1，我们用 300 多行代码拼出了 Agent 的核心公式：

> ### Agent = LLM × 工具 × 循环

但我们也留了一个遗憾——**这条公式只解决「能不能干活」，不解决「干完就忘」**。

demo1 的 Agent 就像一条**金鱼**：上一个任务做完，下一次再问，**什么都不记得**。复杂任务丢给它，它还会**走一步看一步**，很容易迷失在细节里、走偏方向、达不到目标。

demo2 就是要补上这两块：**让它记得过去，让它规划未来**。

### 【屏显 · 表格 1-1 — demo2 vs demo1】

| 维度 | demo1 | demo2 |
|---|---|---|
| **工具集** | execute_bash / read_file / write_file | **不变** |
| **任务执行** | 单步 ReAct 循环 | 单步 ReAct **不变**（不考虑并发） |
| **跨任务记忆** | ❌ 任务结束 messages 清空 | ✅ `agent_memory.md` 文件持久化 |
| **任务规划** | ❌ 整个任务丢给 LLM 走一步看一步 | ✅ Plan 模式先拆 3-5 步再执行 |
| **多步上下文** | ❌ messages 是 task 内部局部变量 | ✅ messages 在 step 之间共享 |

### 【口播 · 三处不变】

注意——**demo1 的三个核心都没动**：
1. **工具集不变**（execute_bash / read_file / write_file）
2. **单步执行不变**（每个 step 内部仍然是 ReAct：思考→行动→观察）
3. **路由表设计不变**（字典映射）

我们只增加**两个新能力**：**记忆** 和 **规划**。然后把 messages 从「task 内部局部变量」升级成「step 之间共享的上下文」。

新公式：

> ### demo2 = demo1 × 记忆 × 规划

接下来 5 章，逐层拆开。

## 第 2 章 记忆系统：信息搬运的本质

### 【口播 · 为什么需要记忆】

我们用 Claude Code、Cursor 的时候，会发现它能在**一个长会话里持续记住**用户之前说过的话、做过的操作。

但是 demo1 的 Agent——**任务一结束，messages 全清空**。下一次再启动，**金鱼记忆**。

为什么不直接用大模型自己的记忆？因为**大模型本质上没有真正的记忆**——它只是有一个**有限的上下文窗口**。窗口塞满就溢出。

所以记忆这件事，**本质上是一个信息搬运问题**：

### 【屏显 · 表格 2-1 — 记忆的本质】

| 角色 | 能力 | 限制 |
|---|---|---|
| **大模型** | 有一定上下文窗口、有一定推理能力 | 窗口有限，**没有真正的持久记忆** |
| **本地 Agent** | 可以读写外部存储（文件/数据库/向量库） | 搬运什么、搬多少，要自己决定 |

**记忆 = 把外部存储的信息，有选择地搬运进 prompt**。所有方案的区别，就是「**存在哪、怎么存、搬多少**」三个问题的不同答案。

### 【口播 · 一个比喻：内存 vs 外存】

可以这么理解：**大模型的上下文窗口 = 内存**（快但容量有限），**文件/数据库/向量库 = 外存**（容量大但需要主动加载）。

本地 Agent 有读写外存的能力。**会话过程中按需把外存里有用的东西搬运进内存（prompt），让大模型思考时能看到**——这就是记忆系统的全部工作。

### 2.1 存储方案：一个 Markdown 文件

### 【口播】

demo2 用**最原始**的方案——一个 `agent_memory.md` 文件。不引入数据库，不引入向量库。**作为演示，足够揭示原理。**

每次任务结束，我们追加三件事：**什么时间、做了什么任务、得到什么结果**。

### 【屏显 · 代码 2-1 — append_memory】

```python
MEMORY_FILE = "agent_memory.md"

def append_memory(task: str, result: str) -> None:
    timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    result_preview = result[:500]  # 限长，防止文件膨胀
    entry = (
        f"\n## [{timestamp}]\n"
        f"**任务**: {task}\n"
        f"**结果**: {result_preview}\n"
    )
    with open(MEMORY_FILE, "a", encoding="utf-8") as f:
        f.write(entry)
```

### 【屏显 · agent_memory.md 实际内容示例】

```markdown
## [2026-06-29 14:23:11]
**任务**: 统计当前目录下有多少个 Python 文件
**结果**: 已统计完成，当前目录下共有 12 个 Python 文件...

## [2026-06-29 14:25:42]
**任务**: 创建一个 hello world 的 py 文件
**结果**: 已创建 hello.py，内容为 print("Hello, World!")
```

### 【口播】

每条记忆就是一个**时间戳 + 任务 + 结果摘要**。任务结束后追加。这就是 demo1 在结尾提到「**金鱼记忆**」要解决的核心。

### 2.2 加载方案：滑动窗口（最后 50 行）

### 【口播】

存进去了，**怎么读出来**？

最朴素的方案：**整个文件全塞进 prompt**？——不行。一个 Agent 跑了几十次任务，文件会越长越大，最终把大模型的上下文窗口撑爆。

所以我们做一次简化：**只加载最后 50 行**。

### 【屏显 · 代码 2-2 — load_memory】

```python
MEMORY_WINDOW_LINES = 50  # 滑动窗口

def load_memory() -> str:
    if not os.path.exists(MEMORY_FILE):
        return ""
    with open(MEMORY_FILE, encoding="utf-8") as f:
        lines = f.readlines()
    # 滑动窗口：只取最后 N 行
    return "".join(lines[-MEMORY_WINDOW_LINES:])
```

### 【口播】

这就是**滑动窗口**——窗口大小固定 50 行，旧记忆会随新记忆累积被挤出窗口。

为什么是 50 行？——**没有任何科学依据**，这是个演示用的占位值。真实场景里，**50 行可能很短，也可能很长**，取决于单条记忆的大小和大模型的上下文窗口剩余空间。重要的是这个**「限制 + 截取」**的模式本身。

### 2.3 Progressive Context：怎么注入 prompt

### 【口播】

记忆加载出来之后，**塞到哪？**——塞到 System Prompt 的**后缀**里，作为「**Progressive Context**」（渐进式上下文）。

### 【屏显 · 代码 2-3 — build_system_prompt】

```python
def build_system_prompt() -> str:
    memory = load_memory()
    if not memory.strip():
        return SYSTEM_PROMPT_BASE
    return SYSTEM_PROMPT_BASE + "\n\n## 历史任务记忆（最近）\n\n" + memory
```

### 【屏显 · 图 2-1 — System Prompt 的两层结构】

```
┌─────────────────────────────────────────────┐
│  System Prompt                              │
│  ┌───────────────────────────────────────┐  │
│  │  基础 Prompt（不变）                    │  │
│  │  「你是一个有用的助手，可以调用工具...」  │  │
│  └───────────────────────────────────────┘  │
│  ┌───────────────────────────────────────┐  │
│  │  Progressive Context（动态）            │  │  ← demo2 新增
│  │  最近 50 行历史任务记忆                  │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

### 【口播 · 记忆流程图】

我们看完整的运行时序：

### 【屏显 · 图 2-2 — 记忆时序】

```
首次运行（agent_memory.md 不存在）:
    System (基础 prompt)
        ↓
    user: "统计当前目录有多少 .py 文件"
        ↓
    [ReAct 循环] → 结果 "12 个"
        ↓
    append_memory → 写入 agent_memory.md


第二次运行（agent_memory.md 已有内容）:
    System = 基础 prompt + Progressive Context(最后 50 行)
        ↓                                       ↑
        ↓                  ┌─── 看到第一次任务 + 结果
        ↓                  │
    user: "再创建一个 hello.py"  ← 大模型知道之前做了什么
        ↓
    [ReAct 循环] → 结果 "已创建"
        ↓
    append_memory → 追加到 agent_memory.md
```

### 【口播 · 关键观察】

注意一个细节：**System Prompt 整个 task 内不变**。我们在 `run_agent` 入口构建一次 system prompt，传给所有 step 共用。任务进行中不会重新读记忆文件——因为任务还没结束，记忆不会更新。

**只有任务结束 → append_memory 写入 → 下一个任务开始 → 才会读到新记忆**。这是个简单但重要的设计选择。

### 2.4 这套方案揭示的原理

### 【口播】

demo2 这套记忆方案**非常粗糙**，但它揭示了一个**普适原理**：

> **大模型有上下文窗口限制，本地必须把外部存储的信息有选择地搬运进 prompt。**

不管你用什么方案：

### 【屏显 · 表格 2-2 — 记忆方案的演进谱系】

| 方案 | 存在哪 | 怎么存 | 搬多少 |
|---|---|---|---|
| **demo2（演示）** | Markdown 文件 | 追加 | 最后 50 行 |
| **向量数据库** | Chroma / Pinecone | 嵌入向量 | 语义相关的 top-K |
| **记忆压缩** | Markdown / SQLite | 大模型压缩后存 | 关键事实摘要 |
| **Memory 工具** | 外部存储 | Agent 自主存取 | 按需检索 |
| **分层记忆** | 多个文件 | 系统/项目/用户级 | 按层级叠加 |

**底层都是同一个模式：存在哪、怎么存、搬多少**。第 6 章会展开讲这些演进方向。

## 第 3 章 规划系统：先想再做

### 3.1 为什么需要规划

### 【口播】

回到 demo1 的执行方式：**整个任务直接丢给大模型，让它一步一步摸索**。

简单任务没问题。但**面对复杂任务**，这种方式有个致命问题：

### 【屏显 · 表格 3-1 — 走一步看一步的三大风险】

| 风险 | 描述 |
|---|---|
| **迷失细节** | 任务太大，大模型一直往下走，陷入局部细节 |
| **走偏方向** | 完成了很多局部目标，但偏离了整体目标 |
| **迟迟不收敛** | 没有全局视角，可能死循环或达不到预期 |

### 【口播】

打个比方：**没有规划的 Agent，像是在迷宫里乱走**——可能走出去了，但更可能绕远路、走死胡同、甚至原地打转。

demo2 的解法是引入一个**可选的规划阶段**——让大模型**先有全局视角**，把任务拆成几个清晰步骤，再逐步执行。

### 3.2 Plan 阶段：让大模型拆任务

### 【口播】

规划本身也是**一次大模型调用**——只不过这次调用的不是工具，而是让它**返回一个结构化的步骤列表**。

### 【屏显 · 代码 3-1 — get_plan】

```python
PLAN_TOOL = {
    "name": "submit_plan",
    "description": "提交任务执行规划",
    "input_schema": {
        "type": "object",
        "properties": {
            "steps": {
                "type": "array",
                "items": {"type": "string"},
                "minItems": 1,
                "maxItems": 10,
            }
        },
        "required": ["steps"],
    },
}

PLAN_SYSTEM_PROMPT = """你是任务规划助手。把用户任务拆成 3-5 个有序步骤。
必须通过 submit_plan 工具返回规划，不要直接输出文本。"""

def get_plan(user_input: str) -> list:
    response = client.messages.create(
        model=MODEL,
        system=PLAN_SYSTEM_PROMPT,
        tools=[PLAN_TOOL],   # ← 只暴露规划工具，不暴露执行工具
        messages=[{"role": "user", "content": user_input}],
    )
    for block in response.content:
        if block.type == "tool_use" and block.name == "submit_plan":
            steps = block.input["steps"]
            if 1 <= len(steps) <= 10:
                return steps
    return [user_input]   # ← 防御性降级
```

### 【口播 · 三个设计点】

注意这里三个**关键设计**：

### 【屏显 · 表格 3-2 — 规划系统的三个设计点】

| 设计点 | 代码 | 为什么这么做 |
|---|---|---|
| **专用 System Prompt** | `PLAN_SYSTEM_PROMPT` | 规划阶段的角色是「**规划助手**」，不是「执行助手」——和大模型对话要先给定角色 |
| **结构化返回** | Function Calling `submit_plan` | 强制 json 返回，避免解析自由文本的麻烦 |
| **防御性降级** | `return [user_input]` | 大模型不可靠，任何异常都退回 demo1 单步模式 |

### 【口播 · 重点强调】

第三个设计点**非常重要**。大模型规划可能失败、可能不调工具、可能格式不对——**任何异常我们都降级到 demo1 的单步模式**。

```python
try:
    ...
except Exception as e:
    return [user_input]   # 规划失败 → 单步执行
```

这是**防御性编程**——大模型调用是**不可靠的外部依赖**，必须有兜底。**Agent 不能因为规划阶段失败就整个崩掉**。

### 3.3 两种执行范式对比

### 【口播】

现在我们有两种执行方式，可以按任务复杂度选择：

### 【屏显 · 表格 3-3 — 直接 ReAct vs Plan 模式】

| 维度 | 直接 ReAct（demo1） | Plan 模式（demo2） |
|---|---|---|
| **流程** | task → ReAct 循环 | task → **规划** → step1 ReAct → step2 ReAct → ... |
| **大模型视角** | 局部，走一步看一步 | 先有全局，再逐步执行 |
| **优势** | 简单灵活 | 不容易迷失，有全局方向 |
| **劣势** | 复杂任务容易走偏 | 规划可能不准，多一次 LLM 调用 |
| **适用** | 简单任务 | 复杂任务 |

### 【口播】

**Plan 模式不是万能的**——大模型拆出来的步骤**不一定准确**。但**有全局视角，总比走一步看一步强**。

demo2 的 REPL 里，用户可以**自由切换**两种模式：

```bash
python agent.py            # 默认直接 ReAct
python agent.py --plan     # 启动时全局开启 Plan 模式
```

REPL 内还能随时切换：
```
/plan      # 切换 Plan 模式
/no-plan   # 切回直接 ReAct
/memory    # 查看当前记忆文件内容
```

## 第 4 章 多步执行：Message 的演化

### 【口播】

到这一步，规划已经拆出了 3 个 step。**怎么把它们串起来？**

这是 demo2 跟 demo1 最大的**结构性区别**——**messages 从「task 内部局部变量」变成「step 之间共享的上下文」**。

### 4.1 demo1 的结构：每个 task 一个独立 messages

### 【屏显 · 代码 4-1 — demo1 的 run_agent（回顾）】

```python
def run_agent(user_input: str) -> str:
    messages = [{"role": "user", "content": user_input}]   # ← task 内局部变量

    for loop_idx in range(1, MAX_ITERATIONS + 1):
        response = client.messages.create(...)
        ...

    return final_text   # ← 只返回文本，messages 丢弃
```

### 【口播】

demo1 里，`messages` 是 `run_agent` 的**局部变量**。一个 task 调用完，messages **整个丢掉**。

这没问题——因为 demo1 一个 task 就是一个完整的 ReAct 循环。

### 4.2 demo2 的结构：messages 在 step 间共享

### 【口播】

但 demo2 不一样了。一个 task 被拆成 3 个 step。**如果每个 step 还各自维护 messages**：

- step1 的工具调用结果，**step2 看不到**
- step2 想用 step1 找到的数据，**得重新调用工具**——浪费 token、浪费时间

正确做法是：**step1 的所有轨迹留在 messages 里，step2 接力使用**。

### 【屏显 · 代码 4-2 — run_agent_step（demo2 新结构）】

```python
def run_agent_step(
    step: str,
    messages: list,         # ← 外部传入
    system_prompt: str,
) -> tuple:                 # ← 返回 (result, messages)
    messages = messages + [{"role": "user", "content": step}]

    for loop_idx in range(1, MAX_ITERATIONS + 1):
        response = client.messages.create(
            system=system_prompt,
            tools=TOOLS,
            messages=messages,
        )
        if response.stop_reason != "tool_use":
            result = extract_text(response.content)
            return result, messages   # ← 把累积的 messages 带出去

        messages.append({"role": "assistant", "content": response.content})
        ...
        messages.append({"role": "user", "content": tool_results})
```

### 【口播 · 三个关键变化】

对比 demo1，三个变化：

### 【屏显 · 表格 4-1 — run_agent 的三个结构变化】

| 变化 | demo1 | demo2 |
|---|---|---|
| **messages 来源** | 函数内部新建 | 外部传入 |
| **messages 去向** | 函数内丢弃 | 通过返回值传出 |
| **返回值** | `str`（只返回文本） | `tuple[str, list]`（文本 + messages） |

### 4.3 编排层：把 step 串起来

### 【口播】

有了 `run_agent_step`，编排就简单了——一个循环：

### 【屏显 · 代码 4-3 — run_agent 编排】

```python
def run_agent(user_input: str, use_plan: bool = False) -> str:
    system_prompt = build_system_prompt()   # ← 整个 task 内不变

    steps = get_plan(user_input) if use_plan else [user_input]

    messages = []                            # ← task 级共享上下文
    final_result = ""
    for step in steps:
        final_result, messages = run_agent_step(
            step=step,
            messages=messages,               # ← 接力
            system_prompt=system_prompt,
        )

    append_memory(user_input, final_result)  # ← task 结束写入记忆
    return final_result
```

### 【口播 · 三层记忆区分】

到这里 demo2 出现了**三层记忆**，请仔细区分：

### 【屏显 · 表格 4-2 — demo2 的三层记忆】

| 层级 | 范围 | 载体 | 生命周期 |
|---|---|---|---|
| **step 内消息** | 单次 step | `messages` 列表 | step 结束仍然保留 |
| **task 级共享上下文** | 多个 step 之间 | 同一个 `messages` 列表 | task 结束丢弃 |
| **跨 task 长期记忆** | 跨 task、跨会话 | `agent_memory.md` 文件 | 持久（滑动窗口 50 行） |

注意第二层和第三层**不一样**：
- **task 级共享上下文** = step1、step2、step3 共享的 `messages`，task 结束就丢
- **跨 task 长期记忆** = 所有 task 结束后写入 `agent_memory.md`，下次启动能读

这两层解决的是**不同问题**，不要混淆。

## 第 5 章 示例解读：找所有 todo 整理到 todo.md

### 【口播】

讲到这里，用一个完整例子把 demo2 跑起来。

任务：**「找到代码里所有的 todo，整理到 todo.md 文件中」**，用 `python agent.py --plan` 启动。这是个**真实跑通**的例子——下面所有输出都是实测截取的，不是虚构。

### 【屏显 · 图 5-1 — 整个任务的 3 个阶段】

```
阶段 0: 加载记忆 + 规划
   agent_memory.md → Progressive Context
   get_plan → 拆出 3 个 step

阶段 1: 多步串联执行（messages 在 step 间共享）
   Step 1: 搜索 todo         → 多轮 ReAct（含错误修正）
   Step 2: 整理去重           → 不调工具，从历史上下文整理
   Step 3: 写入 todo.md      → 调用工具

阶段 2: 写入长期记忆
   整个 task 结束 → append_memory → agent_memory.md
```

### 5.1 阶段 0：加载记忆 + 规划

### 【口播】

任务进来先做两件事：

1. **加载记忆** —— 读 `agent_memory.md` 最后 50 行，拼到 System Prompt 后缀（Progressive Context）。首次运行则为空。
2. **规划** —— `get_plan` 用专用 system prompt 调一次大模型，让它通过 `submit_plan` 工具返回步骤列表。

### 【屏显 · 实测输出】

```
[记忆] 已加载 2 条历史任务（8 行）作为 Progressive Context
   ## [2026-06-29 22:13:05]
   ## [2026-06-29 22:14:12]

[规划] 调用大模型拆解任务...
[规划] 拆解出 3 个步骤:
   Step 1: 在项目目录下递归搜索所有源码中的 TODO/FIXME/HACK/XXX 标记
   Step 2: 把搜索结果去重整理，按文件+行号+优先级归纳
   Step 3: 把清单写入 todo.md 文件
```

注意规划阶段用的是**专用 system prompt**——角色是「规划助手」，只暴露 `submit_plan` 工具，不暴露执行工具。规划失败自动降级为单步执行。

### 5.2 阶段 1：多步串联执行

### 【口播】

规划完成后，进入串联执行。整个 plan 共享一个 `messages` 列表——step1 的所有工具调用结果，step2 / step3 都能看到。

#### Step 1：搜索 todo（含错误修正，多轮 ReAct）

这一步最有意思——大模型默认按 **Linux 习惯**调用工具，结果在 **Windows 环境**接连碰壁：

```
Step 1 ReAct 循环（实测）:
  思考: 用 grep 递归搜索
  action: execute_bash("grep -rn 'TODO' *.py")
  observation: ❌ "系统找不到指定的路径"（grep 在 Windows 不存在）

  思考: 切到 findstr
  action: execute_bash("findstr /S /N 'TODO' *.py")
  observation: ❌ 仍然报错（findstr 通配符 + 路径问题）

  思考: 用 PowerShell
  action: execute_bash("powershell -c \"Select-String -Pattern 'TODO' -Path *.py\"")
  observation: ✅ 命中 4 条
     agent.py:157:  TODO: 加白名单 / 审批约束
     agent.py:239:  TODO: 文件超过阈值时压缩成摘要
     agent.py:348:  TODO: 拆完后让用户逐 step 确认
     agent.py:458:  TODO: 多 step 独立步骤支持并发

  → end_turn（step 完成）
```

**这是 ReAct 最有价值的时刻**——大模型**通过观察工具返回的错误自我修正**：grep → findstr → PowerShell，三次试错才找到能用的工具。demo1 的 ReAct 循环本身就具备这个能力，demo2 完整保留。

Step 1 结束时，messages 已经累积了 **十多条**（多次失败的工具调用全部留在上下文里）。**这堆失败的尝试原封不动传给 Step 2**——这正是共享上下文的价值。

#### Step 2：整理去重（关键 — 不调工具）

```
Step 2 ReAct 循环:
   user(整理去重) → LLM 看到历史 messages（含 Step 1 的全部搜索结果）
                 → 直接 end_turn，返回带优先级标注的清单
```

**Step 2 没有调用任何工具**——因为 Step 1 找到的 4 个 TODO **已经在共享的 messages 里**。大模型直接从历史上下文整理，并自动按风险加了优先级标注（🔴 高 / 🟡 中 / 🟢 低）。

这就是 demo2 多步共享 messages 的核心价值：**信息在 step 间累积，避免重复劳动**。省 token、省时间、保证一致性。**如果 step2 重新搜索一遍，等于把 Step 1 走过的弯路再走一次**。

#### Step 3：写入 todo.md

```
Step 3 ReAct 循环:
   user(写入 todo.md) → LLM 决策 → tool_use(write_file, path=todo.md, content=完整清单)
                      → tool_result(成功)
                      → end_turn（step 完成）
```

Step 3 也只用 1 次工具调用——大模型从历史上下文已经知道写什么、怎么组织。

最终生成的 `todo.md` 实测 **201 行 / 7786 字节**，包含 6 个章节：

### 【屏显 · 实际生成的 todo.md 结构】

```markdown
# TODO 清单
## 一、搜索总览        ← 标记统计表（TODO×4 / FIXME×0 / HACK×0 / XXX×0）
## 二、快速索引        ← 4 条目的一览表（文件/行号/优先级/简述）
## 三、逐条详情        ← 每条 TODO 展开成卡片（代码上下文 + 建议）
## 四、处理进度 Checklist  ← 4 个 - [ ] 复选框
## 五、按优先级汇总     ← 🔴1 / 🟡2 / 🟢1
## 六、文档排除说明     ← 主动说明 README/讲稿里的虚构示例已剔除
```

**第六节是亮点**——大模型发现 `README.md` 和 `讲稿.md` 里也出现了 `agent.py:120: TODO: add error handling` 这种**虚构示例**，会污染搜索结果，于是**主动排除并解释**。这是共享上下文让大模型获得「全局视角」的直接证据。

### 5.3 阶段 2：写入长期记忆

### 【口播】

3 个 step 全部跑完，**整个 task 才算结束**。这时候把**原始任务 + 最终结果摘要**写入 `agent_memory.md`：

### 【屏显 · 写入的记忆】

```markdown
## [2026-07-01 10:23:48]
**任务**: 找到代码里所有的 todo，整理到 todo.md 文件中
**结果**: 已找到 4 个真实 TODO（agent.py: L157/L239/L348/L458），
        整理成 201 行的 todo.md（含 6 个章节、优先级标注、文档排除说明）...
```

注意这里写入的是 **task 级**——**不是 step 级**。step 之间共享的 messages 此时**整体丢弃**，但记忆已落盘。下次启动新任务时，这条记忆会作为 Progressive Context 自动加载。

### 5.4 messages 演化回顾

### 【屏显 · 图 5-2 — messages 在多步间累积（实测数字）】

```
任务开始:     messages = []
                ↓
[规划]        拆出 3 个 step（规划用独立调用，不污染主 messages）
                ↓
Step 1:       messages 累积 ~30 条（3 次试错 + 1 次成功 + 多轮思考）
                ↓
Step 2:       messages 累积 ~33 条（不调工具，仅 user + assistant text）
                ↓              ← 关键：Step 2 直接读 Step 1 留下的搜索结果
Step 3:       messages 累积 ~39 条（user + tool_use + tool_result + end_turn）
                ↓
[task 结束]   append_memory → agent_memory.md
                ↓
            messages 整体丢弃（记忆已落盘）
```

### 【口播 · 设计精髓】

整个过程中 messages 从 0 条**累积到 39 条**。**Step 1 走过的弯路（grep / findstr 失败）全部留在上下文里**——Step 2 / Step 3 因此能直接复用最终搜索结果，不用重走一遍。

如果换作 demo1 的「每个 task 一个独立 messages」结构，39 条历史**完全没法保留**——每个 task 都是孤儿。这就是 demo2 的核心增量：

> **有了规划，有了多步共享上下文，Agent 不再是「每步从零开始」。**

另外，Step 1 那段「grep 失败 → findstr 失败 → PowerShell 成功」的试错，**是 ReAct 范式最诚实的演示**——Agent 不是一开始就全对的，但它能在循环里通过观察工具返回值自我修正。这比一个一次成功的虚构例子有教学价值得多。

## 第 6 章 局限与演进方向

### 【口播】

demo2 能干活了，也学会了记忆和规划。但它的实现**很粗糙**——我们看两块局限，以及业界是怎么演进的。

### 6.1 记忆局限：滑动窗口太粗糙

### 【口播】

我们用「最后 50 行」做了简化，但这个数字**没有科学依据**：
- **太少**——丢掉了很多早期重要信息
- **太多**——如果单条记忆很长，50 行就把上下文塞满
- **不精准**——无差别加载，无法按当前任务的相关性筛选

业界有 4 个主流演进方向：

### 【屏显 · 表格 6-1 — 记忆方案演进】

| 方向 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| **向量数据库 + 语义检索** | 把历史记忆向量化存入 Chroma / Pinecone，按当前任务取相似度 top-K | 只加载相关的，省 token | 不"语义相似"但重要的细节可能被漏掉 |
| **记忆压缩 / 蒸馏** | 接近窗口上限时调大模型压缩，提取关键事实、去细节（如 100 行压缩成 5 行） | 自适应窗口，能保留尽可能多的记忆 | 多一次大模型调用；每次压缩都会丢一些细节 |
| **Memory 工具** | 给 Agent 一个 `search_memory` 工具，让它按需检索外存 | 按需取用，平时不占 system prompt | Agent 要学会何时该用；本质跟向量库类似 |
| **分层记忆**（Claude Code 风格） | 系统 / 项目 / 用户多层级，按当前场景只加载对应层 | 大幅减少每次加载量；不同粒度分离 | 多文件管理复杂 |

### 【口播】

注意这 4 个方向**不是互斥**的——成熟框架往往组合使用。比如 Claude Code 就是「分层记忆 + 按需压缩」组合。

但不管用什么方案，底层都是同一个模式：**存在哪、怎么存、搬多少**。demo2 用最简单的「Markdown 文件 + 最后 50 行」揭示了原理，剩下都是工程优化。

### 6.2 规划局限：写死模式 + 步骤不可控

### 【口播】

规划这块有两个更明显的局限：

### 【屏显 · 表格 6-2 — 规划的两个局限】

| # | 局限 | 当前 demo2 | 理想状态 |
|---|---|---|---|
| 1 | **模式写死** | 用户手动 `--plan` 或 `/plan` 切换 | Agent 自动判断任务复杂度，简单直接 ReAct，复杂自动走 Plan |
| 2 | **步骤不可控** | 大模型拆出来的 step 直接执行，对不对全凭运气 | 用户参与打磨、相似任务复用历史 Plan |

### 【口播 · 局限 1：模式写死】

现在 Plan 模式是**用户手动开启**——这很**单板**。理想情况下，Agent 应该**自己判断**：简单任务直接 ReAct，复杂任务自动走 Plan。

业界一些成熟框架已经做到这一点：**自动判断任务是否需要规划**，需要的话切到 Plan 模式拆 step，让用户确认后再执行。

### 【口播 · 局限 2：步骤不可控】

`get_plan` 让大模型拆任务，但**拆出来的不一定准**——步骤可能重叠、可能遗漏、甚至可能根本不可执行。

三个演进方向：

### 【屏显 · 表格 6-3 — 规划质量演进方向】

| 方向 | 做法 | 价值 |
|---|---|---|
| **多轮交互打磨** | 拆完后每个 step 让用户确认，需要改进就再调大模型调整 | 不完全依赖大模型一次性规划，通过人机交互收敛 |
| **失败重规划** | 某个 step 执行失败后，重新规划剩余部分 | 把 Plan 当成动态过程，不是一次性产物 |
| **技能化（Skills）** | 把常见任务的规划**预先固化**——下次遇到相似任务直接套用历史 Plan | 不用每次都让大模型重新拆；相似任务复用，效率稳定 |

**技能化（Skills）是核心演进方向**——把"反复出现的规划"沉淀下来，下次遇到相似任务**直接调用已经规划好的 step**，跳过规划阶段。这一方向本系列 demo 不展开实现，留给读者探索；demo3 会从另一个角度解决"规划要手动开"——让 LLM 自己决定要不要拆步骤。

### 【口播 · 收尾】

这两块局限——**记忆太粗、规划太死**——就是 demo2 留下的两个遗憾。下一节我们会看到，业界成熟框架（Claude Code、Cursor）都已经用工程化方式解决了这些 problems。

## 第 7 章 下一节预告：MCP / Rules / plan 自动决策

### 【口播】

到 demo2 为止，我们的 Agent 已经有了：

### 【屏显 · 表格 7-1 — demo 进度回顾】

| 能力 | demo1 | demo2 |
|---|---|---|
| LLM × 工具 × 循环 | ✅ | ✅ |
| 跨任务长期记忆 | ❌ | ✅ |
| 规划 + 多步串联 | ❌ | ✅ |

### 【口播】

但还有**三个问题没解决**——demo3 会一次性补上：

### 【屏显 · 表格 7-2 — 三个未解决问题 → demo3 的三件新事】

| # | 未解决问题 | demo3 的解法 |
|---|---|---|
| 1 | **工具硬编码**——只有 execute_bash / read_file / write_file，想接外部能力怎么办？ | **MCP**（Model Context Protocol）：把工具搬到独立 HTTP 服务，JSON-RPC 协议跨进程调用 |
| 2 | **无行为约束**——想执行什么就执行什么（`rm -rf` 都行）；写代码也不约束风格 | **Rules**：`.agent/rules.md` 拼进 system prompt，约束命名/函数行数等"软规范" |
| 3 | **规划靠手动触发**——用户要记得敲 `--plan` / `/plan`，Agent 不知道什么时候该规划 | **plan 自动决策**：把 plan 做成工具，让 LLM 在 `run_agent` 顶层 1 轮决策自己分叉 |

### 【口播 · 收尾金句】

demo1 解决了「**能不能干活**」，demo2 解决了「**记得过去、规划未来**」。

下一节 demo3，我们要让 Agent **接入更广的能力、守规矩、还学会自己判断要不要规划**——一步步向 Claude Code、Cursor 这样的成熟框架演进。

> ## demo2 = demo1 × 记忆 × 规划
> ## demo3 = demo2 × Rules × MCP（plan 自动决策）

下一节见。
