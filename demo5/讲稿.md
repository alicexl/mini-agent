## 第 1 章 整体把握：从临时工到正式员工

### 【口播 · 回顾 demo4】

> ### demo4 = demo3 × Subagent（分工合作）

demo4 我们做了**subagent**——可以临时派去多个专家来干活。subagent 的本质是**一个一次性的临时工**：生存 → 干活 → 返回摘要 → 消亡。

很多场景下这套够用了。但**现实生活中**你会发现——

我们建立一个团队之后、交给你一个任务之后，**不会立马解散**。我会雇佣你、给你一个角色，整个项目做完的时候，**这个项目团队才会解散**。

为什么？因为不同的角色之间**必须交流**：

- 同一个成员在一个项目里**会被多次唤起**
- 互相之间有**依赖**——A2 要用 A1 的结果
- 我需要知道：在这个项目里我之前做了什么、别人给了我什么、我要继续做什么

### 【口播 · demo4 subagent 做不到这些】

demo4 的 subagent **做不到**：

- 每次调用都是**全新**的、**没有记忆**的临时工
- 互相之间**不能通信**，没办法告诉对方"我需要什么" / "我做了什么"

**这就是 demo5 要解决的事情——把临时工升级成正式员工（team）**。

### 【屏显 · 表格 1-1 — 临时工 vs 正式员工】

| 维度 | demo4 subagent（临时工） | demo5 Agent（正式员工） |
|---|---|---|
| **身份** | 临时拼一个 role | **有 name + 固定 role** |
| **记忆** | 用完即丢 | **跨多轮 chat 累积 memory** |
| **通信** | 完全隔离 | **有 inbox + 收发消息机制** |
| **生命周期** | 完成任务立即消亡 | **持续到整个项目解散** |
| **类比** | 一次性外包 | **项目组的正式成员** |

### 【口播 · 一减一加】

demo5 在 demo4 基础上做**一减一加**：

- **减法**：
  - **去掉 demo4 的 subagent 工具**——临时工模式演不下去了
  - MCP / plan 沿用 demo4 的状态：依然没有
- **加法**：
  - **新增 `Agent` 类**——身份 + 记忆 + inbox + chat 方法
  - **新增 `Team` 类**——招募 / 通信（一对一 + 群发）/ 解散 + `run_team` 协作流程
  - **新增 任务状态机 + 事件驱动调度**——每个任务有 `pending/reviewing/redoing/passed/failed` 五态；质检员持续监听，任务一完成立即质检；单任务最多 3 次质检，3 次不过即失败；依赖了失败任务的任务自动级联失败
  - **保留 demo3 的 Rules 加载器**——`load_rules()` 在启动时检查 `.agent/rules.md`，Team 级持有并注入到项目经理 prompt（Agent 不加载，保持独立）

## 第 2 章 Agent 类：身份、记忆、inbox、chat

### 2.1 本质：从函数升级为持久化对象

### 【口播】

demo4 的 subagent 是一个**一次性函数** `_run_subagent(role, task)`——拼一个临时 prompt、新建一份 messages、跑完返回、messages 立刻丢弃。

demo5 把它**升级为持久化对象**——`Agent` 类的实例。一个 Agent 只要对象活着，就可以**一直和它通信**。

### 【屏显 · 图 2-1 — Agent 类的属性】

```python
class Agent:
    def __init__(self, name, role, tools, local_fns, verbose=True):
        self.name        = name        # ← 固定身份（demo4 只有临时 role）
        self.role        = role        # ← 固定角色
        self.tools       = tools       # ← 可用工具列表
        self.local_fns   = local_fns   # ← 本地函数字典

        self.inbox: list[tuple[str, str]] = []   # ← 收件箱（其他 agent 发来的消息）
        self.messages: list[dict]       = []     # ← 长期记忆（跨多次 chat 累积）
        self.system_prompt = build_agent_system_prompt(name, role)
```

与 demo4 的对比——**消息从函数的局部变量变成了实例的属性**：

| 维度 | demo4 `_run_subagent` | demo5 `Agent` |
|---|---|---|
| messages | 函数局部变量，函数返回即丢 | **实例属性 self.messages，对象活着就在** |
| system_prompt | 函数内临时拼 | **实例属性 self.system_prompt，固定身份** |
| inbox | 不存在 | **self.inbox 接收其他 agent 的消息** |
| 跨多轮对话 | 不支持 | **支持——多次 chat() 共享 self.messages** |

### 2.2 两个核心方法：chat / receive

### 【屏显 · 代码 2-2 — Agent 的两个核心方法】

```python
def receive(self, sender: str, message: str) -> None:
    """被其他 Agent 调用：往 inbox 里塞一条消息"""
    self.inbox.append((sender, message))


def chat(self, task=None) -> str:
    """
    一次 chat = 消化 inbox → 执行任务 → 走 ReAct 循环
    """
    # 1) 消化 inbox：把所有未读消息包装成 user message 灌进 messages
    if self.inbox:
        for sender, msg in self.inbox:
            wrapped = f"[来自 {sender} 的消息] {msg}"
            self.messages.append({"role": "user", "content": wrapped})
        self.inbox.clear()

    # 2) 追加本次任务
    if task:
        self.messages.append({"role": "user", "content": task})

    # 3) 走 ReAct 循环（与 demo4 的 _react_loop 同构）
    return self._react_loop()
```

### 【口播 · chat 的三段式】

`chat()` 是 demo5 的核心方法——**一次 chat 就是三段**：

1. **先看 inbox**：如果有人给我发消息，必须**先消化**——包装成 user message 灌进我的 messages
2. **执行任务**：把本次 task 也追加到 messages
3. **走 ReAct**：调大模型 → 工具调用 → 最终回复

> 关键洞察：inbox 清空了，但 messages **永远累积**——下次再 chat，前面所有对话都还在。

### 【口播 · receive 是通信入口】

`receive()` 是被**其他 Agent**（或 Team 编排器）调用的入口——别人调你的 receive，就给你的 inbox 加一条消息。**inbox 是临时缓存**（chat 时清空），messages 是长期记忆（项目结束才销毁）。

### 2.3 system_prompt：身份 + 专注

### 【屏显 · 代码 2-3 — build_agent_system_prompt】

```python
def build_agent_system_prompt(name: str, role: str) -> str:
    return (
        f"你是团队成员「{name}」，你的角色是：**{role}**。\n"
        f"请专注完成交给你的任务。如果收到其他成员的消息，把它当作上下文输入。\n"
        f"做完后用一两句话汇报结果。"
    )
```

## 第 3 章 Team 类：招募、通信、解散、协作

### 3.1 四个核心动作

### 【口播】

demo5 把协调工作从"LLM 调 subagent 工具"改成**Team 类外部编排**。Team 提供**四个核心动作**。

### 【屏显 · 表格 3-1 — Team 的四个核心动作】

| 动作 | 方法签名 | 类比 |
|---|---|---|
| **招募** | `recruit(name, role)` | 招人入项目 |
| **一对一通信** | `send(sender, receiver, message)` | @某人发私信（依赖注入 / 质检反馈打回） |
| **群发** | `broadcast(sender, message)` | 群里通报全员（成员完成任务后通报） |
| **解散** | `dismiss()` | 项目结束，团队解散 |

### 【屏显 · 代码 3-2 — 四个动作的实现】

```python
def recruit(self, name, role):
    """招募一个新成员"""
    agent = Agent(name=name, role=role, tools=self.tools, ...)
    self.agents[name] = agent

def send(self, sender, receiver, message):
    """一对一通信：sender → receiver（依赖注入 / 质检反馈打回）"""
    self.agents[receiver].receive(sender, message)

def broadcast(self, sender, message):
    """群发：sender → 所有其他成员（成员完成任务后通报全员）"""
    for name, agent in self.agents.items():
        if name == sender:
            continue
        agent.receive(sender, message)

def dismiss(self):
    """项目结束，解散团队——所有成员销毁"""
    self.agents.clear()
```

### 【口播 · 为什么 send / broadcast 是 Team 的方法而不是 Agent 的】

注意——`send` / `broadcast` 是 Team 的方法，**不是 Agent 的**。为什么？

因为**通信需要路由**：发消息要找到目标 agent、群发要遍历成员清单——这些事由 Team 这个"项目组容器"来做最自然。Agent 自己只管 `receive`——**别人怎么找到我、怎么路由，是 Team 的事**。

`broadcast` 还承担一个关键角色——**成员完成任务后通报全员**。编排器在每个任务 chat 完之后调 `broadcast(assignee, 结果)`，让其他成员（特别是质检员）的 inbox 收到完成事件。这就是下一节"质检员一直在监听"的基础。

### 3.2 完整协作流程：run_team

### 【口播】

四个动作是"原子操作"。把它们串起来的完整流程入口是 `run_team(user_input)`——**这是 demo5 真正的演示入口**。

### 【屏显 · 图 3-3 — run_team 的事件驱动流程】

```
run_team(user_input)
   │
   ├─ Step 1: 项目经理拆任务（强制含 1 名质检员 + 注入 team 级 rules）
   │    · 输出 JSON：members + tasks（标注 depends_on）
   │    · _find_reviewer 校验 members 里有 role 为「质检员」的成员
   │
   ├─ Step 2: recruit 每个成员
   │
   ├─ Step 3: 事件循环 _event_loop（核心）
   │    每轮按优先级处理一个事件：
   │      ① reviewing → 质检员立即介入（质检员一直在监听）
   │      ② redoing   → 立即重做（assignee 收到反馈）
   │      ③ pending 且依赖全 passed → 派发（chat + broadcast）
   │      ④ pending 且依赖 failed → 标记 failed（级联失败）
   │
   ├─ Step 4: 所有任务进入终止态 → 统计项目状态
   │    · ✅ 通过 N 个 / ❌ 失败 M 个
   │
   └─ Step 5: dismiss 解散团队
```

### 【口播 · 任务状态机——demo5 的核心抽象】

每个任务在生命周期里处于 5 个状态之一：

```
pending ──依赖全 passed──> reviewing ──质检通过──> passed (终态)
                              │
                              ├──质检不过 + attempts<3──> redoing ──> (回 reviewing)
                              │
                              └──质检不过 + attempts=3──> failed (终态)

pending ──依赖 failed──> failed (级联，终态)
```

```python
TASK_PENDING   = "pending"    # 等待执行（依赖未满足）
TASK_REVIEWING = "reviewing"  # 执行完，待质检员验收
TASK_REDOING   = "redoing"    # 质检未过，待重做
TASK_PASSED    = "passed"     # 质检通过（终态）
TASK_FAILED    = "failed"     # 3 次质检不过，或依赖失败（终态）

MAX_REVIEW_ATTEMPTS = 3       # 单任务最多质检 3 次
```

### 【口播 · 为什么用事件循环而不是顺序派发】

最直觉的 Team 实现是"按 tasks 列表顺序派发，全跑完再统一 review"——但这不对：

- 真实场景里，**质检员应该一直在监听**——任务一完成就立即质检，不等其他任务
- 任务之间**互相依赖**——B 用 A 的结果，A 没通过质检就启动 B 是浪费
- 单任务可能**多次重做**——不能阻塞整个流水线

事件循环让质检员始终处于"待命 → 立即响应"的状态，新任务在依赖 passed 后才启动，级联失败传播自动——这是真正接近真实项目组的协作模式。

> 关键：Agent.messages 不是线程安全的，所以 demo5 用**单线程事件循环**模拟"质检员一直在监听"的语义，不引入真并发的锁复杂度。

### 【口播 · 为什么不让 LLM 自己决定何时组建 team】

注意——我们**也可以**做一个 `team` 工具，让 LLM 自己决定是否组建 team。但 demo5 **为了演示简化**，直接走 `run_team()` 入口——**用户输入直接进 team 流程**。

这不是必须的，只是为了把"协作机制"这一件事讲透。真实场景里，是否走 team 可以由 LLM 决策。

### 3.3 项目经理 prompt：拆解成结构化 JSON

### 【屏显 · 代码 3-4 — _plan_team 的核心 prompt】

```python
planning_prompt = (
    "你是一个项目规划师。请把用户的任务拆解成多个角色（成员）和子任务序列，输出严格 JSON。\n\n"
    "规则：\n"
    "1) 每个成员有简短的 name（如 A1、A2、B1）和 role（角色）。\n"
    "   role 只能取两种值：\n"
    "     · 「执行者」——承担具体任务（A1、A2、B1...）\n"
    "     · 「质检员」——承担验收任务（只设 1 名，name 建议 Q1）\n"
    "2) 每个任务标注 assignee（执行者 name）和 task（具体要做的事）\n"
    "3) 如果某任务依赖前序任务的结果，在 depends_on 数组里列出前序 assignee 的 name\n"
    "4) 任务应覆盖用户问题所有需要完成的部分\n"
    "5) **必须**分配且仅分配 1 名质检员——在 members 里加一个 name（如 Q1）、"
    "role 写「质检员」。质检员**不接 task**（不在 tasks 的 assignee 里出现），"
    "它由编排器在事件循环里对每个完成的任务做验收。\n\n"
    "输出格式（严格 JSON）：\n"
    "{\n"
    '  "members": [{"name": "A1", "role": "执行者"}, {"name": "A2", "role": "执行者"}, '
    '{"name": "Q1", "role": "质检员"}],\n'
    '  "tasks": [\n'
    '    {"assignee": "A1", "task": "...", "depends_on": []},\n'
    '    {"assignee": "A2", "task": "...", "depends_on": ["A1"]}\n'
    '  ]\n'
    "}\n"
)

# team 级 rules 注入——让项目经理拆任务时守规矩
if self.rules:
    planning_prompt += f"\n---\n以下是项目级行为规范：\n{self.rules}"
```

### 【口播 · 强制质检员 + 单任务级质检】

两个关键改动相对早期版本：

1. **角色只分两类**——role 取值只能是「执行者」或「质检员」。不再让项目经理发明"加法计算专家/天气查询助手"等花式角色名——每个执行者的行为完全由它接到的 task 文本决定，role 字段仅起标识作用
2. **质检员是必选**，不再可选——plan 必须含 1 名 role 为「质检员」的成员。编排器用 `_find_reviewer` 扫 members 找出来，找不到直接抛错
3. **质检员不接 task**——它由编排器在事件循环里被"完成事件"自动调起，对每个任务做单任务级质检（不是项目级总闸门）
4. **去掉 task 的 review 字段**——质检由编排器统一调度，不让项目经理在 plan 里给单个 task 打标记

### 【口播 · 依赖通过 depends_on 显式声明】

注意第 3 条——**依赖通过 depends_on 显式声明**。编排器看到 `depends_on: ["A1"]`，会在 A1 进入 `passed` 终态后，把 A1 的结果通过 `team.send("A1", "A2", "A1 的结果：xxx")` 注入到 A2 的 inbox，然后才启动 A2。**A1 没通过质检，A2 永远不会启动**——这就是状态机的好处。

### 3.4 质检员：事件驱动的单任务级质检（demo5 进阶能力）

### 【口播 · 质检员一直在监听】

demo5 的质检不是"所有任务跑完后统一验收"，而是**事件驱动**——质检员持续监听 inbox，任何任务一进入 `reviewing` 状态（即 assignee chat 完成并 broadcast 了结果），下一轮事件循环就立即调起质检员对这个任务做质检。

```
assignee.chat() 完成
   │
   ▼
broadcast(assignee, 结果) ──> 质检员 inbox 收到完成事件
   │
   ▼
任务状态：reviewing
   │
   ▼
下一轮事件循环（优先级 ①）
   │
   ▼
质检员.chat(验收 prompt) ──> 走 ReAct，可调 read_file / execute_bash 复查
   │
   ▼
输出 JSON: {"pass": true|false, "feedback": "..."}
   │
   ├─ pass=true             → 任务状态：passed（终态）
   │
   └─ pass=false
       │
       ├─ attempts < 3      → send(质检→assignee, feedback)
       │                       任务状态：redoing → 重做 → 回到 reviewing
       │
       └─ attempts = 3      → 任务状态：failed（终态）
```

### 【屏显 · _review_one_task 的核心逻辑】

```python
def _review_one_task(self, task, reviewer):
    task.review_attempts += 1
    # 质检员接收完成通知（receive → 下次 chat 消化）
    self.agents[reviewer].receive(
        task.assignee,
        f"[待验收任务（第 {task.review_attempts} 次）]\n"
        f"任务：{task.task_text}\n执行结果：{task.result}"
    )
    # reviewer 走 ReAct——可以用 read_file / execute_bash 实际复查
    verdict_text = self.agents[reviewer].chat(
        "请严格验收上条任务。可以调用 read_file / execute_bash 实际复查。\n"
        "严格只输出 JSON（不要 markdown 代码块、不要任何解释）：\n"
        '{"pass": true|false, "feedback": "若不通过，说明具体怎么改"}'
    )
    # 直接解析——prompt 已要求纯 JSON
    try:
        v = json.loads(verdict_text.strip())
        passed = bool(v.get("pass") is True)
        feedback = v.get("feedback") or ""
    except json.JSONDecodeError:
        passed = False
        feedback = f"[质检员输出不是合法 JSON，默认不通过] 原文：{verdict_text}"

    if passed:
        task.status = TASK_PASSED
        self.broadcast(reviewer, f"✅ {task.assignee} 通过质检")
    elif task.review_attempts >= MAX_REVIEW_ATTEMPTS:  # = 3
        task.status = TASK_FAILED
        self.broadcast(reviewer, f"❌ {task.assignee} 3 次未通过，任务失败")
    else:
        # 单独 send 给 assignee 反馈，让其重做
        self.send(reviewer, task.assignee, f"质检反馈：{feedback}")
        task.status = TASK_REDOING
```

### 【口播 · 三个关键设计】

1. **单任务级，最多 3 次质检**——每个任务独立计质检次数（`MAX_REVIEW_ATTEMPTS = 3`）。3 次都不过即标记 `failed`，进入终止态，不再重试
2. **质检员是普通 Agent**——通过 `recruit("Q1", "质检员")` 创建，有 messages / inbox / tools / `_react_loop`。它能用 `read_file` 真打开 `result.txt` 复查、用 `execute_bash` 验算——不是死板的 if/else 判定
3. **不通过用 send 单独打回**——`send(reviewer, assignee, feedback)` 一对一，不污染其他成员。assignee 下次 chat 时消化 inbox，看到自己上一版 + 质检反馈（messages 跨 chat 累积），重做后状态回到 `reviewing`

### 【口播 · 级联失败——状态机的副产品】

依赖了 `failed` 任务的任务，不会无限等待。事件循环优先级 ④ 检查：

```python
# 优先级 ④：死锁检查
for t in tasks:
    if t.status == TASK_PENDING:
        failed_deps = [dep for dep in t.depends_on
                       if task_by_assignee[dep].status == TASK_FAILED]
        if failed_deps:
            t.status = TASK_FAILED
            t.result = f"[依赖失败] 依赖的 {failed_deps} 未通过质检，本任务无法执行"
```

**示例**：A1 三次不过 → failed；A2 依赖 A1；下一轮循环 A2 自动标记 failed 并写明原因，A2 的 chat 从未被调过——不浪费 token。

## 第 4 章 真实案例：有依赖的多步任务

### 【口播】

理论讲完，跑一个**真实的多 Agent 协作案例**——比 demo4 的"四个独立任务"更进一步，这次任务之间**有依赖**：

- A2 的乘法要用 A1 的加法结果
- B1 要把前面所有结果写到文件里
- 项目经理**必须分配一名质检员**（规则已在 prompt 里强制）——所有任务完成后由质检员逐个验收，3 次不过即失败

## 第 5 章 三方对比 + 总结 + 未来

### 5.1 Plan vs Subagent vs Team 三方对比

### 【屏显 · 表格 5-1 — 三种协作模式】

| 维度 | Plan（demo2/3） | Subagent（demo4） | **Team（demo5）** |
|---|---|---|---|
| **执行单元** | 主 Agent + step 列表 | 主 Agent + 一次性 Subagent | **Team 编排多个持久 Agent** |
| **messages** | 所有 step 共享一份 | 每个 Subagent 独立一份 | **每个 Agent 独立一份 + inbox** |
| **通信** | step 间通过共享 messages 看到 | 完全隔离 | **send 一对一 + broadcast 群发** |
| **生命周期** | 主 Agent 持续整个 plan | Subagent 一次性 | **Agent 持续整个 team（多次 chat）** |
| **身份** | 全程同一个 system_prompt | 临时 role | **固定 name + role** |
| **调度模型** | 按列表顺序执行 | 按调用顺序执行 | **事件循环 + 任务状态机** |
| **质检机制** | 无 | 无 | **质检员持续监听，单任务最多 3 次** |
| **依赖处理** | 共享 messages 自然可见 | 处理不了 | **depends_on + 状态机级联传递** |
| **失败处理** | 整个 plan 失败 | subagent 失败 | **单任务 failed 自动级联到下游** |
| **适合场景** | 严格步骤序列 | 相互独立 | **有依赖、需要协作、要质检** |

### 【口播 · 三种模式的取舍】

```
任务复杂吗？
├─ 否 → 直接 ReAct
└─ 是 → 子任务之间什么关系？
        ├─ 严格按步骤、有依赖 → Plan（共享 messages，简单粗暴）
        ├─ 相互独立 → Subagent（独立 messages，省 token）
        └─ 有依赖 + 需要通信 + 多次唤起同一角色 → Team（消息驱动，最灵活）
```

### 【口播 · Team 的额外能力】

team 比 plan 更灵活的地方：

- **可反复调用**：plan 的 step 只能一个一个去调；team 的 Agent 可以**多次被 chat**——做完一轮，再交给它一件事
- **事件驱动调度**：质检员持续监听 inbox，任务一完成立即质检，不等其他任务；单任务最多 3 次质检，3 次不过即 failed
- **状态机 + 级联失败**：任务有 `pending/reviewing/redoing/passed/failed` 五态；依赖了 failed 的任务自动级联 failed，不浪费 token
- **可纠正重做**：质检不通过用 `send` 单独打回给执行者；执行者下次 chat 时消化 inbox（含自己上一版 + 质检反馈），重做后状态回到 `reviewing`

### 5.2 关键洞察

### 【口播】

Team 看起来"很高级"，但底层**没有任何新协议、新框架**——它就是：

1. **把 subagent 工具函数 `_run_subagent` 升级为持久化的 `Agent` 类**
2. **加上 inbox / messages 两个属性**（inbox = 临时缓存，messages = 长期记忆）
3. **加上 Team 容器**做协调（招募、一对一通信、群发、解散）
4. **加上任务状态机 + 事件循环**——让质检员持续监听、任务独立质检、依赖自动传递、失败级联传播

### 5.3 未解决的问题：上下文压缩

### 【口播 · 留给下一节的悬念】

注意——demo5 解决了"Agent 之间能通信、有记忆"，但**记忆本身就是个新问题**：

- 每个 Agent 的 messages **跨多轮 chat 累积**——做久了 messages 会**越长越长**
- 即使每次任务本身很短，**累积的记忆也会撑爆上下文窗口**
- team 里多个 Agent 同时累积，**整体上下文压力比 demo4 的 subagent 更大**

demo5 的最后一章我们看到：**即使单个任务短，Agent 的 messages 仍可能存爆**。所以**下一节**我们会做**上下文压缩**——把累积的 messages 做一次"摘要+压缩"，让 Agent 在保留关键信息的同时，把上下文长度控制在合理范围内。

### 【口播 · 收尾金句】

demo1 解决了「能不能干活」，demo2 解决了「记得过去、规划未来」，demo3 让 Agent 守规矩、接外部能力，demo4 让 Agent 能分包合作，**demo5 让 Agent 升级为有身份、有记忆、能协作、会被质检的团队成员——并通过事件驱动状态机让协作真正像项目组一样运转**。

> ### demo2 = demo1 × 记忆 × 规划
> ### demo3 = demo2 × Rules × MCP
> ### demo4 = demo3 × Subagent（分工合作）
> ### demo5 = demo4 × Team × 事件驱动状态机（协作与编排）
> ### demo6 = demo5 × 上下文压缩

下一节，我们会讲**Agent 的上下文压缩**——把累积的长期记忆做摘要、做压缩，让 Agent 跑得再久也不会"撑爆"。这才是真正能跑在生产环境的 Agent。
