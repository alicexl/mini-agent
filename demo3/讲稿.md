## 第 1 章 结论：demo3 比 demo2 多了什么

### 【口播 · 开场】

前两节，我们走完了 Agent 最核心的演进路径：

> ### demo1 = LLM × 工具 × 循环
> ### demo2 = demo1 × 记忆 × 规划

demo2 已经能干活、记得过去、会规划。但 demo2 在结尾留了**三个遗憾**——这正是 demo3 要补上的：

1. **工具硬编码**：所有工具都写死在 `agent.py` 里，想接外部能力（查天气、查数据库、调第三方 API）就得改代码
2. **无行为约束**：大模型想写什么代码就写什么代码、想跑什么命令就跑什么命令
3. **规划要手动开**：用户要记得敲 `--plan` 或 `/plan`，Agent 自己不知道什么时候该规划

### 【屏显 · 表格 1-1 — demo3 vs demo2】

| 维度 | demo2 | demo3 |
|---|---|---|
| **工具来源** | 全部本地硬编码 | **本地 4 + MCP 3 = 7 个工具，跨进程发现** |
| **plan 入口** | 独立 `--plan` / `/plan` 命令，用户手动开关 | **`run_agent` 顶层 1 轮决策，LLM 自主分叉** |
| **行为约束** | ❌ 无 | ✅ `.agent/rules.md` 注入 system prompt |
| **跨进程能力** | ❌ 工具只能同进程函数调用 | ✅ JSON-RPC 2.0 over HTTP |

### 【口播 · 三处不变】

注意——**demo2 的三个核心都没动**：

1. **本地工具集基本不变**（execute_bash / read_file / write_file）
2. **单步执行不变**（每个 step 内部仍然是 ReAct：思考→行动→观察）
3. **记忆系统不变**（agent_memory.md + 滑动窗口）

我们只做**三件新事**：把工具搬到独立服务（MCP）、加一份规范文件（Rules）、把 plan 改成 `run_agent` 顶层的自动决策分叉。

新公式：

> ### demo3 = demo2 × Rules × MCP（plan 自动决策）

接下来 6 章，逐层拆开。

## 第 2 章 Rules：上下文约束（最简的增强）

### 【口播 · 为什么需要约束】

写代码的 Agent 最容易出**两个毛病**：

- **风格漂移**——一会儿用 `camelCase`，一会儿用 `snake_case`；一会儿 Python 3.8 语法，一会儿 3.11 语法
- **行为失范**——`rm -rf /` 之类危险命令也敢直接执行；写出 500 行的巨型函数

demo2 完全没有约束机制。怎么补？**最简方案：用一个 Markdown 文件，拼到 system prompt 里**。

这就是 Rules。

### 【屏显 · 目录结构】

```
demo3/
├── agent.py
├── mcp_server.py
└── .agent/                ← 隐藏目录（demo3 新增）
    ├── rules.md           ← Rules 规范文件
    └── skills/            ← Skills 占位目录（后续单讲）
```

### 2.1 Rules 文件长什么样

### 【屏显 · .agent/rules.md（节选）】

```markdown
## 一、代码生成规范

1. **Python 版本**：仅使用 Python 3.10 及以上语法
2. **命名规范**：
   - 变量 / 函数 / 模块：snake_case
   - 类：PascalCase
   - 常量：UPPER_SNAKE_CASE
3. **函数长度**：单个函数不超过 100 行。超过就拆分。
4. **类型注解**：所有公开函数必须带类型注解。
```

### 【口播】

注意——**这就是普通 Markdown**，不是配置文件、不是代码。每一项都是用自然语言写"我希望 Agent 怎么做事"。

### 2.2 Rules 怎么生效：拼到 system prompt

### 【口播】

Rules 文件**不会被大模型主动读取**。它生效的方式是：**Agent 启动时把它整个拼到 system prompt 的后缀**，让大模型每次决策都能"看到"这份规范。

### 【屏显 · 代码 2-1 — build_system_prompt】

```python
def build_system_prompt(verbose: bool = False) -> str:
    parts = [
        "你是一个有用的助手，可以通过工具与本地系统、远程 MCP 服务交互。",
        ...
    ]

    rules = load_rules()                                    # ← 读 Rules
    if rules:
        parts.append("\n## 项目规范（Rules）\n\n" + rules)   # ← 拼到后缀

    memory = load_memory()                                  # ← 读记忆（沿用 demo2）
    if memory:
        parts.append("\n## 历史任务记忆（最近）\n\n" + memory)

    return "\n".join(parts)
```

### 【口播 · Rules 的本质】

**Rules 是 demo2「记忆系统」的同构延伸**——都是「把外部文件的内容拼到 system prompt」这件事。区别只是：

- **记忆** 拼的是「过去做了什么」（动态、随任务增长）
- **Rules** 拼的是「希望 Agent 怎么做」（静态、人写好的）

底层完全是同一个机制：**有选择地把外部信息搬运进 prompt**。

## 第 3 章 MCP 协议：从函数调用到 RPC

### 【口播 · demo2 工具的局限】

demo2 的工具都是**本地函数**：

```python
AVAILABLE_FUNCTIONS = {
    "execute_bash": execute_bash,
    "read_file":    read_file,
    "write_file":   write_file,
}
```

大模型说「调用 execute_bash」，Agent 通过这张字典直接调函数。**简单、高效，但能力被锁在同一个 Python 进程里**。

现实世界的 Agent 需要的能力远不止这些——查天气、查数据库、调第三方 API、用企业内部服务...这些能力往往：

- **不在本地**——是别人维护的远程服务
- **不是 Python**——可能是 Node、Go、Rust 写的
- **不能 import**——服务跑在容器里、内网里、云上

**怎么把"任何外部能力"统一接入 Agent？** 这就是 MCP（Model Context Protocol）要解决的问题。

### 3.1 MCP 的本质：把工具搬到 HTTP 服务后面

### 【口播】

MCP 的核心思路一句话说清：**把工具从"Agent 进程里的函数"改成"独立 HTTP 服务暴露的 RPC 方法"**。

### 【屏显 · 表格 3-1 — 本地工具 vs MCP 工具】

| 维度 | 本地工具（demo2） | MCP 工具（demo3） |
|---|---|---|
| **载体** | Python 函数 | HTTP 服务（任何语言） |
| **发现方式** | 看 `agent.py` 源码 | 调 `tools/list` 拿 schema 列表 |
| **调用方式** | `fn(**args)` 函数调用 | POST JSON-RPC `tools/call` |
| **位置** | 同进程 | 同机 / 跨机 / 跨云 |
| **扩展** | 改代码、重启 Agent | 加一个 MCP server，Agent 自动发现 |

### 【口播】

打个比方：本地工具像是**自己家里的工具箱**——你随时拿，但容量有限；MCP 工具像是**云端工具租赁服务**——你要先查目录（`tools/list`）、再下单租用（`tools/call`），但理论上能租到任何东西。

### 3.2 JSON-RPC 2.0：MCP 的传输协议

### 【口播】

MCP 用的传输协议叫 **JSON-RPC 2.0**——本质就是 **HTTP POST + 约定格式的 JSON**。

### 【屏显 · 表格 3-2 — JSON-RPC 2.0 请求 / 响应格式】

**请求**（Agent → MCP Server）：

```json
{
  "jsonrpc": "2.0",
  "id":      1,
  "method":  "tools/call",
  "params":  {"name": "add", "arguments": {"a": 2, "b": 3}}
}
```

**响应 · 成功**：

```json
{"jsonrpc": "2.0", "id": 1, "result": {"content": [{"type": "text", "text": "5"}]}}
```

**响应 · 失败**：

```json
{"jsonrpc": "2.0", "id": 1, "error": {"code": -32601, "message": "未知方法: foo"}}
```

### 【口播 · MCP 三大核心 method】

MCP 协议规定了**三个核心 method**，这就是 MCP Server 必须实现的全部接口：

### 【屏显 · 表格 3-3 — MCP 三大核心 method】

| method | 中文名 | 作用 |
|---|---|---|
| `initialize` | 握手 | 协议版本协商、能力声明（真实场景还会做鉴权） |
| `tools/list` | 工具发现 | 返回完整工具 schema 列表 |
| `tools/call` | 工具调用 | 按 `name` + `arguments` 执行，返回 `content` 包装结果 |

## 第 4 章 MCP Server 实现（mcp_server.py）

### 【口播】

理论讲完，开始写代码。`mcp_server.py` 全文不到 300 行，分 4 个 Part。

### 4.1 Part 1：工具定义（schema + 实现）

### 【口播】

每个工具仍然是「schema + fn」两件套——和本地工具**完全同构**。schema 字段名都统一用 `input_schema`（snake_case），方便 Agent 合并。唯一的区别：它们不被 Agent 进程直接 import，而被一个独立的 HTTP 服务持有。

### 【屏显 · 代码 4-1 — 三个工具的 schema + 实现】

```python
TOOLS = [
    {
        "name": "add",
        "description": "计算两个数字的和",
        "input_schema": {                      # 与 agent.py 本地工具统一用 input_schema
            "type": "object",
            "properties": {
                "a": {"type": "number"},
                "b": {"type": "number"},
            },
            "required": ["a", "b"],
        },
    },
    # multiply 同构
    # weather 查询指定城市（演示用，写死/随机）
]

def fn_add(a, b):
    return a + b

def fn_weather(city):
    if city in _WEATHER_DB:
        condition, temp = _WEATHER_DB[city]
    else:
        condition = random.choice(["晴", "多云", "阴", "小雨"])
        temp = random.randint(10, 35)
    return f"{city} 今天天气：{condition}，气温 {temp}°C"

# 路由表：工具名 → 实现函数（与 Agent 端的 AVAILABLE_FUNCTIONS 同构）
TOOL_FUNCTIONS = {"add": fn_add, "multiply": fn_multiply, "weather": fn_weather}
```

### 【口播】

**这一层和 demo1 完全一样**——schema 描述能力、fn 实现能力、字典做路由表。MCP 没有重新发明工具，它只是**把工具搬到了 HTTP 后面**。

### 4.2 Part 3：JSON-RPC Handler（三大 method 分发）

### 【口播】

Handler 是 MCP 协议的核心。它接收 JSON-RPC 请求，按 `method` 字段分发到三个内部函数。

### 【屏显 · 代码 4-2 — handle_request 主分发】

```python
def handle_request(payload: dict) -> dict:
    method = payload.get("method")
    req_id = payload.get("id")
    params = payload.get("params", {}) or {}

    try:
        if method == "initialize":
            result = _handle_initialize(params)
        elif method == "tools/list":
            result = _handle_tools_list(params)
        elif method == "tools/call":
            result = _handle_tools_call(params)
        else:
            # JSON-RPC 标准错误码 -32601 (Method not found)
            return _error(req_id, -32601, f"未知方法: {method}")

        return {"jsonrpc": "2.0", "id": req_id, "result": result}

    except Exception as e:
        # 标准错误码 -32603 (Internal error)
        return _error(req_id, -32603, f"服务端内部错误: {e}")
```

### 【屏显 · 代码 4-3 — 三个 method 的实现】

```python
def _handle_initialize(params):
    """握手：协议版本 + 能力声明（真实场景这里会做鉴权）"""
    return {
        "protocolVersion": "2024-11-05",
        "capabilities": {"tools": {}},
        "serverInfo": {"name": "demo3-mcp-server", "version": "1.0.0"},
    }

def _handle_tools_list(params):
    """工具发现：返回完整 schema 列表"""
    return {"tools": TOOLS}

def _handle_tools_call(params):
    """工具调用：name + arguments → 执行 → content 包装返回"""
    name = params.get("name")
    arguments = params.get("arguments", {}) or {}

    if name not in TOOL_FUNCTIONS:
        raise ValueError(f"未知工具: {name}")

    fn = TOOL_FUNCTIONS[name]
    value = fn(**arguments)                          # ← 关键：跨进程 RPC 与本地函数的衔接点

    # MCP 协议规定 tools/call 返回值用 content 数组包装（便于多模态扩展）
    return {"content": [{"type": "text", "text": str(value)}]}
```

### 【口播 · 一个关键细节】

注意 `tools/call` 最后那一行：`{"content": [{"type": "text", "text": ...}]}`。今天我们只塞纯文本，但这个结构是为**多模态**留的——未来同一个工具调用可以同时返回 text、image、audio。MCP 协议层已经做好了扩展接口。

### 4.3 Part 4：HTTP Server 入口

### 【口播】

最后用一个标准库 `http.server` 起一个 HTTP 服务。所有 JSON-RPC 走同一个端点 `/mcp`。

### 【屏显 · 启动 MCP Server】

```bash
$ python mcp_server.py
============================================================
Demo3 MCP Server 已启动
监听:   http://127.0.0.1:8888/mcp
协议:   JSON-RPC 2.0 over HTTP
工具:   add, multiply, weather
============================================================
等待 Agent 调用...
```

服务起来后，可以用 `curl` 直接调它，验证 MCP 协议是**完全开放的**——任何 HTTP 客户端都能用：

### 【屏显 · 用 curl 调用 MCP】

```bash
$ curl -X POST http://127.0.0.1:8888/mcp \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"add","arguments":{"a":2,"b":3}}}'

{"jsonrpc":"2.0","id":1,"result":{"content":[{"type":"text","text":"5"}]}}
```

**就这么简单**——MCP Server 的本质就是一个 HTTP 服务，按 JSON-RPC 协议处理三个 method。没有任何黑魔法。

## 第 5 章 Agent 端：MCP Client + 工具合并

### 【口播】

Server 端讲完，回到 Agent 端。Demo3 给 `agent.py` 加了三块新东西：

1. **MCP Client**（Part 5）—— 把 JSON-RPC 调用封装成 Python 方法
2. **工具合并**（Part 6）—— 本地工具 + MCP 工具 = 完整工具列表
3. **统一路由**（Part 7）—— 主循环里按工具名分发：本地走函数、MCP 走 client

### 5.1 MCP Client：把 JSON-RPC 调用包装成方法

### 【口播】

每次手写 JSON-RPC 请求太繁琐。封装成一个 `MCPClient` 类，把"组装 payload / POST / 解析响应 / 错误处理"全部藏在 `send()` 里，再在 `send` 之上封三个对应 MCP 三大 method 的方法。

### 【屏显 · 代码 5-1 — MCPClient 三层封装】

```python
class MCPClient:
    def __init__(self, url: str):
        self.url = url
        self._id = 0

    def _next_id(self) -> int:
        self._id += 1
        return self._id                          # JSON-RPC 请求 id 自增

    def send(self, method: str, params: Optional[dict] = None) -> dict:
        """发送一次 JSON-RPC 2.0 请求并返回 result 字段"""
        payload = {
            "jsonrpc": "2.0",
            "id":      self._next_id(),
            "method":  method,
            "params":  params or {},
        }
        resp = requests.post(self.url, json=payload, timeout=30)
        if resp.status_code != 200:
            raise RuntimeError(f"MCP HTTP {resp.status_code}")
        data = resp.json()
        if "error" in data:
            raise RuntimeError(f"MCP 调用失败: {data['error']}")
        return data.get("result", {})

    # ---- 三个上层封装 ----
    def initialize(self):  return self.send("initialize", {...})
    def list_tools(self):  return self.send("tools/list", {}).get("tools", [])
    def call_tool(self, name, arguments):
        result = self.send("tools/call", {"name": name, "arguments": arguments})
        return "\n".join(b.get("text", "")
                         for b in result.get("content", [])
                         if b.get("type") == "text")
```

### 【口播】

这三层封装让 MCP 调用看起来就像普通 Python 方法——`mcp.list_tools()`、`mcp.call_tool("add", {"a":1,"b":2})`——但底层每一次都是完整的 JSON-RPC 往返。

### 5.2 工具合并：一行代码搞定

### 【口播】

启动时一次性拉取 MCP 工具列表，与本地工具合并，得到完整工具集。**整个 task 内不变**。

由于两端 schema 格式完全一致（都用 `input_schema`），合并就是**直接拼接**——不需要任何格式转换。

### 【屏显 · 代码 5-2 — merge_tools】

```python
def merge_tools(local_tools: list, mcp_tools: list) -> list:
    """两端 schema 格式一致，直接拼接。"""
    return list(local_tools) + list(mcp_tools)
```

### 【口播 · 设计取舍】

> **真实 MCP 协议规范用的是 `inputSchema`（驼峰）**。工业级实现要在 Agent 端做一次字段名转换。demo3 既然两端都自己写，就让 MCP server 直接用 `input_schema`，省掉转换函数。这是教学版的"特事特办"——真实场景接入第三方 MCP server 时仍需做这层兼容。

合并之后，**大模型根本不知道哪个工具是本地的、哪个是 MCP 的**——它只看到一个 7 工具的列表。真正的差异藏在 Agent 主循环的**路由分发**里。

### 【屏显 · 表格 5-1 — 合并后的完整工具列表（7 个）】

| 来源 | 工具名 | 用途 |
|---|---|---|
| 本地 | `execute_bash` | 执行 shell 命令 |
| 本地 | `read_file` | 读文件 |
| 本地 | `write_file` | 写文件 |
| 本地 | `plan` | 任务拆解（demo3 新增，下一节详讲） |
| MCP | `add` | 加法 |
| MCP | `multiply` | 乘法 |
| MCP | `weather` | 天气查询 |

### 5.3 主循环路由：`_dispatch_tool` 只管本地 / MCP

### 【口播】

主循环不再 if/else 硬写路由——普通工具调用统一交给 `_dispatch_tool`，**它只负责本地 or MCP 二选一**（plan 由 `run_agent` 顶层拦截，根本不进这里）。

### 【屏显 · 代码 5-3 — 简化的分发器】

```python
def _dispatch_tool(name, args, local_fns, mcp_client, verbose):
    """普通工具分发：本地 or MCP。plan 不在这里。"""
    if name in local_fns:
        return str(local_fns[name](**args))
    return mcp_client.call_tool(name, args)
```

### 【口播 · 关键观察】

注意——**对大模型来说，调用本地工具和调用 MCP 工具是完全一样的体验**：都是 `tool_use` 块、都拿到 `tool_result`。路由差异全藏在这个分发器里，对 LLM 完全透明。

## 第 6 章 Plan 决策：从手动开关到顶层分叉

### 【口播 · demo2 留下的遗憾】

demo2 的 plan 有个尴尬处境——**用户要手动开**：

```bash
python agent.py --plan     # 启动时全局开启
# 或 REPL 内
/plan                      # 切换 Plan 模式
```

这意味着：

- 用户**忘了开**，复杂任务就走一步看一步
- 用户**不知道任务复杂度**，开关时机难判断
- 大模型**最懂任务**，但它没有决定权

demo3 的解法是：**让 `run_agent` 先做 1 轮顶层决策**——LLM 看到工具列表里的 plan 工具，自己判断要不要调。调了就走 plan 分叉（遍历 steps）；没调就直接进 ReAct 循环。

### 6.1 plan 工具的 schema

### 【屏显 · 代码 6-1 — plan 作为工具】

```python
LOCAL_TOOLS = [
    # execute_bash / read_file / write_file 不变
    {
        "name": "plan",
        "description": (
            "任务规划工具。当用户的任务复杂、需要拆解成多个有序步骤时调用。"
            "大模型通过此工具返回结构化的 steps 列表，由 Agent 逐步执行。"
            "简单任务无需调用此工具。"               # ← 关键：让模型自己判断
        ),
        "input_schema": {
            "type": "object",
            "properties": {
                "steps": {
                    "type": "array",
                    "items": {"type": "string"},
                    "description": "3-5 个有序步骤",
                    "minItems": 1,
                    "maxItems": 10,
                }
            },
            "required": ["steps"],
        },
    },
]
```

### 【口播 · description 是关键】

注意 description 那段话——**"复杂任务调用、简单任务无需调用"**。这就是给大模型的"使用提示"。它读了这段话，就会自己判断：用户问"1+2+3=?"——简单，不调 plan；用户问"找所有 todo 整理到 todo.md"——复杂，先调 plan 拆步骤。

### 6.2 plan 的实现：顶层分叉 + 共享 messages 的 ReAct

### 【口播】

demo3 的核心思路：**两种路径都跑同一个 `run_agent_steps`**（共享 messages 的 ReAct 循环），只是工具集不同。`run_agent` 只做 1 轮顶层决策——LLM 用全部工具（含 plan）选一条路。

大模型只负责"要不要拆、怎么拆"；**拆完之后怎么老实一步步做完，是 Agent 的事**。step 间共享 messages，前一步的结果后一步能直接看到，与 demo2 的设计一致。

### 【屏显 · 代码 6-2 — run_agent 顶层分叉 + run_agent_steps 共享 messages】

```python
def run_agent_steps(messages, tools, local_fns, mcp_client,
                    system_prompt, verbose, initial_response=None):
    """共享 messages 的 ReAct 子循环"""
    response = initial_response
    for i in range(1, STEP_MAX_ITERATIONS + 1):
        if response is None:
            response = client.messages.create(
                model=MODEL, tools=tools, system=system_prompt, messages=messages)
        if response.stop_reason != "tool_use":
            return "".join(b.text for b in response.content if b.type == "text")

        messages.append({"role": "assistant", "content": response.content})
        tool_results = []
        for block in response.content:
            if block.type != "tool_use":
                continue
            # name 不可能是 "plan"（step_tools 已去掉），统一走 _dispatch_tool
            result = _dispatch_tool(block.name, block.input or {},
                                     local_fns, mcp_client, verbose)
            tool_results.append({"type": "tool_result",
                                 "tool_use_id": block.id, "content": str(result)})
        messages.append({"role": "user", "content": tool_results})
        response = None
    return f"[未在 {STEP_MAX_ITERATIONS} 轮内完成]"


def run_agent(user_input, all_tools, local_fns, mcp_client, ...):
    messages = [{"role": "user", "content": user_input}]
    response = client.messages.create(tools=all_tools, messages=messages, ...)

    plan_block = next((b for b in response.content
                       if b.type == "tool_use" and b.name == "plan"), None)

    if plan_block:
        # ===== Plan 场景：遍历 steps，共享 messages =====
        steps = plan_block.input.get("steps", []) or []
        messages.append({"role": "assistant", "content": response.content})
        messages.append({  # plan ack：明确告诉 LLM「只做当前 step」
            "role": "user",
            "content": [{"type": "tool_result", "tool_use_id": plan_block.id,
                         "content": (
                             f"已规划 {len(steps)} 个步骤。我会逐个发给你下一步任务，"
                             f"请严格遵守：只执行当前这一步，完成后立即 end_turn，"
                             f"不要预先做后续 step。")}],
        })
        step_tools = [t for t in all_tools if t.get("name") != "plan"]   # 禁止嵌套

        for i, step in enumerate(steps, 1):
            # step 消息也包装一遍，双重保险
            step_msg = f"【Step {i}/{len(steps)}】请只执行这一步：\n{step}\n\n完成后立即 end_turn。"
            messages.append({"role": "user", "content": step_msg})
            run_agent_steps(messages, step_tools, ...)   # 共享 messages

        # 所有 step 完成后，让 LLM 看完整上下文做最终总结
        return run_agent_steps(messages, step_tools, ...)

    # ===== 非 Plan 场景：首轮响应直接灌进 ReAct =====
    return run_agent_steps(messages, all_tools, ..., initial_response=response)
```

### 【屏显 · 表格 6-1 — plan 设计要点】

| # | 设计 | 用意 |
|---|---|---|
| 1 | **`run_agent` 只做 1 轮顶层决策** | 让 LLM 用全部工具（含 plan）选一条路，不参与 step 执行 |
| 2 | **两种场景共享 `run_agent_steps`** | 同一份 ReAct 循环代码，工具集不同（plan 场景去掉 plan） |
| 3 | **step 间共享 messages** | 与 demo2 一致——上下文跨 step 积累，前一步的结果后一步能看见 |
| 4 | **step_tools 去掉 plan（禁止嵌套）** | step 里再调 plan 会让大模型迷路；执行权交给 Agent |
| 5 | **prompt 强制单步执行** | plan ack + step 包装双重告诉 LLM「只做当前 step、不预先做后续」——否则 LLM 会主动链式把所有工具一口气调完，让 step 遍历失去意义 |
| 6 | **plan 完成后显式调 LLM 总结** | 看完整 step 上下文做总结，不会"再来一遍" |

### 【口播 · 为什么还要 Agent 接管 step 执行】

为什么不让大模型在 plan 后自己逐步骤执行？

因为实战中**大模型会"迷路"**——长任务里它会忘了自己在第几步、跳步、或者绕圈。更狡猾的是，**当你把 plan 全貌放进 messages，LLM 看到 plan 列表 + 工具都齐，往往会主动链式一口气把所有工具调完**——表面上效率高，实际上让 step 遍历、进度条、可中断/重试都失去了意义。

`run_agent` 显式遍历 steps、显式追加到共享 messages、显式调 LLM 做最终总结，并且在 **plan ack + step 包装消息里双重告诉 LLM「只做当前 step、不预先做后续」**。这样**大模型只做它擅长的（拆 / 单步执行 / 总结），不擅长的（状态机推进）由代码兜底**。

### 【屏显 · 图 6-1 — demo3 的顶层分叉 + 共享 messages ReAct】

```
  run_agent（顶层 1 轮决策）
    ├ LLM tool_use=plan([s1, s2, s3])
    ├ plan ack 回灌 → 剥离 plan 工具
    ├ 遍历 steps（共享 messages）：
    │   ├ messages.append(user=s1) → run_agent_steps(...)
    │   ├ messages.append(user=s2) → run_agent_steps(...)   ← 看到 s1 的结果
    │   └ messages.append(user=s3) → run_agent_steps(...)   ← 看到 s1/s2 的结果
    └ run_agent_steps(...)（最终总结）→ end_turn
```

### 【口播 · 与 demo2 同构】

注意——**demo3 的结构和 demo2 同构**：都是「`run_agent` 编排 + `run_agent_steps` 执行」的双层。区别只在：demo2 由用户手动开 plan，demo3 由 LLM 在第 1 轮决策自动分叉。

这正是工业级 Agent 框架的设计思路：**让 LLM 用工具决定一切**，包括"要不要先规划"；**让 Agent 代码兜底执行细节**，不让大模型独自承担长程任务的状态管理。

### 【口播】

讲到这里，把 demo3 跑起来看一个真实案例。任务**故意选成有自然多步结构**——加法 → 乘法 → 查天气 → 写文件 → 读回验证，5 个 step，让大模型有充分的理由调 plan。

### 【屏显 · 第 1 轮 ReAct（LLM 选择 plan）】

```

  - tool_use : plan({'steps': [

### 【口播 · 三点观察】

| # | 观察 | 启示 |
|---|---|---|
| 1 | **每个 step 恰好 1 次工具调用** | prompt 双重约束（plan ack + step 包装）让 LLM 严格只做当前 step，调完即停 |
| 2 | **Step 2 直接复用 Step 1 的 82** | 共享 messages 让 multiply 拿到 add 的 tool_result——无需 LLM 重算 |
| 3 | **Step 4 write_file 内容包含前 3 步结果** | LLM 从 messages 里看到 add/multiply/weather 三条 tool_result，组装出报告内容 |

注意——**大模型根本不知道 `add` / `multiply` / `weather` 是远程工具**。Step 1-3 走 MCP RPC、Step 4-5 走本地函数，路由差异全藏在 `_dispatch_tool`，对 LLM 完全透明。

### 【口播 · 为什么必须强制单步】

如果不强制，会发生什么？**LLM 进入 Step 1 时看到 messages 里 plan 的全貌（5 个 step 列表）+ 工具都齐，它会主动链式一口气把 5 个工具全调完**——表面上效率高，实际上：

- ❌ Step 2-5 的进入变成纯浪费（LLM 看到「做过了」就 end_turn，每个 step 烧 1 次 LLM 请求什么也没做）
- ❌ plan 的「分步」「进度条」「可中断/重试」语义全失去意义
- ❌ 长任务里 LLM 一口气调完很容易中途跑偏，没法逐步兜底

所以 `run_agent` 在 **plan ack 内容** + **step 包装消息**双重告诉 LLM「只做当前 step、完成后立即 end_turn」。这样**大模型只做它擅长的（拆 / 单步执行 / 总结），状态机推进由代码兜底**。

### 7.4 完整执行结构图

### 【屏显 · 图 7-1 — 一次 plan 任务的完整结构】

与图 6-1 同构，只是把具体步骤填进去：

```

  ├ LLM tool_use=plan([s1..s5])  ← 1 次 LLM
  ├ plan ack 回灌（含「只做当前 step」指令）→ 剥离 plan 工具

| # | 观察 | 启示 |
|---|---|---|
| 1 | **每个 step 严格 1 次工具调用** | prompt 强制单步（plan ack + step 包装）→ LLM 调完工具立即 end_turn，不主动链式 |
| 2 | **Step 2 复用 Step 1 的 82** | 共享 messages 让后续 step 直接看到前序 tool_result，无需重算 |
| 3 | **MCP 工具和本地工具在每个 step 里混用** | 大模型无感——add/multiply/weather 走 MCP RPC，write_file/read_file 走本地函数，路由差异藏在 `_dispatch_tool` |
| 4 | **最终总结无新工具调用** | 5 个 step 跑完后 `run_agent_steps` 只让 LLM 看 messages 收尾，0 次冗余 tool_use |
| 5 | **总计 12 次 LLM 请求 + 5 次工具调用** | 每个 step 2 次 LLM（tool_use + end_turn）+ 顶层 1 次 + 最终总结 1 次 |

### 【口播 · 为什么这个案例值得讲】

不是大模型变聪明了，是**架构选得对**：

- ✅ 顶层分叉：plan 走共享 messages 子循环、非 plan 走透传 initial_response —— **两条路径都不重发请求**
- ✅ prompt 强制单步：让 plan 真正成为「分步推进」而非「LLM 一口气做完」—— **状态机推进由代码兜底，不让大模型独自承担长程任务**
- ✅ 共享 messages：上下文跨 step 积累，Step 2 直接看到 Step 1 的 tool_result —— **消灭信任鸿沟**
- ✅ 任务选得干净：算术走 MCP、文件读写走本地，避开了 shell/Python 跨平台坑，让架构特性得以纯粹展示

**真实任务往往没有这么干净**——一旦碰到 `execute_bash`、跨操作系统、外部 API，就会看到 step 失败、重试、ReAct 兜底等机制真正发挥兜底作用。但作为「plan 设计意图 + 单步强制 + 共享 messages」的展示，这个真实跑通的案例已经把 `run_agent` 分叉 + `run_agent_steps` 的协作讲透了。

### 7.6 对照：非 plan 简单场景（顶层分叉的另一条路径）

### 【口播】

刚才看的是 plan 路径——大模型主动拆步骤、共享 messages 逐步执行。但日常用户提问大多是**单步就能搞定**的，比如「查一下上海天气」。这种场景走的是 `run_agent` 顶层分叉的另一条路——**直接进入 ReAct**，没有 plan 拆解、没有 step 遍历。

### 【屏显 · 简单查询的真实运行轨迹】

```
用户: 用 weather 工具查一下上海天气

============================================================
顶层决策：plan or 直接执行
============================================================
[messages] 当前 1 条消息
  [0] user     : 用 weather 工具查一下上海天气

[LLM 决策] stop_reason = tool_use
  - tool_use : weather({'city': '上海'})      ← LLM 直接选了 MCP 工具，没拆 plan

[非 Plan] 直接进入 ReAct 循环（共享 messages）
  [LLM] weather({'city': '上海'})
  [工具 · MCP]  weather({'city': '上海'})
  [结果] 上海 今天天气：多云，气温 22°C
  [迭代 2 完成] 🌤️ 上海天气：多云，22°C，适宜出行

助手: 🌤️ **上海天气查询结果：**
       - 天气状况：多云
       - 气温：22°C
       今天上海多云，气温适宜，适合出行！
```

### 【口播 · 三点观察】

| # | 观察 | 启示 |
|---|---|---|
| 1 | **LLM 第 1 轮直接选 weather** | 任务简单时 LLM 不会无脑拆 plan，顶层决策自然走到非 plan 分叉 |
| 2 | **`initial_response` 透传** | `run_agent` 把顶层那一次的 response 直接传给 `run_agent_steps` 当起点，**不重发 LLM 请求**——零冗余 |
| 3 | **总计 2 次 LLM 请求 + 1 次工具调用** | 顶层决策 1 次 + ReAct 收尾 1 次，工具调用只发生在 ReAct 子循环里 |

### 【口播 · 两条路径的对称性】

回头看 `run_agent` 的顶层分叉——

- **plan 路径**：1 次顶层 LLM 请求拿到 plan tool_use → 遍历 steps，每步 `run_agent_steps`（共享 messages）→ 最终一次 `run_agent_steps` 收尾
- **非 plan 路径**：1 次顶层 LLM 请求拿到非 plan 的 tool_use → 把这次 response **透传**给 `run_agent_steps` 当起点 → 一轮 ReAct 收尾

两条路径**用同一个 `run_agent_steps` 兜底**，差别只在要不要遍历 steps、要不要透传 initial_response。这就是顶层分叉设计的美感——**「判断走哪条路」和「走路」彻底分开**，主循环保持极简。

回顾整节，demo3 完成了**三次升级**：

| 1 | **Rules** | `.agent/rules.md` 拼进 system prompt | 无行为约束 |
| 2 | **MCP** | 工具搬到独立 HTTP 服务，JSON-RPC 协议 | 工具硬编码、无法接外部能力 |
| 3 | **plan 自动决策** | `run_agent` 顶层 1 轮决策分叉 | 大模型无法自主决定是否规划 |

**三次升级三次对症下药**——Rules 解决行为约束、MCP 解决外部工具、plan 自决策解决手动开关。

### 【口播 · 关键洞察】

这三件事底层都是**同一个机制的不同应用**——**"把外部信息搬运进 prompt / 把内部决策暴露给外部"**：

- **Rules** = 把规范文件搬运进 system prompt
- **记忆**（demo2）= 把历史任务搬运进 system prompt
- **plan 工具** = 把"拆步骤"这个决策暴露给大模型（`run_agent` 顶层拦截后分叉）
- **MCP** = 把"工具调用"这个动作暴露给远程服务

一旦你看清这个模式，再看 Claude Code、Cursor、Devin 这些成熟框架，会发现**它们的所有"高级特性"都是这个模式的变体**。
