## 第 1 章 结论：demo3 比 demo1 多了什么

### 【口播 · 开场】

demo1 给我们打下了 Agent 的心跳——`LLM × 工具 × 循环`。但 demo1 的工具只有三件套——`execute_bash` / `read_file` / `write_file`，全部硬编码在 `agent.py` 里。这在真实场景下有两个遗憾：

1. **改文件只能整文件覆写**——改一行也要重发整个文件内容，对大文件极浪费 token
2. **工具被锁在同一个进程里**——想接外部能力（查天气、查数据库、调第三方 API）就得改代码、重启 Agent

demo3 的轴是「工具」，就解决这两件事。

> ### demo3 = base × 工具
> （demo1 是 base，demo3 在 base 上独立加「工具」轴——不叠加 demo2 的记忆轴）

### 【屏显 · 表格 1-1 — demo3 vs demo1】

| 维度 | demo1 | demo3 |
|---|---|---|
| **本地工具** | bash / read / write 三件套 | **+ edit（string replacement）= 四件套** |
| **远程工具** | ❌ 只有同进程函数调用 | ✅ **MCP：JSON-RPC 2.0 over HTTP** |
| **工具总数** | 3（全本地） | **4 本地 + N MCP**（N 由 server 决定） |
| **能力边界** | 同一个 Python 进程 | **跨进程 / 跨机器 / 跨语言** |

### 【口播 · base 不变】

注意——**demo1 的核心都没动**：

1. **ReAct 主循环不变**（LLM → tool_use → tool_result → ... → end_turn）
2. **三个本地工具不变**（execute_bash / read_file / write_file 原封不动）
3. **system prompt + messages 结构不变**

我们只做**两件新事**：加一个本地工具 `edit`、接一个外部工具协议 MCP。

「工具轴」的两个增量分别对应**能力维度**和**协议维度**的扩展：
- **(A) 能力维度**：edit 提供 read+write 做不到的精细修改（只发改动部分）
- **(B) 协议维度**：MCP 让工具可以跨进程 / 跨机器 / 跨语言复用

「工具轴」的两个增量分别对应**能力维度**和**协议维度**的扩展：
- **(A) 能力维度**：edit 提供 read+write 做不到的精细修改（只发改动部分）
- **(B) 协议维度**：MCP 让工具可以跨进程 / 跨机器 / 跨语言复用

## 第 2 章 本地工具扩展：edit

### 【口播 · 为什么需要 edit】

demo1 改文件的唯一方式是 `write_file`——**整文件覆写**。这套机制有个隐藏的问题：

假设你要改一个 500 行的配置文件里的某一个端口号。用 `write_file` 你得：
1. 先 `read_file` 读全文（500 行进 LLM 上下文）
2. 让 LLM 在脑子里改一处
3. `write_file` 把改完的 500 行**完整发回**给工具（500 行出 LLM 上下文）

改一行，来回 1000 行。对 10k 行的代码文件，每次改一个标识符都要重发 10k 行——**token 成本爆炸**。

**edit 只需要两小段**：

```
old = "PORT=8080"
new = "PORT=9090"
```

磁盘上也只重写差异。这就是 Claude Code 的 Edit 工具的核心设计动机。

### 【屏显 · 表格 2-1 — edit vs write_file】

| 维度 | write_file | edit |
|---|---|---|
| **输入** | 整文件内容 | old + new 两段 |
| **适合** | 创建新文件 / 重写文件 | 改一行 / 改一个值 / 改一个标识符 |
| **token 成本** | O(文件大小) | O(改动大小) |
| **误伤风险** | 低（完整覆写） | 需精确匹配 old（含空格/缩进） |

### 2.1 edit 的 schema

### 【屏显 · 代码 2-1 — edit 工具定义】

```python
{
    "name": "edit",
    "description": (
        "精确替换文件中的一段文本（string replacement）。"
        "比 write_file 整文件覆写更精细，适合改一行 / 改一个标识符 / 改一个值。"
        "若 old 在文件中出现多次，默认只替换第一处；replace_all=true 替换全部。"
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "path":        {"type": "string",  "description": "要编辑的文件路径"},
            "old":         {"type": "string",  "description": "要替换的原文本（必须精确匹配，含空格/缩进）"},
            "new":         {"type": "string",  "description": "替换为的新文本"},
            "replace_all": {"type": "boolean", "description": "是否替换全部匹配处（默认 false，只替换第一处）"},
        },
        "required": ["path", "old", "new"],
    },
}
```

### 【口播 · description 的设计】

注意 description 里的三个关键设计：

1. **「精确替换」**——告诉 LLM 这不是模糊匹配，old 必须一字不差
2. **「含空格/缩进」**——这是 LLM 最容易踩的坑（它看不见缩进）
3. **「默认只替换第一处；replace_all=true 替换全部」**——让 LLM 知道有这个开关

description 就是工具的「使用说明书」。写得好，LLM 少走弯路；写得差，LLM 反复试错。

### 2.2 edit 的实现

### 【屏显 · 代码 2-2 — edit 函数】

```python
def edit(path: str, old: str, new: str, replace_all: bool = False) -> str:
    """精确替换文件中的一段文本（demo3 新增）"""
    try:
        if not os.path.exists(path):
            return f"[错误] 文件不存在: {path}"
        if not old:
            return "[错误] old 不能为空字符串（会无限匹配）"

        with open(path, "r", encoding="utf-8") as f:
            content = f.read()

        occurrences = content.count(old)
        if occurrences == 0:
            return f"[错误] 未在 {path} 中找到要替换的文本。请用 read_file 确认精确内容（含空格/缩进）。"

        if replace_all:
            new_content = content.replace(old, new)
            which = f"全部 {occurrences} 处"
        else:
            new_content = content.replace(old, new, 1)
            which = f"第 1 处（共 {occurrences} 处匹配，未替换的可用 replace_all=true）"

        with open(path, "w", encoding="utf-8") as f:
            f.write(new_content)

        return (
            f"[成功] {path} 替换 {which}；"
            f"文件 {len(content)} → {len(new_content)} 字符"
        )
    except Exception as e:
        return f"[错误] edit 失败: {e}"
```

### 【口播 · 三个防御性设计】

注意 edit 的三个防御性检查：

1. **文件不存在** → 报错，不创建（edit 不是 write_file）
2. **old 为空字符串** → 报错，防止无限匹配（空字符串在 Python 里匹配无数次）
3. **old 不在文件里** → 报错 + 提示「请用 read_file 确认精确内容」——这是给 LLM 的恢复路径

第三个尤其重要。LLM 看不见缩进/空格，经常给出差一点的 old。**报错时告诉它怎么恢复**（用 read_file 看精确内容），它就能自我纠正，而不是瞎试。

### 【口播 · 返回值的设计】

返回值也讲究——包含**替换处数**（`第 1 处 / 全部 3 处`）+ **文件大小变化**（`512 → 518 字符`）。这让 LLM 知道：
- 到底改了几处（是不是改少了？要不要 replace_all？）
- 文件大小变化是否合理（改一个端口号不应该让文件从 500 字符变成 5000）

这些反馈让 LLM 在后续决策里有依据。

## 第 3 章 MCP 协议：从函数调用到 RPC

### 【口播 · demo1 工具的局限】

demo1 的工具都是**本地函数**：

```python
AVAILABLE_FUNCTIONS = {
    "execute_bash": execute_bash,
    "read_file":    read_file,
    "write_file":   write_file,
}
```

大模型说「调用 execute_bash」，Agent 通过这张字典直接调函数。**简单、高效，但能力被锁在同一个 Python 进程里**。

现实世界的 Agent 需要的能力远不止这些——查天气、查数据库、调第三方 API、用企业内部服务...这些能力往往：

- **不在本地**——是别人维护的远程服务
- **不是 Python**——可能是 Node、Go、Rust 写的
- **不能 import**——服务跑在容器里、内网里、云上

**怎么把"任何外部能力"统一接入 Agent？** 这就是 MCP（Model Context Protocol）要解决的问题。

### 3.1 MCP 的本质：把工具搬到 HTTP 服务后面

### 【口播】

MCP 的核心思路一句话说清：**把工具从"Agent 进程里的函数"改成"独立 HTTP 服务暴露的 RPC 方法"**。

### 【屏显 · 表格 3-1 — 本地工具 vs MCP 工具】

| 维度 | 本地工具（demo1） | MCP 工具（demo3） |
|---|---|---|
| **载体** | Python 函数 | HTTP 服务（任何语言） |
| **发现方式** | 看 `agent.py` 源码 | 调 `tools/list` 拿 schema 列表 |
| **调用方式** | `fn(**args)` 函数调用 | POST JSON-RPC `tools/call` |
| **位置** | 同进程 | 同机 / 跨机 / 跨云 |
| **扩展** | 改代码、重启 Agent | 加一个 MCP server，Agent 自动发现 |

### 【口播】

打个比方：本地工具像是**自己家里的工具箱**——你随时拿，但容量有限；MCP 工具像是**云端工具租赁服务**——你要先查目录（`tools/list`）、再下单租用（`tools/call`），但理论上能租到任何东西。

### 3.2 JSON-RPC 2.0：MCP 的传输协议

### 【口播】

MCP 用的传输协议叫 **JSON-RPC 2.0**——本质就是 **HTTP POST + 约定格式的 JSON**。

### 【屏显 · 表格 3-2 — JSON-RPC 2.0 请求 / 响应格式】

**请求**（Agent → MCP Server）：

```json
{
  "jsonrpc": "2.0",
  "id":      1,
  "method":  "tools/call",
  "params":  {"name": "add", "arguments": {"a": 2, "b": 3}}
}
```

**响应 · 成功**：

```json
{"jsonrpc": "2.0", "id": 1, "result": {"content": [{"type": "text", "text": "5"}]}}
```

**响应 · 失败**：

```json
{"jsonrpc": "2.0", "id": 1, "error": {"code": -32601, "message": "未知方法: foo"}}
```

### 【口播 · MCP 三大核心 method】

MCP 协议规定了**三个核心 method**，这就是 MCP Server 必须实现的全部接口：

### 【屏显 · 表格 3-3 — MCP 三大核心 method】

| method | 中文名 | 作用 |
|---|---|---|
| `initialize` | 握手 | 协议版本协商、能力声明（真实场景还会做鉴权） |
| `tools/list` | 工具发现 | 返回完整工具 schema 列表 |
| `tools/call` | 工具调用 | 按 `name` + `arguments` 执行，返回 `content` 包装结果 |

## 第 4 章 MCP Server 实现（mcp_server.py）

### 【口播】

理论讲完，开始写代码。`mcp_server.py` 全文不到 300 行，分 4 个 Part。

### 4.1 Part 1：工具定义（schema + 实现）

### 【口播】

每个工具仍然是「schema + fn」两件套——和本地工具**完全同构**。schema 字段名都统一用 `input_schema`（snake_case），方便 Agent 合并。唯一的区别：它们不被 Agent 进程直接 import，而被一个独立的 HTTP 服务持有。

### 【屏显 · 代码 4-1 — 三个工具的 schema + 实现】

```python
TOOLS = [
    {
        "name": "add",
        "description": "计算两个数字的和",
        "input_schema": {                      # 与 agent.py 本地工具统一用 input_schema
            "type": "object",
            "properties": {
                "a": {"type": "number"},
                "b": {"type": "number"},
            },
            "required": ["a", "b"],
        },
    },
    # multiply 同构
    # weather 查询指定城市（演示用，写死/随机）
]

def fn_add(a, b):
    return a + b

def fn_weather(city):
    if city in _WEATHER_DB:
        condition, temp = _WEATHER_DB[city]
    else:
        condition = random.choice(["晴", "多云", "阴", "小雨"])
        temp = random.randint(10, 35)
    return f"{city} 今天天气：{condition}，气温 {temp}°C"

# 路由表：工具名 → 实现函数（与 Agent 端的 LOCAL_FUNCTIONS 同构）
TOOL_FUNCTIONS = {"add": fn_add, "multiply": fn_multiply, "weather": fn_weather}
```

### 【口播】

**这一层和 demo1 完全一样**——schema 描述能力、fn 实现能力、字典做路由表。MCP 没有重新发明工具，它只是**把工具搬到了 HTTP 后面**。

### 4.2 Part 3：JSON-RPC Handler（三大 method 分发）

### 【口播】

Handler 是 MCP 协议的核心。它接收 JSON-RPC 请求，按 `method` 字段分发到三个内部函数。

### 【屏显 · 代码 4-2 — handle_request 主分发】

```python
def handle_request(payload: dict) -> dict:
    method = payload.get("method")
    req_id = payload.get("id")
    params = payload.get("params", {}) or {}

    try:
        if method == "initialize":
            result = _handle_initialize(params)
        elif method == "tools/list":
            result = _handle_tools_list(params)
        elif method == "tools/call":
            result = _handle_tools_call(params)
        else:
            # JSON-RPC 标准错误码 -32601 (Method not found)
            return _error(req_id, -32601, f"未知方法: {method}")

        return {"jsonrpc": "2.0", "id": req_id, "result": result}

    except Exception as e:
        # 标准错误码 -32603 (Internal error)
        return _error(req_id, -32603, f"服务端内部错误: {e}")
```

### 【屏显 · 代码 4-3 — 三个 method 的实现】

```python
def _handle_initialize(params):
    """握手：协议版本 + 能力声明（真实场景这里会做鉴权）"""
    return {
        "protocolVersion": "2024-11-05",
        "capabilities": {"tools": {}},
        "serverInfo": {"name": "demo3-mcp-server", "version": "1.0.0"},
    }

def _handle_tools_list(params):
    """工具发现：返回完整 schema 列表"""
    return {"tools": TOOLS}

def _handle_tools_call(params):
    """工具调用：name + arguments → 执行 → content 包装返回"""
    name = params.get("name")
    arguments = params.get("arguments", {}) or {}

    if name not in TOOL_FUNCTIONS:
        raise ValueError(f"未知工具: {name}")

    fn = TOOL_FUNCTIONS[name]
    value = fn(**arguments)                          # ← 关键：跨进程 RPC 与本地函数的衔接点

    # MCP 协议规定 tools/call 返回值用 content 数组包装（便于多模态扩展）
    return {"content": [{"type": "text", "text": str(value)}]}
```

### 【口播 · 一个关键细节】

注意 `tools/call` 最后那一行：`{"content": [{"type": "text", "text": ...}]}`。今天我们只塞纯文本，但这个结构是为**多模态**留的——未来同一个工具调用可以同时返回 text、image、audio。MCP 协议层已经做好了扩展接口。

## 第 5 章 Agent 端：MCP Client + 工具合并

### 【口播】

Server 端讲完，回到 Agent 端。Demo3 给 `agent.py` 加了三块新东西：

1. **MCP Client**（Part 5）—— 把 JSON-RPC 调用封装成 Python 方法

### 5.1 MCP Client：把 JSON-RPC 调用包装成方法

### 【口播】

每次手写 JSON-RPC 请求太繁琐。封装成一个 `MCPClient` 类，把"组装 payload / POST / 解析响应 / 错误处理"全部藏在 `send()` 里，再在 `send` 之上封三个对应 MCP 三大 method 的方法。

### 【屏显 · 代码 5-1 — MCPClient 三层封装】

```python
class MCPClient:
    def __init__(self, url: str):
        self.url = url
        self._id = 0
        self.initialized = False

    def _next_id(self) -> int:
        self._id += 1
        return self._id                          # JSON-RPC 请求 id 自增

    def send(self, method: str, params: Optional[dict] = None) -> dict:
        """发送一次 JSON-RPC 2.0 请求并返回 result 字段"""
        payload = {
            "jsonrpc": "2.0",
            "id":      self._next_id(),
            "method":  method,
            "params":  params or {},
        }
        try:
            resp = requests.post(self.url, json=payload, timeout=30)
        except requests.RequestException as e:
            raise RuntimeError(f"MCP 网络错误 ({method}): {e}") from e

        if resp.status_code != 200:
            raise RuntimeError(
                f"MCP HTTP {resp.status_code} ({method}): {resp.text[:200]}")
        data = resp.json()
        if "error" in data:
            raise RuntimeError(f"MCP 调用失败 ({method}): {data['error']}")
        return data.get("result", {})

    # ---- 三个上层封装 ----
    def initialize(self) -> dict:
        result = self.send("initialize", {
            "protocolVersion": "2024-11-05",
            "capabilities":    {},
            "clientInfo":      {"name": "demo3-tools-agent", "version": "1.0.0"},
        })
        self.initialized = True
        return result

    def list_tools(self) -> list:
        return self.send("tools/list", {}).get("tools", [])

    def call_tool(self, name: str, arguments: dict) -> str:
        result = self.send("tools/call", {"name": name, "arguments": arguments})
        content = result.get("content", [])
        texts = []
        for block in content:
            if isinstance(block, dict) and block.get("type") == "text":
                texts.append(block.get("text", ""))
        return "\n".join(texts) if texts else "[MCP 工具无文本返回]"
```

### 【口播】

这三层封装让 MCP 调用看起来就像普通 Python 方法——`mcp.list_tools()`、`mcp.call_tool("add", {"a":1,"b":2})`——但底层每一次都是完整的 JSON-RPC 往返。

### 5.2 工具合并：一行 `+` 搞定

### 【口播】

启动时一次性拉取 MCP 工具列表，与本地工具合并，得到完整工具集。**整个会话内不变**。

由于两端 schema 格式完全一致（都用 `input_schema`），合并就是**直接 `+` 拼接**——不需要任何包装函数或格式转换。

### 【屏显 · 代码 5-2 — 工具合并就是一行 +】

```python
# main() 里：
mcp_tools = mcp_client.list_tools()           # 拿 MCP server 的 schema 列表
all_tools = LOCAL_TOOLS + mcp_tools            # 直接拼接，完事
```

### 【口播 · 为什么这么简单】

这就是 schema 统一的好处——本地工具和 MCP 工具的结构**完全同构**（都是 `{name, description, input_schema}`），对大模型来说没有任何差异。合并不需要函数、不需要转换、不需要标记来源。**一行 `+` 就够了**。

> **真实 MCP 协议规范用的是 `inputSchema`（驼峰）**。工业级实现要在 Agent 端做一次字段名转换。demo3 既然两端都自己写，就让 MCP server 直接用 `input_schema`，省掉转换。这是教学版的"特事特办"——接入第三方 MCP server 时仍需做这层兼容。

合并之后，**大模型根本不知道哪个工具是本地的、哪个是 MCP 的**——它只看到一个工具列表。真正的差异藏在 Agent 主循环的**路由分发**里。

### 【屏显 · 表格 5-1 — 合并后的完整工具列表（7 个）】

| 来源 | 工具名 | 用途 |
|---|---|---|
| 本地 | `execute_bash` | 执行 shell 命令 |
| 本地 | `read_file` | 读文件 |
| 本地 | `write_file` | 写文件（整文件覆写） |
| 本地 | `edit` | 精确替换文件中的一段文本（demo3 新增） |
| MCP | `add` | 加法 |
| MCP | `multiply` | 乘法 |
| MCP | `weather` | 天气查询 |

### 5.3 主循环路由：`_dispatch_tool` 只管本地 / MCP

### 【口播】

主循环不再 if/else 硬写路由——工具调用统一交给 `_dispatch_tool`，**它只负责本地 or MCP 二选一**。

### 【屏显 · 代码 5-3 — 统一分发器】

```python
def _dispatch_tool(name, args, mcp_client, verbose):
    """统一工具分发：本地 or MCP。LLM 不需要知道工具在哪，只按名字调用。"""
    if name in LOCAL_FUNCTIONS:
        if verbose:
            print(f"  [工具 · 本地] {name}({_preview(str(args), 80)})")
        try:
            return str(LOCAL_FUNCTIONS[name](**args))
        except Exception as e:
            return f"[错误] 本地工具 {name} 执行失败: {e}"

    # 不在本地 → 走 MCP
    if verbose:
        print(f"  [工具 · MCP]  {name}({_preview(str(args), 80)})")
    try:
        return mcp_client.call_tool(name, args)
    except Exception as e:
        return f"[错误] MCP 工具 {name} 调用失败: {e}"
```

### 【口播 · 关键观察】

注意——**对大模型来说，调用本地工具和调用 MCP 工具是完全一样的体验**：都是 `tool_use` 块、都拿到 `tool_result`。路由差异全藏在这个分发器里，对 LLM 完全透明。

这也是 MCP 协议的精髓——**把"工具在哪"这个工程细节从 LLM 的视野里抹掉**。LLM 只管「我要做什么」，不管「工具在哪个进程」。

### 5.4 降级模式：MCP 不可用时仍可运行

### 【口播】

真实场景里 MCP Server 可能没启动、可能挂了、网络可能不通。demo3 的 `main()` 做了**优雅降级**：

### 【屏显 · 代码 5-4 — 降级逻辑】

```python
mcp_client = MCPClient(MCP_URL)
mcp_tools = []
try:
    info = mcp_client.initialize()
    print(f"[MCP] 握手成功：{info.get('serverInfo', {})}")
    mcp_tools = mcp_client.list_tools()
    print(f"[MCP] 发现 {len(mcp_tools)} 个远程工具：{', '.join(t['name'] for t in mcp_tools)}")
except Exception as e:
    print(f"[MCP] 连接失败，降级为仅本地工具模式。原因: {e}")
    print(f"[MCP] 请确认已在另一个终端运行：python mcp_server.py")

all_tools = LOCAL_TOOLS + mcp_tools
```

MCP 连不上时 `mcp_tools` 就是空 list，`LOCAL_TOOLS + []` 照常返回 4 个本地工具。Agent 仍可正常运行——只是 add / multiply / weather 不可用。这就是**优雅降级**（graceful degradation）。

### 6.2 案例 1：MCP 远程调用

### 【屏显 · 简单查询的真实运行轨迹】

```
用户: 用 add 算一下 128+256，再用 multiply 算一下结果 ×10

----- ReAct 第 1 轮 -----
[messages] 当前 1 条消息
  [0] user     : 用 add 算一下 128+256，再用 multiply 算一下结果 ×10
[LLM 决策] stop_reason = tool_use
  - tool_use : add({'a': 128, 'b': 256})

----- ReAct 第 2 轮 -----
[messages] 当前 3 条消息
  [工具 · MCP]  add({'a': 128, 'b': 256})
  [结果] 384
[LLM 决策] stop_reason = tool_use
  - tool_use : multiply({'a': 384, 'b': 10})       ← 384 来自上一轮 tool_result

----- ReAct 第 3 轮 -----
[messages] 当前 5 条消息
  [工具 · MCP]  multiply({'a': 384, 'b': 10})
  [结果] 3840
[LLM 决策] stop_reason = end_turn
  - text     : 128 + 256 = 384，384 × 10 = 3840

助手: 计算结果：
  1. 128 + 256 = **384**
  2. 384 × 10 = **3840**
```

### 【口播 · 两点观察】

| # | 观察 | 启示 |
|---|---|---|
| 1 | **add / multiply 走 MCP 通道** | 日志打印 `[工具 · MCP]`，底层是 JSON-RPC POST |
| 2 | **第 2 轮 multiply 直接复用 add 的 384** | 共享 messages 让 LLM 拿到上一轮的 tool_result |

**大模型根本不知道 add / multiply 是远程工具**——它只看到 7 个工具的列表，调用体验完全一致。路由差异全藏在 `_dispatch_tool` 里。

### 6.3 案例 2：edit 精细修改

### 【口播】

现在看 edit 怎么在大文件上做小改动。先用 write_file 创建一个配置文件，再用 edit 改一个值。

### 【屏显 · edit 的真实运行轨迹】

```
用户: 先用 write_file 创建 config.txt 写入三行端口配置（8080/8443/9090），
      然后用 edit 把 8080 改成 80

----- ReAct 第 1 轮 -----
[LLM 决策] stop_reason = tool_use
  - tool_use : write_file({'path': 'config.txt', 'content': 'PORT_HTTP=8080\nPORT_HTTPS=8443\nPORT_ADMIN=9090\n'})
  [工具 · 本地] write_file(...)
  [结果] [成功] 文件已写入: config.txt (47 字符)

----- ReAct 第 2 轮 -----
[LLM 决策] stop_reason = tool_use
  - tool_use : edit({'path': 'config.txt', 'old': 'PORT_HTTP=8080', 'new': 'PORT_HTTP=80'})
  [工具 · 本地] edit(...)
  [结果] [成功] config.txt 替换 第 1 处（共 1 处匹配）；文件 47 → 44 字符

----- ReAct 第 3 轮 -----
[LLM 决策] stop_reason = end_turn
  - text     : 已完成。config.txt 的 HTTP 端口从 8080 改为 80。

助手: ✅ 已完成：
  1. 创建 config.txt（3 行端口配置）
  2. edit 把 PORT_HTTP=8080 改成 PORT_HTTP=80
  文件 47 → 44 字符（只少 3 个字符，正是 8080→80 的差异）
```

### 【口播 · edit 的价值】

| # | 观察 | 启示 |
|---|---|---|
| 1 | **edit 只发 old + new 两段** | 不用重发整个文件内容——对 500 行的文件改一行，节省数百倍 token |
| 2 | **返回值含替换处数 + 文件大小变化** | `第 1 处（共 1 处匹配）；47 → 44 字符`——LLM 有依据判断改动是否合理 |
| 3 | **日志打印 `[工具 · 本地]`** | 与 MCP 工具的 `[工具 · MCP]` 对照，路由差异可视化 |

### 6.4 案例 3：edit 的防御性报错

### 【口播】

LLM 看不见缩进/空格，edit 最常见的失败是 old 不精确匹配。demo3 的报错会引导 LLM 自我纠正。

### 【屏显 · edit 报错的真实轨迹】

```
用户: 把 config.txt 里的 HTTPS 端口改成 443

----- ReAct 第 1 轮 -----
[LLM 决策] stop_reason = tool_use
  - tool_use : edit({'path': 'config.txt', 'old': 'PORT_HTTPS=8443', 'new': 'PORT_HTTPS=443'})
  [工具 · 本地] edit(...)
  [结果] [错误] 未在 config.txt 中找到要替换的文本。请用 read_file 确认精确内容（含空格/缩进）。

----- ReAct 第 2 轮 -----
[LLM 决策] stop_reason = tool_use
  - tool_use : read_file({'path': 'config.txt'})       ← LLM 按报错提示恢复
  [工具 · 本地] read_file(...)
  [结果] PORT_HTTP=80
PORT_HTTPS=8443
PORT_ADMIN=9090

----- ReAct 第 3 轮 -----
[LLM 决策] stop_reason = tool_use
  - tool_use : edit({'path': 'config.txt', 'old': 'PORT_HTTPS=8443', 'new': 'PORT_HTTPS=443'})
  [结果] [成功] config.txt 替换 第 1 处；文件 44 → 44 字符
```

> 注：上例为讲解用的典型轨迹。实际运行时 LLM 可能一次就匹配成功（取决于它生成 old 字符串时是否精确）。关键点是——**即使失败，报错也给了恢复路径**（read_file 看精确内容），LLM 能自我纠正。

### 【口播 · 报错设计的三层智慧】

| 层 | 做法 | 用意 |
|---|---|---|
| 1 | 报 `[错误]` 而非静默失败 | 让 LLM 知道这轮没成功 |
| 2 | 报「未找到要替换的文本」 | 精准告诉 LLM 失败原因（不是文件不存在，是 old 不匹配） |
| 3 | 提示「请用 read_file 确认精确内容」 | 给恢复路径——LLM 下一步就知道该干什么 |

这三层加在一起，让 edit 的失败**不致命**——LLM 能自己爬起来重试。这就是防御性设计的力量。

## 第 7 章 总结与下一节预告

### 【口播 · demo3 做了什么】

回顾整节，demo3 完成了**工具轴的两次扩展**：

### 【屏显 · 表格 7-1 — demo3 的两次扩展】

| # | 扩展 | 做法 | 解决的问题 |
|---|---|---|---|
| 1 | **edit（本地）** | 新增 string replacement 工具 | write_file 整文件覆写改一行太浪费 |
| 2 | **MCP（远程）** | 工具搬到独立 HTTP 服务，JSON-RPC 协议 | 工具硬编码、无法接外部能力 |

**两次扩展分别对应能力维度和协议维度**——edit 让本地工具更精细，MCP 让工具能力突破进程边界。

### 【口播 · 关键洞察】

这两件事底层都是**同一个模式**——**「把工具的能力边界往外推一步」**：

- **edit** = 把"修改文件"的能力从「整文件覆写」推到「精确替换」
- **MCP** = 把"工具所在的位置"从「同进程」推到「跨进程 / 跨机器」

一旦你看清这个模式，再看 Claude Code、Cursor、Devin 这些成熟框架，会发现**它们的所有"工具扩展"都是这个模式的变体**——更精细的本地工具、更丰富的远程服务、更开放的协议。
