## 第 1 章 结论：demo3 比 demo1 多了什么

### 【口播 · 开场】

> ### demo1 = LLM × 工具 × 循环 × 状态
> ### demo3 = base × 工具

### 【口播 · base 不变】

注意——**demo1 的核心都没动**：

1. **ReAct 主循环不变**（LLM → tool_use → tool_result → ... → end_turn）
2. **三个本地工具不变**（execute_bash / read_file / write_file 原封不动）
3. **system prompt + messages 结构不变**

我们只做**两件新事**：加一个本地工具 `edit`、接一个外部工具协议 MCP。

### 【屏显 · 表格 1-1 — demo3 vs demo1】

| 维度 | demo1 | demo3 |
|---|---|---|
| **本地工具** | bash / read / write 三件套 | **+ edit（string replacement）= 四件套** |
| **远程工具** | ❌ 只有同进程函数调用 | ✅ **MCP：基于 JSON-RPC 2.0 的工具协议，通过不同 Transport（如 stdio、HTTP）传输消息** |
| **工具总数** | 3（全本地） | **4 本地 + N MCP**（N 由 server 决定） |
| **能力边界** | 同一个 Python 进程 | **跨进程 / 跨机器 / 跨语言** |

「工具轴」的两个增量分别对应**能力维度**和**协议维度**的扩展：
- **(A) 能力维度**：edit 提供 read+write 做不到的精细修改
- **(B) 协议维度**：MCP 让工具可以跨进程 / 跨机器 / 跨语言复用

## 第 2 章 本地工具扩展：edit

### 【口播 · 为什么需要 edit】

`edit` 提供了精细修改文件内容的能力——LLM 只发 `old` + `new` 两段，工具在文件里定位并替换，其余内容不用碰。

```
old = "PORT=8080"
new = "PORT=9090"
```

### 2.1 edit 的 schema

### 【屏显 · 代码 2-1 — edit 工具定义】

```python
{
    "name": "edit",
    "description": (
        "精确替换文件中的一段文本（string replacement）。"
        "比 write_file 整文件覆写更精细，适合改一行 / 改一个标识符 / 改一个值。"
        "若 old 在文件中出现多次，默认只替换第一处；replace_all=true 替换全部。"
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "path":        {"type": "string",  "description": "要编辑的文件路径"},
            "old":         {"type": "string",  "description": "要替换的原文本（必须精确匹配，含空格/缩进）"},
            "new":         {"type": "string",  "description": "替换为的新文本"},
            "replace_all": {"type": "boolean", "description": "是否替换全部匹配处（默认 false，只替换第一处）"},
        },
        "required": ["path", "old", "new"],
    },
}
```

### 2.2 edit 的实现

### 【屏显 · 代码 2-2 — edit 函数】

```python
def edit(path: str, old: str, new: str, replace_all: bool = False) -> str:
    """精确替换文件中的一段文本（demo3 新增）"""
    try:
        if not os.path.exists(path):
            return f"[错误] 文件不存在: {path}"
        if not old:
            return "[错误] old 不能为空字符串（会无限匹配）"

        with open(path, "r", encoding="utf-8") as f:
            content = f.read()

        occurrences = content.count(old)
        if occurrences == 0:
            return f"[错误] 未在 {path} 中找到要替换的文本。请用 read_file 确认精确内容（含空格/缩进）。"

        if replace_all:
            new_content = content.replace(old, new)
            which = f"全部 {occurrences} 处"
        else:
            new_content = content.replace(old, new, 1)
            which = f"第 1 处（共 {occurrences} 处匹配，未替换的可用 replace_all=true）"

        with open(path, "w", encoding="utf-8") as f:
            f.write(new_content)

        return (
            f"[成功] {path} 替换 {which}；"
            f"文件 {len(content)} → {len(new_content)} 字符"
        )
    except Exception as e:
        return f"[错误] edit 失败: {e}"
```

### 【口播 · 三个防御性设计】

注意 edit 的三个防御性检查：

1. **文件不存在** → 报错，不创建（edit 不是 write_file）
2. **old 为空字符串** → 报错，防止无限匹配（空字符串在 Python 里匹配无数次）
3. **old 不在文件里** → 报错 + 提示「请用 read_file 确认精确内容」——这是给 LLM 的恢复路径

第三个尤其重要。LLM 看不见缩进/空格，经常给出差一点的 old。**报错时告诉它怎么恢复**（用 read_file 看精确内容），它就能自我纠正。

## 第 3 章 MCP 协议：工具发现与远程调用

### 【口播 · demo1 工具的局限】

demo1 的工具都是**本地函数**：

```python
AVAILABLE_FUNCTIONS = {
    "execute_bash": execute_bash,
    "read_file":    read_file,
    "write_file":   write_file,
}
```

demo1 的工具有一个天然限制：**工具必须存在 Agent 进程内部**。

如果工具来自——另一个团队维护、另一种语言实现、独立部署服务——传统方式只能继续改 Agent 代码。

**MCP 解决的问题：如何让 Agent 动态发现并调用外部工具。**

### 3.1 MCP 的本质：定义工具发现和调用协议

MCP 不关心工具运行在哪里，它解决的是 Agent 如何**发现**工具、**理解**工具、**调用**工具的问题。

### 【屏显 · 表格 3-1 — 本地工具 vs MCP 工具】

| 维度 | 本地工具（demo1） | MCP 工具（demo3） |
|---|---|---|
| **载体** | Python 函数 | HTTP 服务（任何语言） |
| **发现方式** | 看 `agent.py` 源码 | 调 `tools/list` 拿 schema 列表 |
| **调用方式** | `fn(**args)` 函数调用 | POST JSON-RPC `tools/call` |
| **位置** | 同进程 | 同机 / 跨机 / 跨云 |
| **扩展** | 改代码、重启 Agent | 加一个 MCP server，Agent 自动发现 |

### 3.2 JSON-RPC 2.0：MCP 的传输协议

### 【口播】

MCP 用的传输协议叫 **JSON-RPC 2.0**——本质就是 **HTTP POST + 约定格式的 JSON**。

### 【屏显 · 表格 3-2 — JSON-RPC 2.0 请求 / 响应格式】

**请求**（Agent → MCP Server）：

```json
{
  "jsonrpc": "2.0",
  "id":      1,
  "method":  "tools/call",
  "params":  {"name": "add", "arguments": {"a": 2, "b": 3}}
}
```

**响应 · 成功**：

```json
{"jsonrpc": "2.0", "id": 1, "result": {"content": [{"type": "text", "text": "5"}]}}
```

### 【口播 · MCP 三大核心 method】

本 demo 只实现 MCP 的 tools 能力，因此涉及三个主要交互：

### 【屏显 · 表格 3-3 — MCP 三大核心 method】

| method | 中文名 | 作用 |
|---|---|---|
| `initialize` | 握手 | 协议版本协商、能力声明 |
| `tools/list` | 工具发现 | 返回完整工具 schema 列表 |
| `tools/call` | 工具调用 | 按 `name` + `arguments` 执行，返回 `content` 包装结果 |

### 【屏显 · demo3 整体架构】

```
                  LLM
                   |
               tool_use
                   |
           +-------+-------+
           |               |
     Local dispatcher    MCP Client
           |               |
     Python fn         JSON-RPC
                           |
                      HTTP POST
                           |
                      MCP Server
                           |
                    add / multiply / weather
```

LLM 只管发 `tool_use`，由 Agent 端的 `_dispatch_tool` 决定走本地函数还是 MCP 远程调用。两条通道对 LLM 完全透明。

## 第 4 章 MCP Server 实现（mcp_server.py）

### 4.1 Part 1：工具定义（schema + 实现）

每个工具仍然是「schema + fn」两件套——demo3 把本地工具的 schema 设计成接近 MCP schema（都用 `input_schema` 字段），因此可以低成本合并。唯一的区别：它们不被 Agent 进程直接 import，而被一个独立的 HTTP 服务持有。

### 【屏显 · 代码 4-1 — 三个工具的 schema + 实现】

```python
TOOLS = [
    {
        "name": "add",
        "description": "计算两个数字的和",
        "input_schema": {                      # 与 agent.py 本地工具统一用 input_schema
            "type": "object",
            "properties": {
                "a": {"type": "number"},
                "b": {"type": "number"},
            },
            "required": ["a", "b"],
        },
    },
    # multiply / weather 同构，schema 略
]

def fn_add(a, b):
    return a + b

# 路由表：工具名 → 实现函数
TOOL_FUNCTIONS = {"add": fn_add, "multiply": fn_multiply, "weather": fn_weather}
```

### 【口播】

**这一层和 demo1 完全一样**——schema 描述能力、fn 实现能力、字典做路由表。MCP 没有重新定义工具能力，它定义的是工具发现、调用和交互协议。

### 4.2 Part 3：JSON-RPC Handler（三大 method 分发）

Handler 接收 JSON-RPC 请求，按 `method` 字段分发到三个内部函数。

### 【屏显 · 代码 4-2 — handle_request 主分发】

```python
def handle_request(payload: dict) -> dict:
    method = payload.get("method")
    req_id = payload.get("id")
    params = payload.get("params", {}) or {}

    try:
        if method == "initialize":
            result = _handle_initialize(params)
        elif method == "tools/list":
            result = _handle_tools_list(params)
        elif method == "tools/call":
            result = _handle_tools_call(params)
        else:
            # JSON-RPC 标准错误码 -32601 (Method not found)
            return _error(req_id, -32601, f"未知方法: {method}")

        return {"jsonrpc": "2.0", "id": req_id, "result": result}

    except Exception as e:
        # 标准错误码 -32603 (Internal error)
        return _error(req_id, -32603, f"服务端内部错误: {e}")
```

### 【屏显 · 代码 4-3 — 三个 method 的实现】

```python
def _handle_initialize(params):
    """握手：协议版本协商 + 能力声明"""
    return {
        "protocolVersion": "2024-11-05",
        "capabilities": {"tools": {}},
        "serverInfo": {"name": "demo3-mcp-server", "version": "1.0.0"},
    }

def _handle_tools_list(params):
    """工具发现：返回完整 schema 列表"""
    return {"tools": TOOLS}

def _handle_tools_call(params):
    """工具调用：name + arguments → 执行 → content 包装返回"""
    name = params.get("name")
    arguments = params.get("arguments", {}) or {}

    if name not in TOOL_FUNCTIONS:
        raise ValueError(f"未知工具: {name}")

    fn = TOOL_FUNCTIONS[name]
    value = fn(**arguments)                          # ← 关键：跨进程 RPC 与本地函数的衔接点

    # MCP 协议规定 tools/call 返回值用 content 数组包装（便于多模态扩展）
    return {"content": [{"type": "text", "text": str(value)}]}
```

### 4.3 Part 4：HTTP Server 入口

用标准库 `http.server` 起一个 HTTP 服务，所有 JSON-RPC 走同一个端点 `/mcp`。

### 【屏显 · 启动 MCP Server】

```bash
$ python mcp_server.py
============================================================
Demo3 MCP Server 已启动
监听:   http://127.0.0.1:8888/mcp
协议:   JSON-RPC 2.0 over HTTP Transport
工具:   add, multiply, weather
============================================================
等待 Agent 调用...
```

**就这么简单**——MCP Server 的本质就是一个 HTTP 服务，按 JSON-RPC 协议处理三个 method。

## 第 5 章 Agent 端：MCP Client + 工具合并

### 【口播】

Server 端讲完，回到 Agent 端。Demo3 给 `agent.py` 加了三块新东西：

1. **MCP Client**（Part 5）—— 把 JSON-RPC 调用封装成 Python 方法
2. **工具合并**（Part 6）—— 本地工具 + MCP 工具 = 完整工具列表
3. **统一路由**（Part 7）—— 主循环里按工具名分发：本地走函数、MCP 走 client

### 5.1 MCP Client：把 JSON-RPC 调用包装成方法

### 【口播】

每次手写 JSON-RPC 请求太繁琐。封装成一个 `MCPClient` 类，把"组装 payload / POST / 解析响应 / 错误处理"全部藏在 `send()` 里，再在 `send` 之上封三个对应 MCP 三大 method 的方法。

### 【屏显 · 代码 5-1 — MCPClient 三层封装】

```python
class MCPClient:
    def __init__(self, url: str):
        self.url = url
        self._id = 0

    def send(self, method: str, params: Optional[dict] = None) -> dict:
        """发送一次 JSON-RPC 2.0 请求并返回 result 字段"""
        payload = {
            "jsonrpc": "2.0",
            "id":      self._next_id(),
            "method":  method,
            "params":  params or {},
        }
        resp = requests.post(self.url, json=payload, timeout=30)
        data = resp.json()
        if "error" in data:
            raise RuntimeError(f"MCP 调用失败 ({method}): {data['error']}")
        return data.get("result", {})

    # ---- 三个上层封装 ----
    def initialize(self) -> dict:
        """握手：协议版本协商 + 拿 server 能力声明"""
        return self.send("initialize", {
            "protocolVersion": "2024-11-05",
            "capabilities":    {},
            "clientInfo":      {"name": "demo3-tools-agent", "version": "1.0.0"},
        })

    def list_tools(self) -> list:
        return self.send("tools/list", {}).get("tools", [])

    def call_tool(self, name: str, arguments: dict) -> str:
        result = self.send("tools/call", {"name": name, "arguments": arguments})
        return "\n".join(b["text"] for b in result["content"] if b.get("type") == "text")
```

### 【口播】

这三层封装让 MCP 调用看起来就像普通 Python 方法——`mcp.list_tools()`、`mcp.call_tool("add", {"a":1,"b":2})`——但底层每一次都是完整的 JSON-RPC 往返。

### 5.2 工具合并：一行 `+` 搞定

### 【口播】

启动时一次性拉取 MCP 工具列表，与本地工具合并，得到完整工具集。**整个会话内不变**。

由于 demo3 主动让本地工具 schema 向 MCP 对齐（都用 `input_schema`），合并就是**直接 `+` 拼接**——不需要任何包装函数或格式转换。

### 【屏显 · 代码 5-2 — 工具合并就是一行 +】

```python
# main() 里：
mcp_tools = mcp_client.list_tools()           # 拿 MCP server 的 schema 列表
all_tools = LOCAL_TOOLS + mcp_tools            # 直接拼接，完事
```

### 【口播 · 为什么这么简单】

这就是 schema 设计接近的好处——本地工具和 MCP 工具的结构对齐（都是 `{name, description, input_schema}`），对大模型来说没有任何差异。合并不需要函数、不需要转换、不需要标记来源。**一行 `+` 就够了**。

> **真实 MCP 协议规范用的是 `inputSchema`（驼峰）**。demo3 因为演示简化了设计，两端统一用 `input_schema`，省掉字段名转换。

合并之后，**在工具描述层面，LLM 不需要感知工具来源**——它只看到一个工具列表。真正的差异藏在 Agent 主循环的**路由分发**里。

### 【屏显 · 表格 5-1 — 合并后的完整工具列表（7 个）】

| 来源 | 工具名 | 用途 |
|---|---|---|
| 本地 | `execute_bash` | 执行 shell 命令 |
| 本地 | `read_file` | 读文件 |
| 本地 | `write_file` | 写文件（整文件覆写） |
| 本地 | `edit` | 精确替换文件中的一段文本（demo3 新增） |
| MCP | `add` | 加法 |
| MCP | `multiply` | 乘法 |
| MCP | `weather` | 天气查询 |

### 5.3 主循环路由：`_dispatch_tool` 只管本地 / MCP

### 【口播】

主循环不再 if/else 硬写路由——工具调用统一交给 `_dispatch_tool`，**它只负责本地 or MCP 二选一**。

### 【屏显 · 代码 5-3 — 统一分发器】

```python
def _dispatch_tool(name, args, mcp_client, verbose):
    """统一工具分发：本地 or MCP。LLM 不需要知道工具在哪，只按名字调用。"""
    if name in LOCAL_FUNCTIONS:
        if verbose:
            print(f"  [工具 · 本地] {name}({_preview(str(args), 80)})")
        try:
            return str(LOCAL_FUNCTIONS[name](**args))
        except Exception as e:
            return f"[错误] 本地工具 {name} 执行失败: {e}"

    # 不在本地 → 走 MCP
    if verbose:
        print(f"  [工具 · MCP]  {name}({_preview(str(args), 80)})")
    try:
        return mcp_client.call_tool(name, args)
    except Exception as e:
        return f"[错误] MCP 工具 {name} 调用失败: {e}"
```

### 【口播 · 关键观察】

注意——**对大模型来说，调用本地工具和调用 MCP 工具是完全一样的体验**：都是 `tool_use` 块、都拿到 `tool_result`。路由差异全藏在这个分发器里，**把"工具在哪"这个工程细节从 LLM 的视野里抹掉**。

## 第 6 章 真实案例：5 步任务的一气呵成

用一个**故意混用所有工具类型**的任务把整章讲过的东西串起来——本地 bash、本地 edit、MCP 算术、MCP 天气。

任务如下：

```
1) 如果有 weather.txt，先删除；
2) 计算 35+47；
3) 计算上一步的结果 × 9；
4) 查询上海天气并写入 weather.txt；
5) 修改 weather.txt 里面的上海为北京。
```

### 6.1 启动：工具发现 + 降级保护

### 【屏显 · Agent 启动】

```
============================================================
Demo3-tools Agent 已启动（工具扩展轴）
模型:   glm-5.2
网关:   https://open.bigmodel.cn/api/anthropic
MCP:    http://127.0.0.1:8888/mcp
============================================================
[MCP] 握手成功：{'name': 'demo3-mcp-server', 'version': '1.0.0'} 协议版本 2024-11-05
[MCP] 发现 3 个远程工具：add, multiply, weather
[Tools] 合并后共 7 个工具：execute_bash, read_file, write_file, edit, add, multiply, weather

```

启动三件事：MCP 握手成功、7 个工具合并、若 MCP 没启动则降级为仅本地工具模式（4 个工具照常可用）。

### 6.2 完整运行轨迹

### 【屏显 · 真实运行日志】

```
----- ReAct 第 1 轮 -----
[LLM 决策] stop_reason = tool_use
  - text     : 步骤1/2/4 互相独立，可以并行
  - tool_use : execute_bash({'command': 'test -f weather.txt && rm weather.txt || echo "weather.txt 不存在"'})
  - tool_use : add({'a': 35, 'b': 47})
  - tool_use : weather({'city': '上海'})
  [工具 · 本地] execute_bash(...)   →  "weather.txt 不存在，无需删除"
  [工具 · MCP]  add({'a': 35, 'b': 47})           →  82
  [工具 · MCP]  weather({'city': '上海'})          →  上海 今天天气：多云，气温 22°C

----- ReAct 第 2 轮 -----
[LLM 决策] stop_reason = tool_use
  - text     : 步骤2：35 + 47 = 82。步骤3：计算 82 × 9
  - tool_use : multiply({'a': 82, 'b': 9})         ← 82 来自上一轮 tool_result
  [工具 · MCP]  multiply({'a': 82, 'b': 9})        →  738

----- ReAct 第 3 轮 -----
[LLM 决策] stop_reason = tool_use
  - text     : 步骤3：82 × 9 = 738。步骤4：将上海天气写入 weather.txt
  - tool_use : write_file({'path': 'weather.txt', 'content': '上海 今天天气：多云，气温 22°C'})
  [工具 · 本地] write_file(...)   →  [成功] 文件已写入: weather.txt (18 字符)

----- ReAct 第 4 轮 -----
[LLM 决策] stop_reason = tool_use
  - text     : 步骤5：将 weather.txt 中的「上海」改为「北京」
  - tool_use : edit({'path': 'weather.txt', 'old': '上海', 'new': '北京'})
  [工具 · 本地] edit(...)   →  [成功] 替换 第 1 处（共 1 处）；文件 18 → 18 字符

----- ReAct 第 5 轮 -----
[LLM 决策] stop_reason = tool_use
  - text     : 最后确认一下文件内容
  - tool_use : read_file({'path': 'weather.txt'})   ← LLM 主动验证
  [工具 · 本地] read_file(...)   →  北京 今天天气：多云，气温 22°C

----- ReAct 第 6 轮 -----
[LLM 决策] stop_reason = end_turn
  - text     : 任务完成汇总（5 步全部 ✅）

助手:
| 步骤 | 任务              | 结果                                            |
|------|-------------------|-------------------------------------------------|
| 1    | 删除 weather.txt  | 文件不存在，无需删除 ✅                         |
| 2    | 计算 35 + 47      | 82 ✅                                           |
| 3    | 计算 82 × 9       | 738 ✅                                          |
| 4    | 查询上海天气写入  | 已写入 ✅                                       |
| 5    | 将「上海」改「北京」 | 已替换，最终内容 `北京 今天天气：多云，气温 22°C` ✅ |
```

### 6.3 五个关键观察

| # | 观察 | 启示 |
|---|---|---|
| 1 | **第 1 轮并行 3 个工具**（execute_bash + add + weather） | LLM 判断任务之间不存在依赖，一次返回多个 tool_use，Agent runtime 决定并行执行 |
| 2 | **同一条响应里本地 + MCP 混调** | 路由差异全藏在 `_dispatch_tool`，LLM 视角下 7 个工具调用体验完全一致 |
| 3 | **第 2 轮 multiply 直接复用 add 的 82** | 共享 messages 让 LLM 拿到上一轮 tool_result，跨轮状态自然传递 |
| 4 | **第 4 轮 edit 只发 `old='上海'` + `new='北京'`** | 没 read 整文件、没 write 整文件——token 成本 O(改动大小) |
| 5 | **第 5 轮 LLM 主动 read_file 验证** | 任务没要求验证，但 LLM 自己加了——好的 Agent 会主动自检 |

这一个案例覆盖了 demo3 的全部能力：本地三件套 + MCP 三件套 + ReAct 主循环的并行 / 状态传递 / 自主验证。**用户只写一段自然语言，LLM 自己拆任务、并行、路由、收尾**。

## 第 7 章 总结

### 【口播 · demo3 做了什么】

回顾整节，demo3 完成了**工具轴的两次扩展**：

### 【屏显 · 表格 7-1 — demo3 的两次扩展】

| # | 扩展 | 做法 | 解决的问题 |
|---|---|---|---|
| 1 | **edit（本地）** | 新增 string replacement 工具 | write_file 整文件覆写改一行太浪费 |
| 2 | **MCP（远程）** | 工具搬到独立 HTTP 服务，JSON-RPC 协议 | 工具硬编码、无法接外部能力 |

Agent 能力扩展本质上有**两个方向**：

- **增强单个工具能力**（tool capability）—— `edit` 把"修改文件"从「整文件覆写」推到「精确替换」
- **扩展工具生态边界**（tool ecosystem）—— `MCP` 把"工具所在的位置"从「同进程」推到「跨进程 / 跨机器」
